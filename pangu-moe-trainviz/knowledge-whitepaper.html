大模型训练并行放置白皮书
Rank / Device Placement 的语义边界与产品表达

本文面向 TrainScope 的并行放置建模，系统区分 device、rank、worker、stage、expert 等对象，并说明 DP（Data Parallel，数据并行）、FSDP（Fully Sharded Data Parallel，全分片数据并行）、PP（Pipeline Parallel，流水线并行）、TP（Tensor Parallel，张量并行）、SP/CP（Sequence Parallel / Context Parallel，序列并行 / 上下文并行）与 EP（Expert Parallel，专家并行）分别对应的数据、模型状态、层段、张量、序列和专家维度。报告结论是：产品界面应表达“多维运行时放置”，而不是把所有并行语义压缩为“每张卡承载哪些层/算子”。

Official Structure · 官方语义边界

不同并行方式虽然都以“并行”命名，但其官方语义指向的数据对象并不相同。先建立切分对象总表，可以为后续图表和产品字段提供一致的判断依据。

DP（Data Parallel，数据并行） 数据 batch shard 同一模型副本跑不同数据，再同步梯度 DP7 replica “这张卡负责第 12-24 层”
FSDP（Fully Sharded Data Parallel，全分片数据并行） 参数 / 梯度 / 优化器状态 训练状态分片，前后向按需 all-gather / reshard state shard 3/8 “卡 12 负责 MLP 层”

Scope · 基本对象

card、device、rank、worker、stage、expert 的层级关系

若产品界面仅使用“卡”作为统一描述，硬件位置、运行时身份、层段分工与专家放置容易被混合表达。本节先建立对象层级，作为后续 placement 建模的基础。

对象层级图 · 物理设备、运行时身份与并行坐标
读图顺序：左侧表示物理设备，中间表示运行时身份，右侧表示混合并行配置下的多维放置标签；这些标签是否全部出现，取决于实际训练配置。

Parallel 01 · Data Parallel

DP（Data Parallel，数据并行）：完整模型副本跑不同数据，再做梯度同步

DP（Data Parallel，数据并行）的核心语义是多个 rank 持有同一份模型副本，各自处理不同的数据 shard，并在反向传播阶段同步梯度。它描述的是数据切分与梯度同步关系，而非模型层段切分。

DP 主图 · 多个老师拿同一本答案书，批不同学生作业
类比：多个老师拿着同一本答案标准批不同学生作业。这里“数据不同”指的是每个人拿到的样本集合不同，例如不同 mini-batch shard；不是说模型参数 tensor 长得不一样。
项说明
实现目标提升吞吐，让多个 rank 同时处理不同数据 shard。
设计理由DP（Data Parallel，数据并行）不切分模型结构；训练正确性的关键在于梯度同步，而非层归属。这里的数据 shard 指样本或 mini-batch 的切分，不表示模型 tensor 被切分。
代码或配置落点页面上应显示 DP replica id、sample range / batch shard、input tensor slice for this replica、gradient sync group，而不是显示层段范围。
验证方法检查多个 DP rank 是否展示同样的模型结构描述，但数据 shard 不同；通信面板应反映梯度同步。
页面口径：DP7 replica 或“完整模型副本之一，处理当前 step 的样本 / mini-batch shard”是准确表达；“这张卡负责第几层”不属于 DP（Data Parallel，数据并行）语义。
Parallel 02 · Fully Sharded Data Parallel

FSDP（Fully Sharded Data Parallel，全分片数据并行）：训练状态分片与按需重建

FSDP（Fully Sharded Data Parallel，全分片数据并行）的重点是训练状态与显存管理：参数、梯度和优化器状态以分片形式保存，并在前向或反向传播的特定阶段按需 all-gather，再在使用后 reshard。

FSDP 主图 · 一本大书拆成多份，轮到某章时再拼起来
类比：一本很厚的参考书被拆成多份，老师们平时只背自己那一份；轮到某章要用时，才临时把那章拼齐。
项说明
实现目标降低显存占用，让更大的模型可训练。
设计理由FSDP（Fully Sharded Data Parallel，全分片数据并行）切分的是参数、梯度与优化器状态，这与“哪张卡负责哪一段层”属于不同问题。
代码或配置落点页面应展示 state shard、all-gather window、reshard point 等语义，而不是层标签。
验证方法检查同一模块在不同 rank 上是否展示成状态切片；前后向关键阶段是否存在 gather / reshard 事件说明。
页面口径：可以写“该 rank 持有模型参数/梯度/优化器状态的一部分分片；前后向期间按需 all-gather”，不要写“卡 12 负责 MLP 层”。
Parallel 03 · Pipeline Parallel

PP（Pipeline Parallel，流水线并行）：连续层段切分与流水执行

PP（Pipeline Parallel，流水线并行）将模型切分为多个连续 stage，每个 stage 运行一段连续层，micro-batch 沿流水线顺序推进。因此，PP 是最适合展示层段范围的并行方式。

PP 主图 · 工厂流水线按工序分站
layers 36-42、stage 5 这类文案在 PP 语义下是准确的。
类比：工厂流水线把一件产品拆成若干站，每一站只负责自己那段工序，上一站做完交给下一站。
项说明
实现目标把深模型按层段切开，让不同 stage 并行工作。
设计理由只有 PP 在官方语义上直接对应“这个 stage 运行模型的哪一部分”。
代码或配置落点适合在 placementMap 中直接保存 layerStart、layerEnd、stageId，并在 inspector 中显示层段范围。
验证方法切换 stage 时，界面必须显示连续层段；micro-batch 流转顺序应与 stage 顺序一致。
页面口径：PP5 · layers 36-42、“该 stage 执行第 36-42 层的前后向”这类说法是可靠的。
Parallel 04 · Tensor Parallel

TP（Tensor Parallel，张量并行）：同层张量分片与协同计算

TP（Tensor Parallel，张量并行）的核心语义是在同一层内部切分张量或线性层权重，由多个 rank 协同完成该层计算。它回答“该层如何分片”，而非“该层归属哪个 rank”。

TP 主图 · 一张大矩阵按列或按行分给多人同时计算
类比：四个人同时算同一页表格的不同列块。每个人都在参与这一层，但没有谁“独自拥有”这一层。
项说明
实现目标把单层内部的大矩阵与大算子切开，扩展单层可并行规模。
设计理由TP（Tensor Parallel，张量并行）的官方语义是行切 / 列切，体现的是同层协同计算，而非整层归属。
代码或配置落点适合显示 op name、tensor shard id、group size，如 Attention QKV shard 2/8。
验证方法切换到 TP（Tensor Parallel，张量并行）视图时，同一层应同时出现在多个 rank 上；说明文字必须强调 shard，而非层归属。
页面口径：卡 7 / rank_7 持有 Attention QKV 的 TP shard 2/8 是准确的；“卡 7 负责 Attention 层”会误导。
Parallel 05 · Sequence / Context Parallel

SP / CP（Sequence / Context Parallel，序列并行 / 上下文并行）：序列与上下文维度切分

SP / CP（Sequence Parallel / Context Parallel，序列并行 / 上下文并行）的核心语义是将 token 序列或上下文窗口切分为多个 shard，由多个 rank 分担长序列计算压力。它描述“哪个 rank 处理哪段 sequence/context”，而非层段归属。

SP / CP 主图 · 一本长小说被拆成连续片段阅读
类比：一本很长的小说被切成连续片段给不同人阅读，大家用同样的规则处理文字，但每人只处理自己那段 token。
项说明
实现目标在长上下文场景下把 sequence 维切开，降低单 rank 的序列负载。
设计理由SP / CP（Sequence Parallel / Context Parallel，序列并行 / 上下文并行）回答的是“哪段 token 由哪个 rank 处理”，而非“哪层由哪个 rank 处理”。
代码或配置落点适合展示 sequence shard id、token range、context window。
验证方法同一层或模块在多个 SP/CP rank 上仍然存在，但 token range 不同；说明文字必须明确是序列切分。
页面口径：CP1 · sequence shard 1/4、“负责当前上下文窗口的一段 token”是准确的；“CP3 承载第 20 层”不准确。
Parallel 06 · Expert Parallel

EP（Expert Parallel，专家并行）：专家放置与动态路由

EP（Expert Parallel，专家并行）的核心语义是专家放置与 token 路由：Gate 决定 token 被路由到哪个 expert，而 expert 再分布到多个 rank 上。它表达 expert placement，而非整个 MoE 层的单点归属。

EP 主图 · 医院分诊台把病人送到不同专科医生
类比：医院分诊台把病人送去不同专科医生。医生分散在不同诊室里，病人去哪个医生由 gate 决定，不是先固定绑卡再解释语义。
项说明
实现目标在 MoE 层内部把 experts 分布到多个 rank 上，提升专家总容量与并发。
设计理由EP（Expert Parallel，专家并行）的主体对象是 experts；它描述“某个 rank 持有哪些 experts 或 expert shards”，而非“某张卡负责整个 MoE 层”。
代码或配置落点适合在 placementMap 中展示 expertIds、expertShardRange、gate route target。
验证方法从任意 token 或 expert bucket 出发，页面能追到它被路由到哪个 rank/device；并且不假设 1:1 固定映射。
页面口径：EP19 · experts 152-159、“该 rank 持有当前 MoE 层的一部分 experts / expert shards”是准确的；“卡 19 负责 MoE 层”不准确。
Framework · 多维并行叠加

真实训练不是单选题，而是多维坐标叠加

在混合并行训练中，同一个 rank 往往同时拥有 PP（Pipeline Parallel，流水线并行）、TP（Tensor Parallel，张量并行）、DP（Data Parallel，数据并行）、CP（Context Parallel，上下文并行）和 EP（Expert Parallel，专家并行）等坐标。这里的“叠加”不是把一张卡拆成多张卡，而是同一个 rank 在多个通信组、切分维度和专家路由维度中各有一个坐标。

与 ParallelDemo 的匹配关系：两者逻辑匹配，但准确表达应是“配置的并行度生成多维 rank 网格，再把同一 rank 投影到不同并行坐标上”。ParallelDemo 当前实现中 world = dp × pp × cp × tp，并用 grid = arange(world).view(dp, pp, cp, tp) 推导 rank、TP（Tensor Parallel，张量并行）组、PP（Pipeline Parallel，流水线并行）组、CP（Context Parallel，上下文并行）组和 DP（Data Parallel，数据并行）组；EP（Expert Parallel，专家并行）/ MoE 在该 Demo 中作为专家放置与 token 路由语义叠加显示，不参与这条 world 乘法。
组合主图 · 一个 rank 的多维身份
读图顺序：先看左边的 rank 身份，再看右边叠加的五类坐标。真正的 placement 是多维标签交叉，而不是单个归类。
数据层级本图采用的口径可靠性边界
官方公开模型配置 openPangu-Ultra-MoE-718B：61 层、hidden 7680、256 路由专家、1 共享专家、top-8、MLA / MTP / sandwich norm 等结构。 official config 可作为确定训练场景的静态模型底座。
派生训练 placement 用 DP=32 / PP=16 / CP=1 / TP=8 落成 world=4096 的 rank mesh；MoE 侧用 EP=32 把 256 experts 聚合为 32 个 expert bucket。 derived 这是产品可解释样例，不声称 openPangu 官方真实训练切分已经公开。
动态训练遥测 loss、grad norm、router z-loss、专家负载、All-to-All 字节数、rank util 等信号需要来自一次自有 Ascend run，或用 OLMoE/BLOOM 逻辑信号重投影。 synthetic / self-collected 未公开的逐 step 数据不能伪装成官方真实遥测。
把 ParallelDemo 落实成真实训练配置 · 从公开模型到单卡计算
证据边界：模型结构来自公开 openPangu-Ultra-MoE-718B config 与 inference/model.py；DP=32 / PP=16 / CP=1 / TP=8 / EP=32 是用于把 ParallelDemo 公式落实到大规模训练的派生 placement，不是官方公开的真实训练切分。模型架构只定义 forward 主干；训练 step 还包含 loss、backward、通信同步与 optimizer update，可参考 PyTorch 训练循环示例。MindSpore Transformers 配置说明提供 data_parallel、model_parallel、context_parallel、pipeline_stage、expert_num、num_experts_chosen 等字段依据，通信 API 提供通信组与 rank 映射依据。与 ParallelDemo 的差异边界：截图中的 H100_0 是 Stage 0，所以显示 Embedding；本图的 rank_299 / PP5 不应显示 Embedding，只显示该 stage 覆盖的 blocks。E152-E159 是盘古产品场景中的 EP bucket，不是直接照搬 ParallelDemo 当前 demo 里按 TP shard 展示专家片的简化逻辑。
概念在训练中的含义在图表中的表达边界
stage PP（Pipeline Parallel，流水线并行）把整网层序列切成的流水线阶段。 Stage 0 表示模型开头的层段，不是第 0 个训练 step；只有 Stage 0 才通常包含 Embedding。
训练样本 / sample 一条训练数据；语言模型里通常是一段 token 序列。 DP（Data Parallel，数据并行）下不同 rank 吃不同 sample shard，不是模型结构不同。
batch / mini-batch shard 一个 step 输入的一组样本；被 DP/CP 等切分后，每个 rank 只处理其中一部分。 可显示 samples 512-1023 或 mini-batch shard，不要把它说成 tensor shard。
tensor 数据、权重、激活、梯度在计算机里的多维数组表示。 TP（Tensor Parallel，张量并行）切的是权重/激活等 tensor 维度，例如 QKV / Out 第 4/8 片。
step 一次训练迭代，通常包含 forward、loss、backward、通信同步、optimizer update。 模型架构图只覆盖 forward 主干；step runtime card 才表达 backward、通信和参数更新。
产品口径：整网模型架构是“前向计算模板”，训练步骤是“执行一次训练迭代”，单卡任务是“整网架构经 placement 投影到某个 rank/device 后，在当前 step 里实际承担的切片与通信”。

问题细化结论
这些并行度能否由开发者配置 可以配置，但不是任意配置。MindSpore Transformers 暴露 parallel_config.data_parallel、model_parallel、context_parallel、pipeline_stage 等字段；这些值必须满足设备数量、模型层数、attention head 数、序列长度、global batch、拓扑和通信开销等约束。ParallelDemo 的滑块表达的是这个配置空间。
与 ParallelDemo 的公式关系 当前 ParallelDemo 使用 world = dp × pp × cp × tp，并将 rank reshape 为 grid[d][p][c][t]。因此 TrainScope 应复用这套 rank 坐标与通信组推导逻辑；如果真实盘古训练引入独立 EP（Expert Parallel，专家并行）维度，需要在数据层显式建模，而不是默认把 EP 固定绑定到 TP 或卡号。
图表应该怎么展示 主视图建议按三层组织：先展示物理层 host/device，再展示运行时层 globalRank/localRank，最后展示并行坐标层 pp/tp/dp/cp/ep。单卡详情面板再展开“本 rank 的卡内计算切片”，避免在总览里平铺真实训练的全部卡。
并行策略对单卡 / 单 rank 内部呈现的影响以盘古 MoE 训练视图为例
PP（Pipeline Parallel，流水线并行） 决定这个 rank 处在第几个流水线 stage，以及该 stage 覆盖哪些连续层段。 Stage 0 可显示 Embedding + early Transformer blocks；Stage N 可显示后续 blocks 或 LM Head。这里才适合写 layers 36-42 这类层段范围。
TP（Tensor Parallel，张量并行） 决定同一层内部的张量 / 权重 / 算子切片，通常需要同组 all-reduce、all-gather 或 reduce-scatter。 Embedding 词表分片、Attention QKV column shard、Attention output row shard、MLP up/down projection shard、LM Head vocab shard；卡片上应标 第 3/4 片，而不是“独占 Attention 层”。
CP（Context Parallel，上下文并行） 决定 attention 看到的 token / context 分片，以及 KV、QKV 或 attention 中间结果在组内如何交换。 同一个 Block 的 Self-Attention 区域显示 CP1 · token shard、KV exchange、Ring / Ulysses / hybrid CP 等算法标签；它不改变该 rank 属于哪个 PP stage。
DP（Data Parallel，数据并行） 决定这个 rank 处理哪份样本 / mini-batch shard，并决定 backward 后的梯度同步组。 卡片或顶部 batch 条显示 samples 512-1023 之类的数据 shard；反向传播后显示 gradient all-reduce，同步的是梯度 / 参数更新，不是把样本数据拼回去。
EP（Expert Parallel，专家并行） / MoE 决定当前 rank 持有哪些专家、token 经 gate 路由到哪些专家，以及 dispatch / combine 的通信路径。 MoE FFN 区域显示 router / gate 复制、experts E4-E5、本 rank 的 top-k token bucket、all-to-all dispatch/combine。不要把 EP组 简化成“这张卡负责整个 MoE 层”。

Findings · 总结

把这件事记成三条判断

如果只保留最重要的结论，应该是下面三条。

核心判断真实大模型训练不是“卡按层和算子分配”一句话能说完，而是多个并行维度共同定义 placement。
准确边界“哪张卡负责哪几层”只适用于 PP（Pipeline Parallel，流水线并行）；TP、DP、FSDP、SP/CP、EP 都需要更精确的语义字段。
产品方向TrainScope 的正确标题是 placement，不是 layer ownership；ParallelDemo 提供的是计算引擎，不是最终 UI 结构。
最终结论：对 `PP（Pipeline Parallel，流水线并行）` 来说，“每张卡承载哪些层”基本准确；对 `TP（Tensor Parallel，张量并行）` 来说，这句话只说对一半；对 `DP（Data Parallel，数据并行）/ FSDP（Fully Sharded Data Parallel，全分片数据并行）/ CP（Context Parallel，上下文并行）` 来说基本不准确；对 `EP（Expert Parallel，专家并行）` 来说，应改成“该 rank/device 持有哪些 experts 或 expert shards”。
