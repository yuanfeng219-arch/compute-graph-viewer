PR / DT 场景划分

指南强调 950 有 PR / DT 两类形态，Prefill、Decode、训练、推荐和多模态瓶颈不同。

AI Core 子系统重点是低比特格式、Cube-Vector 融合通路、Vector 算力提升、SIMT/SIMD、RegBase、NDDMA 和 128B Sector Cache。

CCU — 集合通信单元

集合通信专用引擎，用于通信节点间同步、地址交换、数据传输和规约计算。

目标是降低内存带宽占用、降低通信时延并释放 AI Core；AllGatherMatMul、ReduceScatter、Dispatch/Combine 需要用 HCCL/CCU profiling 验证。

Regbase 架构

RegBase 允许直接操作 Vector 计算寄存器，GM 数据仍需先进入 UB，再显式 Load/Store 到寄存器。

NDDMA 支持多维非连续访存、自动补 PAD；128B Sector Cache 用于改善小包 / 非对齐访问。

新数据类型速览

指南确认 HiF8、FP8、MXFP8、MXFP4 是 950 学习重点；具体编码范围、溢出、饱和和产品支持边界需按当前 CANN API 文档与头文件确认。

类型 格式 最大值 主要用途 注意事项
MXFP4 4-bit，有共享缩放因子 — MatMul 权重量化，最激进压缩 需要 Scale 预存入 UB，否则成为瓶颈
MXFP8 8-bit，有共享缩放因子 ~57344 MoE Activation 量化（A5 量化方案） 动态范围远大于标准 FP8-E4M3（max 448）
HiF8 8-bit 特殊格式 ~32768 训练激活值，高动态范围 非标准 FP8，转换时需特殊处理
FP8-E4M3 1+4+3 448 推理权重，兼容 GPU 动态范围较小，LLM 激活值易溢出
FP8-E5M2 1+5+2 57344 训练梯度 精度低于 E4M3，适合梯度存储

▼ A2 / A3 Profiling 能力

✕通信只看端到端 HCCL 耗时，无法解释 CCU / 片上资源竞争

单算子精细分析
msprof op + MindStudio Insight
对单个慢算子采集 Pipe 流水图 / PC Sampling / SIMT GPR 分析，找 bubble 和 stall。
整网采集命令
# 基础整网采集；具体开关请以 msprof --help 和 CANN 文档为准
msprof \
  --output=./prof_output \
  --application="python train.py" \
  --task-block=on       # 950新增：AICore首尾block
  --sys-hardware-mem=on # 如当前版本支持，用于片上带宽采集
  --process-num=<N>     # 按实际进程数配置

# CCU 通信 profiling
msprof \
  --ccu-profiling=on \  # 如当前版本支持，用于 CCU 相关采集
  --output=./ccu_prof \
  --application="python train.py"
单算子采集命令
# 单算子精细采集；参数名按当前 CANN 版本确认
msprof op \
  --op-name=MatMul \
  --pipe-view=on \      # Pipe流水图 (ID14/ID15)
  --pc-sampling=on \    # 9种stall状态采样
  --simt-gpr=on \       # SIMT寄存器分析
  --simt-ipc=on \       # Core Occupancy+IPC
  --output=./op_prof

# 解析
msprof parse --input=./op_prof

✕Scalar 慢 → 只能反复加 printf 打日志，猜哪条指令在 stall
✕实测优化收益：单凭经验平均 ≤1.5× 改善

三类算子编程场景：Vector / Cube / 融合算子差异
三类算子场景 × 芯片架构路径

Vector、Cube、融合算子的差异需要落到硬件路径上看：连续规则数据看 UB 与 Vector Reg File，离散数据看 SIMT Register File，Cube 低比特主路径看 GM 到 L0A/L0B，融合场景看 AIC 与 AIV 的 C-V 直连。

你是否发现：910 上的 Vector/Cube 经验搬到 950 后，代码能跑但性能路径完全变了——到底该按哪种算子范式重写？
950 的关键变化不是单点算力增强，而是把算子编程拆成三类场景：连续规则数据走 SIMD Reg，离散索引数据走 SIMT 多线程，Cube 主计算和 Vector 后处理走 MIX / C-V 直连融合。
▼ 910 / A3 典型心智
✕Vector 多围绕 UB / LocalTensor 做 Memory-based 流水中间结果经常在 UB 中反复读写
✕Cube 优化重点集中在 MMAD 主循环、L0/L1/UB Tiling 和 Double Buffer
✕融合算子常以两个 kernel 串联或 workspace 中转实现Cube 写 GM，Vector 再从 GM 读回
✕复杂分支、离散索引、多线程任务容易退化成 scalar/访存瓶颈
→
▲ 950 场景化编程心智
✓Vector 算子：连续、规则、表达式型数据优先走 SIMD Reg / VF 融合减少 UB 往返，但要控制寄存器压力
✓Cube 算子：低比特数据类型进入主路径HiF8 / FP8 / MXFP8 / MXFP4、scale 缓存和低比特 layout 需要进 Tiling 设计
✓融合算子：MIX 编程模式让 AIC + AIV 协同执行这里的 MIX 是混合执行/编程模式，不是数据类型
✓SIMT 多线程：离散访问、索引、分支、Gather/Scatter 类数据用 thread block / warp 表达
算子场景 数据形态 910 典型实现 950 关键变化 编程判断
Vector 算子 连续、规则、表达式型数据 Memory-based Vector；UB / LocalTensor 读写；CopyIn / Compute / CopyOut 流水 SIMD Reg / RegBase、VF 融合、显式 Load/Store 到寄存器；SIMD 与 SIMT 可混合 优先把中间结果留在 Reg，少走 UB；重点看 GPR 占用、spill、VF 融合收益
Cube 算子 矩阵乘、GEMM、低比特矩阵计算 L0A/L0B/L0C + L1/UB/GM 搬运；MMAD 主循环；Double Buffer HiF8 / FP8 / MXFP8 / MXFP4 等低比特路径；scale cache、伪量化 n-buffer、低比特 layout 进入主路径 不要只调 MMAD；同时检查 scale 搬运、尾块、L2/L1 命中、低比特转换和 Cube 输出布局
融合算子 Cube 主计算 + Vector 前后处理；Matmul+Epilogue、RmsNormQuant、FIA 常见 Cube 写 GM / workspace，Vector 再读 GM；多个 kernel 串联 MIX 编程模式、AIC+AIV 协同、L0C→UB / UB2L1、DualDest、C-V 直连 先画数据流，再定 kernel 边界；目标是让 Cube 结果尽量直接进入 Vector 后处理，减少 GM 往返
SIMT 离散数据 索引、分支、离散访存、多线程任务 容易落到 scalar 控制、非连续搬运或低效 Vector 表达 thread block / warp / shared memory 风格的 SIMT 编程模型；适合 Gather/Scatter、Hash/Atomic、复杂分支 数据不是连续向量表达式时，不要硬套 SIMD Reg；优先检查 SIMT thread block 切分与访存合并
Vector：SIMD Reg 处理连续规则数据

950 SIMT 编程 — 5 大优化技巧
①
合理切分线程数（512~1024 最佳）
线程数不超 2048，优先用 b32 类型（b64 占两个寄存器，消耗翻倍）。切分过少浪费并行度；切分过多超出寄存器限制触发 spill。
②
访存合并（Coalesced Memory Access）
同一 Warp 内所有线程的 GM 访问请求落在同一 Sector 时可合并为单次内存请求，带宽利用率最高。最差情况：每个线程访不同 Sector，n 线程 = n 次独立请求。
推荐：output[threadIdx.x] = input[threadIdx.x] // 连续访问，同 Sector
③
避免 Warp Divergence（分支在 Warp 间，不在 Warp 内）
SIMT 硬件为每个 Warp 分配单个 PC，Warp 内分支跳转导致两路串行执行。
✗ 错：if (threadIdx.x % 2 == 0)（Warp 内分支，半 Warp 空转）
✓ 对：int warp_id = threadIdx.x/32; if (warp_id % 2 == 0)（整 Warp 分支）
④
代码优化：合并独立操作利于编译器并发
将独立的 Load/Compute/Store 重排分组：先批量 LOAD，再批量 PROCESS，再批量 STORE。编译器更容易检测到无数据依赖并发指令。
⑤
#pragma unroll 与参数量控制
#pragma unroll：提示编译器展开循环（减少分支 + 提高指令并行）；但会增加代码体积和寄存器消耗，权衡使用。
SIMT 入参最大 28×32 Bit；超过时用预留空间中转，但延迟增大。建议将 tiling 参数结构体搬入 UB 展开。

950 AIV 引入 Regbase 架构：向开发者开放 SIMD Reg 寄存器，数据流从 GM↔UB（旧）升级为 GM↔UB↔Reg，减少 UB 访存带宽、支持 OOO 双发射。

迁移注意事项 — 常见坑
你是否把 A3 上测过的代码直接搬到 950，CI 全绿、功能正确，但性能只有预期的 30%~50%，而且 profiling 数据完全看不懂？
以下是从外部性能优化案例中总结的高频踩坑点，每个都需要回到当前 CANN Toolkit、SoC 版本和 profiling 数据验证。
⚠
Profiling 参数随 CANN 版本变化
官方指南只确认 950 的 Profiling 会扩展到 Pipe、PC Sampling、SIMT 寄存器、CCU 和片上带宽等层级；具体采集参数、并发能力和文件格式必须按当前 CANN 版本确认。
✓ 迁移时检查所有 msprof 调用，不把旧脚本参数默认带到新环境
⚠
MXFP4 Scale 因子不预存导致瓶颈
MXFP4 每组数据有共享缩放因子 Scale，若在计算时每次从 GM 读取 Scale，访存成本抵消量化收益。实测从 181μs 优化到 175μs（最后一步）正是源于此。
✓ 在算子 prologue 阶段将 Scale 预加载至 UB，整个计算期间复用
⚠
L1 Bank Conflict（访问步长问题）
L1 存储有 Bank 结构，若矩阵访问步长恰好是 Bank 数倍数，会导致所有访问打到同一 Bank，吞吐下降数倍。Ascend C 程序员从 GPU 迁移后特别容易忽略。
✓ Tiling 时将矩阵行宽 pad 到非 Bank 数倍数（如 +1 列），或使用 Swizzle 访问模式
⚠
Regbase 编程范式转变
950 Regbase 架构下，操作数直接来自寄存器而非 Local Buffer，Large Regbase 操作减少 UB 带宽压力。但 GPR 数量有限，使用过多 Regbase 变量会导致 spill，反而引发性能下降（SIMT GPR 分析可诊断）。
✓ 用 SIMT GPR 视图监控 GPR Count，超过阈值时重构减少临时变量
⚠
CCU 路径需确认对齐与版本约束
官方指南确认 CCU、128B Sector Cache 和访存优化方向，但本地指南未给出统一的 CCU 零拷贝地址对齐规则；是否 fallback、如何分配对齐内存需要按当前 CANN/HCCL 文档和算子实现确认。
✓ 以当前工具链文档、Profiling 采集结果和算子代码为准，避免把外部案例约束当作通用规则
⚠
片上带宽非 profiling 不可见
950 调优需要额外关注片上访存瓶颈、小包、非对齐、L2 miss 和 GM 往返；是否需要显式采集开关取决于 CANN 版本。
✓ 将片上带宽 / PMU 指标纳入团队 profiling baseline，具体参数按版本固化
⚠
HiF8 / MXFP8 与 FP8-E4M3 混淆
950 引入多种 8-bit 格式。HiF8 最大值 ~32768、MXFP8 最大值 ~57344，而标准 FP8-E4M3 最大值仅 448。混用或错误转换会导致激活值溢出截断，引发精度问题但不报错。
✓ 量化代码中明确标注数据类型枚举，不要用 "FP8" 泛指；添加范围校验断言
⚠
TileLang cross_interval 参数调错
TileLang Pipeline 的 cross_interval 控制核间流水交错间隔，设为 0 退化为无流水；设过大则流水 stage 数不足，实测 FA 优化从 63% → 80% 的关键就是找对 num_stages=8, cross_interval=2 组合。
✓ 使用 Pipe 流水图验证 bubble 是否消除，不要凭感觉设置 cross_interval
⚠
尾 Block 负载不均衡（tbmiss 问题）
当矩阵尺寸不能被 Block 大小整除时，最后一个 Block（尾 Block）工作量偏少，所有核中只有该核没有做完，其他核等待，导致整体效率下降。
✓ 用核间负载图 + tbmiss 采集分析；对尾粒进行合并或 padding 均衡处理
⚠
TileLang/Ascend C 未利用 950 L0C→UB 直通路径
910/A3 代码中 AIC 输出数据经 L0C→GM，AIV 再从 GM→UB 读取（两次 GM 搬运）。950 新增 L0C→UB 直通通路，若不修改代码仍走旧路径，融合算子性能潜力白白损失，且多消耗 workspace 内存。
✓ 将 T.copy(c_sh, ws) 从 L0C→GM 改为 L0C→UB；注意 DualDest 的 M/N 对齐要求（M 为 2 倍数/N 为 32 倍数）
⚠
SIMT 编程调试时忘用 msSanitizer
SIMT 范式引入 Warp/Thread Block 后，内存越界、未初始化访问、set_flag 缺失引发的竞争等问题比 SIMD 更难凭 printf 定位。msSanitizer 提供 4 种精确检测模式，但需在编译时加 --sanitizer -g 才能生效，遗漏这两个参数则工具完全不工作。
✓ 将 msSanitizer 编译选项加入 Debug 构建 preset；先跑 memcheck + synccheck 再跑 tracecheck
⚠
A2/A3→950：5 大 Ascend C API 不兼容项（约 5% 代码需改动）
95% API 改 CMake 版本号重编译即可；但以下 5 类接口在 950 上已变更，直接搬代码会编译失败或行为错误：
类别变化影响接口迁移方案
数据通路删除 L1Buffer→GM 路径DataCopy / DumpTensorCube only: 用 MIX 通过 Vector 搬运；Mix: 走 L1→UB→GM
数据通路删除 GM→L0A/L0B 直接路径InitConstValue / LoadData先 LoadData 到 L1，再 Pipeline 到 L0
数据类型Cube 不支持 int4b_tLoadData / Mmad改用 int8_t + 手动转换：Cast(int8)→Mmad
L0A 形状ZZ格式→ZN格式LoadDataWithTranspose / Mmad切分数量 ×2 个 L0A Buffer，Stride 不变
Cube only不支持单独 MMADmatmul_mx 等改为 MIX Core 类型执行
✓ 迁移前先扫描代码中的 L1Buffer DataCopy、int4b_t 类型、cube_only 模式，逐一替换
⚠
Ascend C SIMD/SIMT 范式选错导致性能下降
950 AIV 支持 SIMD 和 SIMT 两种范式，两者共享 UB 但 Register File 组织完全不同。SIMT 的 Reg File 是 per-warp 的 128K 总量；SIMD 的 Reg 是 Regbase 风格。若把 SIMT 密集的算子写成 SIMD，会导致 warp 空转；反之 SIMD 优势（Regbase OOO 双发射）也无法发挥。
✓ 访存密集/gather-scatter 型算子用 SIMT；矩阵规则计算用 SIMD；混合场景用 __mix__(aic_aiv) + SIMT VF + SIMD VF 融合
