这块面板在做什么？ 核代码识别器 · 910B → 950 迁移规则集

一句话

你贴进来的每一行核代码，工具会判断它跑在哪个硬件单元（搬运 / Cube / Vector / SIMT），并对照规则集标出从昇腾 910B 迁移到 950 时的注意点——哪些必须改、哪些要调整、哪些是提速机会。所有判断都来自规则，不需要联网、不调用大模型。

怎么用

  1. 点「上传/粘贴核」，贴代码或选文件；
  2. 左侧源码会高亮关键行，色条表示类别；
  3. 点高亮行，右侧出卡片：硬件单元 + 迁移判定 + 改写建议；
  4. 切换右上角硬件架构（950 / 910B / 对比）看差异。

左侧色条＝硬件单元类别

计算（Compute）— Cube 矩阵乘 / Vector 向量 / SIMT
访存（Memory）— GM↔UB 搬运、DataCopy、格式转换
控制（Control）— 同步、Kernel Launch、迁移检查点

迁移判定的三个等级

必改不改会编译失败或语义错误（如 950 删除的数据通路）
需调整能跑但有陷阱，需要改参数 / 布局
优化机会可选，利用 950 新能力提速

内置的 910B→950 迁移规则

勾选「融合 AI 分析」能补什么

规则集擅长确定、带出处的硬判定（尤其「必改」项），但有天然短板。开启融合 AI（DeepSeek）后，它只做补位与解释、不推翻规则，主要补这 5 块：

维度规则集现状AI 补位
覆盖面 只认写进规则的 API，没见过的就漏 对陌生 intrinsic / 新写法做语义推理，兜底分类
跨函数理解 文件级正则，同名变量可能认错 读懂整个 kernel 的意图（如「这是 Flash Attention」「这段是 tail 处理」）
改写建议 rewriteHint 是写死的模板串 针对你这一行 / 这个核生成真正可用的 950 改写
整体判断 逐行打标签，没有全局视角 自然语言总结整核迁移风险排序 + 改造工作量
依赖 shape/dtype 的「需调整」项 只能给通用提示 结合上下文判断是否真的触发
AI 结果会以「AI 智能推荐」卡片融入对应源码行下，整核摘要挂在首个关键行。需在弹窗里填 DeepSeek API Key（用后即毁，下次需重填）；硬约束：AI 不得与「必改」判定相矛盾。

判据来自哪里

规则集出处已逐条标注，主要来自仓库内两份文档：
· KNOWLEDGE.md（§5.3.5「5 大不兼容项」等）
· perf_engineer_guide.html（910B vs 950 架构差异）
并用昇腾社区公开资料做了基本语义裨证。

已知边界（别误读结果）

· 数据流分析是文件级简易版，不跨函数作用域，同名变量可能认错；
· 只认规则集里写过的 API / intrinsic，没覆盖的会只打标签或不标；
· 这是静态识别辅助，最终请以实际编译 / Profiling 为准。
