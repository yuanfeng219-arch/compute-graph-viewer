# 分析报告输出规范

本文件定义所有性能分析、算子调优、精度分析报告的**通用输出规范**。每次输出报告前，必须先按本规范组织报告结构。

> 阅读者视角：拿到报告的是要尽快动手优化的性能工程师。规则的设计原则是**越靠前越要直接，元数据沉到末尾**。

---

## 规则 1：性能 / 算子调优报告固定章节骨架

**所有性能 Profiling、算子调优、集群诊断类报告（即任何使用 Hermes / Icarus 知识库的报告）必须严格按以下顺序输出 5 个一级章节**，章节名一字不差：

```markdown
# <报告标题>

## 1. 结论速览
## 2. 行动清单
## 3. 问题详情
## 4. 已确认无问题
## 5. 数据与方法（附录）
```

精度分析（Accuracy）报告骨架略有差异，参见本文件末尾"精度报告骨架"小节。

---

### 1. 结论速览（必填，≤6 行）

读者第一眼能拿到的内容（**性能健康度行必须排在最前**）：

- **性能健康度（PHS）** — 仪表盘指标，当前分 + 等级 + `→ 优化后预估分 + 等级` + 各子项，**同一行**
- **一句话定性**：当前瓶颈类型 + 影响面（如"训练单步耗时被通信主导，占比 38%"）
- **头号瓶颈**：最严重的一个问题 + 量化指标
- **收益上限**：若把行动清单 P0/P1 项全部落地，预估能节省多少耗时 / 提升多少利用率（PHS 数字已在首行，此行**不重复**）
- **不要写**：方法论、数据来源、使用了什么工具

**MUST 先调用 `get_skill("performance-health-score")`**，按其公式计算当前 PHS、等级、分项数值，以及 P0/P1 落地后的优化后 PHS 预估值，然后填入下方固定格式。任何性能/算子报告**禁止跳过**这一步。

示例：

```markdown
## 1. 结论速览

- **性能健康度**：68 / 100 (B+) → 优化后预估 **82 / 100 (A)** — 计算 72% · 通信 55% · 调度 82% · 内存 60%
- **结论**：训练单步 1240 ms，被通信和算子调度共同拖慢
- **头号瓶颈**：Rank 5 SDMA 链路带宽仅同类 40%（HCCS 12 GB/s vs 集群均值 30 GB/s）
- **收益上限**：行动清单前 3 项全部落地后节省 ~17% 单步耗时（约 210 ms）
```

PHS 行的字段定义、子项算法、场景权重、N/A 处理、等级映射，**全部以 `performance-health-score/SKILL.md` 为准**，本规则不重复定义。

---

### 2. 行动清单（必填，单张表）

**整份报告最重要的一节**。读者会直接挑前几项开干，所以信息必须自包含：

```markdown
## 2. 行动清单

> 默认按"预期收益"从高到低排序；同档收益按"修改难度"从低到高排序。

| # | 优先级 | 问题 | 预期收益 | 修改难度 |
|---|---|---|---|---|
| 1 | P0 | Rank 5 SDMA 链路带宽仅同类 40% | -8% 单步耗时 | 低 |
| 2 | P0 | MatMul shape 未对齐 padding 浪费 32% | -4% 单步耗时 | 中 |
| 3 | P1 | … | … | … |
```

字段定义：

- **优先级**：`P0` / `P1` / `P2`，P0 = 收益最大且必做
- **预期收益**：量化优先（`-8% 单步耗时`、`MTE 带宽 50% → 80%`）；无法量化时用 `高/中/低` 三档
- **修改难度**：三档
  - `低`：改参数 / 配置 / 编译选项 / tiling，不动算法
  - `中`：局部改算法或数据流（搬运顺序、Cast 位置、同步、Pipe 划分）
  - `高`：重构算子 / 跨模块改动 / 涉及框架或调用方

> 改动位置、问题举证视图均在第 3 章"问题详情"中填写，**行动清单表格不重复**。第 3 章各小节的标题必须与此表"问题"列一字不差。

---

### 3. 问题详情（必填，按"行动清单"中的 # 顺序展开）

每个行动项一个三级小节，结构固定。**小节标题必须与第 2 章行动清单"问题"列完全一致**（标号和优先级可补充，文字不得改动）：

```markdown
### 3.1 [P0] Rank 5 SDMA 链路带宽仅同类 40%

- **证据**：`slow_link.csv` 显示 Rank 5 在 step 100-200 的 HCCS 带宽 12 GB/s，集群同链路均值 30 GB/s
- **影响**：每次 AllReduce 拖慢 18 ms × 64 次/step = 1.15 s（占单步 38%）
- **修复建议**：
  - **改动位置**：NIC 配置 / 网络拓扑层（Rank 5 物理 NIC）
  1. 检查 Rank 5 物理 NIC 配置：`hccn_tool -i 5 -link -g`
  2. 若配置正常，排查交换机端口拥塞 / 邻居 Rank 干扰
  3. 必要时迁移 Rank 5 至其他物理节点
- **问题修改完成的验证方式**：修复后重采 profiling，确认 `slow_link.csv` 中 Rank 5 HCCS 带宽 ≥ 25 GB/s
- **问题举证视图**：通信视图，载入 `evidence/communication_matrix.json`（源：`/data/prof/.../communication_matrix.json`），关注 Rank 5 的链路热力（展示当前问题现象）
```

字段必填（缺一不可）：**证据 / 影响 / 修复建议（含改动位置） / 问题修改完成的验证方式 / 问题举证视图**。证据必须引用具体文件名 + 数值；修复建议首项须标明"改动位置"（文件路径、行号、算子名、Rank ID、配置层等可直接跳转的位置）；问题举证视图须三要素齐全（视图名 + 所需文件 + 关注点，详见规则 4），且"所需文件"必须已复制进报告目录的 `evidence/` 子目录，路径写"副本相对路径（源：原始落盘路径）"两者（详见规则 5）。

---

### 4. 已确认无问题（可选，但推荐）

帮读者省一遍"是不是漏看了"的复盘：

```markdown
## 4. 已确认无问题

- 内存带宽（L2Cache 命中率 92%，无瓶颈）
- 算子调度（Host 下发延迟均值 0.3 ms，正常）
- 各 Rank Compute Time 极差 < 5%，无计算型慢卡
```

若本次未做相应排查，写"未排查"而非"无问题"。

---

### 5. 数据与方法（附录）

元数据沉到末尾。**这里就是原"本次分析说明"块**，包含规则 2/3 的全部内容：

```markdown
## 5. 数据与方法

- **分析日期**：YYYY-MM-DD
- **数据路径**：`/path/to/profiling_data/`
- **数据范围**：Rank 0-7，step 100-200，采集时长 8.3 s
- **使用的 Skills**：
  - `mindstudio_profiler_data_check`：校验 Profiling 数据完整性
  - `cluster-fast-slow-rank-detector`：识别快慢卡
  - `msinsight-view-selector`：为每个诊断结果推荐 Insight 视图
- **Advisor 状态**：
  - `msprof-analyze advisor`：已调用 — 集群各卡耗时极差与慢卡矩阵分析
- **输出位置**：`./Analysis Report/<标识词>_profiling_analysis_YYYYMMDD/`（本次分析所有产物均在此目录下，命名规则见规则 3）
```

## 规则 2：数据与方法 — Advisor 状态条目

> **Advisor 默认必跑**：所有 Hermes 性能 Profiling 报告，**默认都必须调用 `msprof-analyze advisor`**（见 `msprof-analyze-cli` skill 能力二）并将其建议并入报告。情况 A 是常态；情况 B、C 是仅在确有原因时才允许的例外，必须在状态条目中如实写明原因，**禁止无理由跳过 advisor**。

第 5 章"数据与方法"中，**`使用的 Skills` 列表之后必须紧跟一个 `Advisor 状态` 条目**，按实际情况三选一（优先 A）：

**情况 A（默认 / 常态）：本次成功调用 advisor**

```
- **Advisor 状态**：
  - `msprof-analyze advisor`：已调用 — <一句话说明用于哪个分析步骤>，结果已并入第 2 章行动清单 / 第 3 章问题详情
```

**情况 B：调用 advisor 失败**

```
- **Advisor 状态**：
  - `msprof-analyze advisor`：**失败** — <具体原因，如"msprof-analyze 命令不在 PATH；请确认已安装 msprof-analyze 并将其加入环境变量">
```

**情况 C：本次未调用 advisor（例外，需充分理由）**

仅在数据格式/工具确实不支持时才允许，且必须写明硬性原因：

```
- **Advisor 状态**：
  - 未调用 — <硬性原因，如"数据为纯 text 且工具链不可用"；不接受"无需建议"这类主观理由>
```

精度分析等非 Profiling 报告无需 Advisor 状态条目。

## 规则 3：分析过程与结果文件的落盘位置

分析过程中产生的**任何**过程文件和结果文件——包括但不限于：

- 分析报告（Markdown / HTML）
- SQL 查询 / 分析脚本的中间结果（CSV / JSON，如 `timeline_geometry.py`、`compare_op_stats.py` 等脚本输出）
- 脚本生成的图表、对比表
- advisor 输出落盘文件
- 抽取出来的算子明细、慢卡明细

**严禁写入项目内已有的任何目录**（如 `src/`、`scripts/`、`tests/`、`docs/`、`data/`、用户的训练代码目录、profiling 原始数据目录等），避免污染用户代码库与原始数据。

**允许的两种落盘位置（二选一）：**

1. **新建一个一级目录**（推荐）：在项目根下新建专属目录。**目录命名格式固定为：**

   ```
   <被分析对象标识词>_profiling_analysis_YYYYMMDD/
   ```

   - **前缀 = 被分析对象的标识词**：取本次分析的输入文件夹 / 文件名作前缀。例如分析文件夹 `a/` → `a_profiling_analysis_20260605/`。
   - **名字过长时只取标识词**：当输入名很长（如 `MultiProfLevel2MemoryUB_node1_20260602_rank0_8card/`），截取其中最具区分度的一段作前缀（如 `MultiProfLevel2MemoryUB`），去掉时间戳、`rankN`、`Ncard`、`ascend_pt` 等冗余尾巴；标识词控制在 ~20 字符内，空格与特殊字符替换为 `_`。
   - **多次分析区分**：同一对象多次分析靠 `YYYYMMDD` 日期区分；同一天多次再追加 `_2`、`_3` 序号（如 `a_profiling_analysis_20260605_2/`）。
   - 示例：`./a_profiling_analysis_20260605/`、`./MultiProfLevel2MemoryUB_profiling_analysis_20260605/`
   - 推荐子结构：
     ```
     ./a_profiling_analysis_20260605/
     ├── report.md             # 主报告（按规则 1 的 5 章骨架组织）
     ├── Analysis Process.md   # 本次分析过程记录（步骤 / 耗时 / Token，见规则 6）
     ├── advisor_output/       # advisor 原始输出（若有）
     ├── intermediate/         # SQL 中间结果 / CSV / JSON
     ├── evidence/             # 问题举证视图所需的落盘文件副本（见规则 5）
     └── figures/              # 图表
     ```

2. **直接放在项目根目录下**（仅适合单文件输出）：命名同样带标识词前缀
   - 示例：`./a_profiling_analysis_20260605.md`

### 操作前必须先确认

执行任何写文件操作（`Write` / `tee` / `redirect >` / 脚本 `--output=...`）前：

1. **目标路径必须以新建目录或项目根开头**——若发现路径落在 `src/`、用户代码目录、原始 profiling 目录等已有目录里，立即改路径
2. **`profiling_output/` 等原始数据目录视为只读**——只读取，不在其内部写任何文件
3. 用户指定了输出位置时听从用户；用户未指定时按本规则自行选位置，并在第 5 章"数据与方法"中以列表项形式明示落盘根目录

## 规则 4：每个问题点必须附"问题举证视图"字段

> **字段定位**：`问题举证视图` 展示**当前问题出在哪**——让用户在 MindStudio Insight 中直观看到问题的现象。它与"问题修改完成的验证方式"职责不同：前者是"看现在哪里有问题"，后者是"修复后怎么确认已修复"。两个字段均在第 3 章填写，**行动清单表格不重复**。

**每一个**列在第 3 章问题详情的问题点（瓶颈、异常、优化建议）后必须追加"问题举证视图"字段，指明用户去 MindStudio Insight 哪个视图、载入哪个文件、关注什么，便于直观复现当前问题。

**必须先读取以下文件获取视图映射表，再为每个问题点填写视图字段**：

```
get_skill("msinsight-view-selector")
```

或直接读取：`AscendProfKit/skills/msinsight-view-selector/SKILL.md`

### 触发与豁免

- **触发**：所有 Hermes（性能 Profiling）和 Icarus（算子调优）报告
- **豁免**：
  - Accuracy 工作流（精度问题不用可视化复现）
  - 数据完整性校验失败的报告（没有可用数据，谈不上视图）

### 三要素必填

`问题举证视图` 字段必须包含：**视图名 + 所需文件 + 关注点**，三者缺一不可。详细格式与多视图组合写法见 msinsight-view-selector SKILL.md。

其中"所需文件"必须先按**规则 5**复制进报告目录的 `evidence/` 子目录，并以"副本相对路径（源：原始落盘路径）"两者书写，使报告自包含、用户拿到报告目录即可直接载入复现。

写入第 5 章"使用的 Skills" 列表：

```
- `msinsight-view-selector`：为每个诊断结果推荐 Insight 可视化视图
```

## 规则 5：问题举证视图所需文件必须复制到报告目录

> **目的**：让报告**自包含**。用户拿到（或打包带走）报告目录，无需再回到庞大的原始落盘目录翻找，即可把举证文件直接载入 MindStudio Insight 复现每个问题的现象。

### 触发与豁免

- **触发**：所有 Hermes（性能 Profiling）和 Icarus（算子调优）报告——凡第 3 章问题点带了"问题举证视图"字段，其"所需文件"就要复制。
- **豁免**：与规则 4 一致——Accuracy 工作流、数据完整性校验失败的报告无此要求。

### 操作步骤

1. **收集**：汇总第 3 章所有"问题举证视图"字段里列出的"所需文件"，**按源路径去重**（同一文件被多个问题点引用只复制一次）。
2. **复制**：从只读的原始落盘目录，复制到报告目录下的 `evidence/` 子目录。
   - **保留可区分来源的相对结构**，避免跨 rank / 跨节点同名文件互相覆盖。以落盘根为基准保留 rank / 节点这一层，例如：
     - `rank_3_ascend_pt/ASCEND_PROFILER_OUTPUT/trace_view.json` → `evidence/rank_3_ascend_pt/trace_view.json`
     - `rank_5_ascend_pt/ASCEND_PROFILER_OUTPUT/trace_view.json` → `evidence/rank_5_ascend_pt/trace_view.json`
   - **一律复制，不设大小阈值**：`trace_view.json` / `profiler.db` / `*.bin` 等大文件（数百 MB ~ 数 GB）也照常复制进去。
   - 复制命令保留目录：`mkdir -p evidence/<sub>` 后 `cp <src> evidence/<sub>/`（Windows 用 `Copy-Item`）。
3. **只读保护**（承规则 3）：复制方向**永远是 落盘目录 → 报告目录**；绝不修改、删除、或在原始落盘目录内写任何文件。

### 路径引用（承规则 4）

复制后，第 3 章每个"问题举证视图"的"所需文件"**两者都写**——报告内副本相对路径 + 原始落盘绝对路径：

```markdown
- **问题举证视图**：通信视图 — 载入 `evidence/communication_matrix.json`（源：`/data/prof/cluster/communication_matrix.json`），关注 Rank 5 链路热力
```

### 第 5 章追加"举证文件清单"

第 5 章"数据与方法"附录中，追加一张"举证文件清单"表，登记所有复制项：

```markdown
- **举证文件清单**（已复制至 `evidence/`，报告自包含）：

  | 副本路径 | 原始来源 | 引用的问题点 | 大小 |
  |---|---|---|---|
  | `evidence/rank_5_ascend_pt/communication_matrix.json` | `/data/prof/.../rank_5_ascend_pt/.../communication_matrix.json` | 3.1 | 4.2 MB |
  | `evidence/rank_3_ascend_pt/trace_view.json` | `/data/prof/.../rank_3_ascend_pt/.../trace_view.json` | 3.2 | 820 MB |
```

### 失败处理

源文件不存在或无法读取时**不阻塞报告**：在该问题点的"问题举证视图"字段末尾标注 `⚠ 源文件未找到，未复制：<原始路径>`，并在"举证文件清单"中以"未复制"登记原因。

---

## 规则 6：报告生成后必须追加输出"分析过程记录"（Analysis Process.md）

> **目的**：报告（report.md）回答的是"分析出了什么结论"；`Analysis Process.md` 回答的是"这份结论是怎么一步步分析出来的、花了多少代价"。后者用于复盘分析路径、估算成本、做 Demo 演示素材。两份文件职责分离，缺一不可。

### 触发与豁免

- **触发**：所有 Hermes（性能 Profiling）、Icarus（算子调优）、Accuracy（精度分析）报告——即任何走完整分析工作流并产出 report.md 的场景。
- **豁免**：仅做一次性 SQL 速查、未形成正式报告时可不写。
- **时机**：**主报告 report.md 写完之后再写本文件**，因为只有报告定稿才能回填完整的步骤与结论摘要。

### 落盘位置

文件名固定为 `Analysis Process.md`，写入**与 report.md 同级的报告根目录**（即规则 3 中的 `<标识词>_profiling_analysis_YYYYMMDD/` 下）。严禁写入原始 profiling 数据目录或用户代码目录（承规则 3）。

### 固定章节骨架

```markdown
# AI 分析过程记录

> 数据来源：<数据集标识 + 规模，如 16 Rank / 2 节点 × 8 卡 / PyTorch Level2 DB>
> 分析日期：YYYY-MM-DD
> 完整报告：`<同目录>/report.md`

## 一、Token 消耗
## 二、分析过程 TodoList（含耗时占比）
## 三、各阶段 AI 在做什么
## 四、完整诊断报告摘要
```

### 各章填写要求

**一、Token 消耗**（可获取则填，拿不到精确值时如实标注"估算"或"N/A"，不要编造）

| 指标 | 数量 |
|------|------|
| API 调用轮次 | N 次 |
| 输入（含 Prompt Cache 读取） | … tokens |
| 输出（生成） | … tokens |
| 合计（等效） | … tokens |
| 估算费用 | ≈ $X（注明所用模型与单价） |

## 规则 7：时序结构指标看板表（可见表格，既给人读也供前端看板消费）

> **目的**：让报告把 `timeline-swimlane-analyzer` 算出的泳道时序结构指标**直接展示给人看**，同时仍能被可视化前端（MindStudioNext 总览页"指标看板"）稳定消费。早期版本用 `<!-- METRICS -->` HTML 注释（对人不可见），现已改为**可见的 Markdown 表格**——既不藏数据、也便于解析。

### 触发与豁免

- **触发**：Hermes 性能 Profiling 报告中，凡调用了 `timeline-swimlane-analyzer` 算出"泳道时序结构"指标（关键路径占比、重叠率、空挡、抖动等）的，**必须**在报告末尾追加本表。
- **豁免**：Icarus 算子调优、Accuracy 精度报告无此表；未做 timeline 结构分析时不写。

### 格式

在报告**最末尾**（第 5 章之后）追加一个 `###` 级小节，内含一张表。**指标名（首列）必须用下表"指标"列的标准中文名一字不差**，前端按名映射到看板 key：

```markdown
### 时序结构指标看板（timeline-swimlane-analyzer）

> 由 timeline-swimlane-analyzer 从 trace 泳道几何派生；仅列已测得的指标，未采集的不列。前端"总览—指标看板"消费本表。

| 指标 | 值 | 状态 | 说明 |
|---|---|---|---|
| 关键路径占比 | 76% | warn | step 内 24% 为空泡 |
| 计算-通信重叠率 | 58% | warn | 通信掩盖不足 |
| 暴露通信 | 210 ms |  | — |
| step 抖动 (CV) | 4% | ok | 训练稳定 |
```

### 字段约定（标准指标名 ↔ 看板 key）

| 指标（首列固定写法） | 看板 key | 单位 | 值范围 |
|---|---|---|---|
| 关键路径占比 | `critical_path_ratio` | % | 0–100 |
| 计算-通信重叠率 | `overlap_ratio` | % | 0–100 |
| 暴露通信 | `exposed_comm` | ms | 数字 |
| 最空闲泳道空挡 | `max_lane_idle` | % | 0–100 |
| 泳道最大间隔 | `max_lane_gap` | ms | 数字 |
| Host 下发间隙占比 | `host_launch_gap_ratio` | % | 0–100 |
| PP 流水线 bubble 率 | `pp_bubble_ratio` | % | 0–100 |
| step 抖动 (CV) | `step_cv` | % | 数字 |

列含义：

- **值**（必填）：数字 + 单位（如 `76%`、`210 ms`）；前端只取其中的数字。
- **状态**（可选）：`ok` / `warn` / `bad`，显式指定状态色；**留空时前端按阈值自动判定**（如关键路径占比 <80 → warn）。
- **说明**（可选）：卡片底部一行说明；无则留空或写 `—`。

### 硬性约束

1. **只列测得的指标**：某指标未采集/未计算就**整行省略**——前端对缺失指标保持空占位（"暂无数据"），这正是"没数据就空着"的预期行为。
2. **数值来自真实数据**：禁止编造；`timeline_geometry.py` 的输出（`--json`）可直接映射到对应指标行。
3. **置于报告最末**：作为 `###` 附录小节放第 5 章之后，不新增 `##` 一级章节、不影响第 1–5 章骨架。
4. **首列指标名一字不差**：必须用上表标准中文名，前端按名映射 key；写错名该行不会被看板识别。

## 规则 8：数据来源与落盘信息块（`**数据来源与落盘信息**`，识别落盘数据"是什么"）

> **目的**：每份报告都要回答"这批落盘数据是什么"——来源类型、是否 LLM 训练、模型/用途、大小、下载链接，以及**识别依据**。结论必须有证据支撑；**没有依据识别的项一律留空，严禁臆测**。本块供人阅读，也供 MindStudioNext 自动填充"任务信息"页签的"落盘文件信息"卡片。参考样例：`data/analysis_report.html` 顶部总结表 + 各目录"关键证据"卡片。

### 触发与豁免

- **触发**：所有 Hermes（性能 Profiling）、Icarus（算子调优）报告。
- **豁免**：纯 SQL 速查、未形成正式 report.md 的场景；Accuracy 精度报告可选。

### 操作

**MUST 先 `get_skill("dataset-source-identifier")`**（或读 `AscendProfKit/skills/dataset-source-identifier/SKILL.md`），按其识别信号从落盘数据里取证，再填本块。识别依据、留空原则、信号优先级全部以该 skill 为准，本规则不重复。

### 格式（置于第 5 章"数据与方法"，`使用的 Skills` 之后）

```markdown
- **数据来源与落盘信息**（落盘文件信息卡片；无确切识别依据的项留空，不臆测）：
  - 数据目录：`level2/`
  - 来源：分布式训练 Profiling
  - 是否 LLM 训练：是
  - 模型 / 用途：Qwen 系列 LLM（vocab=151936）
  - 落盘大小：~47 MB（evidence 举证副本）
  - 来源链接：[level2.rar](https://gitcode.com/.../level2.rar)
  - 识别依据：vocab=151936 命中 Qwen 系列 tokenizer；evidence 含 FlashAttentionScore/Grad、RmsNormGrad、ApplyAdamWV2 → LLM 训练；具体参数规模无依据，留空。
```

### 硬性约束

1. **字段名一字不差**、顺序固定（数据目录/来源/是否 LLM 训练/模型 / 用途/落盘大小/来源链接/识别依据），前端按此解析。
2. **是否 LLM 训练** 只取 `是`/`否`/`不适用`。
3. **没依据就留空**：保留该行、冒号后为空（或整行省略），**绝不填猜测值**；前端渲染为"—（无识别依据，留空）"。
4. **每个非空字段都要在"识别依据"里能指到具体证据**（算子名/shape/字段/文件）。

写入第 5 章"使用的 Skills"：

```
- `dataset-source-identifier`：识别并记录落盘数据来源/模型/用途（含识别依据，无依据留空不猜）
```
