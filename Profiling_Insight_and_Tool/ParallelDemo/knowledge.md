# 第一部分 · Transformer 基础概念（先建直觉）

## 1. 术语：层、block、子层、算子是什么关系

"层"这个字有歧义，**取决于看的尺度**。先把尺度分清，后面一切才不乱：

| 说法 | 指什么 | 在 demo 里 |
|------|--------|-----------|
| **Transformer 层 / block** | 一整个 `Attention + MLP`（含 Norm、残差）的组合单元 | 滑块"Transformer 层数"数的就是它 |
| **子层 (sub-layer)** | block 内部的 Attention 子层、MLP 子层 | 原论文管 Attn 和 FFN 各叫一个 "sub-layer" |
| **算子 (operator)** | 最细的 `Linear`、`LayerNorm`、`Softmax` 等运算 | 右侧"整网算子图"展开的节点 |

嵌套关系：

```
Transformer Block（= 1 层，可堆叠的最小重复单元）
├── 子层1: LayerNorm + Attention + 残差
│            └── 算子: Linear(Q) Linear(K) Linear(V) Softmax Linear(O)
└── 子层2: LayerNorm + MLP + 残差
             └── 算子: Linear  激活函数  Linear
```

**关键结论：**
- **"Transformer 层" 和 "block" 是同义词**，指 `Attn+MLP` 这个可堆叠单元，也是数"层数"的单位。网络主干就是 `for i in range(N): x = block[i](x)`。
- 说 "LLaMA-7B 32 层"、"BERT-base 12 层"，数的单位都是 **block**，不是内部算子。
- 模型规模、显存、并行切分（尤其 PP 流水线并行按层切）都按 block 这个整数单位算。

> ⚠️ **误区**：以为"层"只有一个意思。其实"多少层"说的是 block；block 内部还能再分子层、算子，那是更细的尺度，平时不计入层数。

### 嵌套图里的两个配角：残差 + Norm

上面嵌套图里，每个子层除了主角（Attention / MLP），还各挂着两个反复出现的"配角"：**残差连接**和 **Norm（LayerNorm / RMSNorm）**。它们不混合 token、不改变语义方向，只负责"维稳"，但缺了就训不动。先在这里把它们讲透，后面（第 4 节合起来看一层的完整动作）就不再重复。

- **残差连接（那个 `+`）**：每个子层的输出不是直接覆盖原向量，而是 `新向量 = 老向量 + 增量Δ`——只往主流里"加"一笔修正，信息被**不断修订而非推倒重来**（详见第 3 节"残差流"）。它还让梯度能几乎无损地回流到浅层，是深层网络训得动的关键。
- **Norm（归一化）**：在每个子层入口，把向量的数值**拉回一个稳定的分布**（均值≈0、方差≈1），再用两个可学习的小参数 γ、β 缩放平移。它是残差流上的"稳压器"——残差一路累加，数值会越滚越大或忽大忽小，Norm 在每层入口把尺度压回正常，否则深层梯度容易爆炸/消失、训练不收敛。

#### 关键澄清：Norm 归一化的是"数值"，不是"权重"——它不干预训练，本身就是训练的一部分

初学最容易卡在这里：**"归一化不是把参数改了吗？改参数岂不是干扰了训练？"** 这是把两样东西搞混了：

- **它归一化的对象是"激活值"，不是"权重"。** Norm 作用在**前向传播时流过网络的那个向量 x**（残差流里的中间结果），把这一串数字的尺度拉正。它**完全不碰** W_Q / W_up 这些权重矩阵——权重一个数都没动。
- **γ、β 虽然是参数，但它们和别的权重一样是被训练"学"出来的**，随反向传播一起更新，不是外部强行设定的。模型可以自己学到"这一层归一化之后要再放大还是缩小"。
- **所以非但不"干预"训练，反而是训练能进行下去的前提。** Norm 是计算图里一个普通的前向算子，梯度照常穿过它；它让数值不爆炸、让梯度稳定——是在**帮**训练收敛，不是打断它。

> 一句话：**归一化改的是"流过的数据的尺度"，不是"模型的知识（权重）"。** 知识全在权重里，权重只由"前向 → 算损失 → 反向 → 更新"这条主线改写；Norm 只负责把每层入口的数据尺度调稳，让这条主线能稳稳跑下去。

#### 一个具体例子：归一化把什么数变成什么数

假设残差流累加到某层时，一个 token 的向量（取 4 维示意）数值已经偏大、不齐：

```
输入 x = [ 100.0, -98.0, 102.0, -101.0 ]       ← 均值≈0.75，标准差≈100（尺度很大）

第①步 归一化（减均值、除标准差）：
   x̂ = [ 0.99, -0.99, 1.01, -1.02 ]            ← 均值≈0，方差≈1（尺度被拉正）

第②步 用可学习的 γ、β 缩放平移（设 γ=[2,2,2,2]，β=[0,0,0,0]，这俩是训练学到的）：
   y = γ·x̂ + β = [ 1.98, -1.98, 2.02, -2.04 ]  ← 交给后面的 Attention/MLP
```

- **变的是数值的"尺度/分布"**：从"忽大忽小、标准差 100"变成"均值 0、方差 1 再按 γ/β 微调"。
- **没变的是"方向/相对关系"**：四个分量的正负、大小顺序基本保持——所以语义（向量指向哪）没被破坏，只是把数值调到了好训练的范围。
- γ、β 是这一层专属的两个长度 H 的向量，由训练决定"归一化之后还要不要再放大/平移一点"——这正是"参数是被学出来的，而非干预训练"的体现。

> （RMSNorm 是简化版：只除以均方根、不减均值、只有 γ 没有 β，思路相同。）

## 2. 首尾的特殊层：embedding 与 head 不计入层数

主干的 block 是同构的，但**首尾两层结构不同，且不算进"层数"**：

```
embedding（输入层，token ID → 向量）        ← 不算层数
  ├─ block 1   ┐  （内部含 Norm）
  ├─ block 2   ├─ 只有这些算 "层数"，如 LLaMA-7B = 32
  ├─ ...       │
  └─ block N   ┘
final norm（最后一层归一化）                ← 不算层数
head / lm_head（输出层，向量 → 词表 logits）  ← 不算层数
```

- demo 里用**不同颜色**区分：embedding 黄、attn 蓝、mlp 绿、norm 灰、head 紫。
- 它们结构不同于可堆叠的同构 block，所以不计入"层数"。

### embedding 层：把"token ID"翻译成"向量"

- **功能**：模型不认识文字，输入先被分词器切成一串整数 token ID（如 `"中国"→[15, 872]`）。embedding 层持有一张大查找表 `W_emb [V×H]`（V=词表大小，H=隐藏维度），**用 token ID 去表里取出对应的那一行向量**——这是 lookup（gather），不是矩阵乘法。
- **产出**：`x0`，也就是第 3 节"残差流"的起点。
- **为什么不算层**：它没有 Attention/MLP 那套结构，只是一次查表；且每个 token 独立取向量，不做 token 间的交互。它是所有 block 的"输入准备"，不是可堆叠的同构计算单元。
- **冷知识**：很多模型让 embedding 和 head 共享同一块权重（weight tying，见第二部分），入口查表、出口投影是同一张表的两种用法。

### 多模态输入：图片 / 视频 / 音频怎么变成向量序列

上面只讲了文字。但 Transformer 主干（block）**根本不关心模态**——它只吃一个形状为 `[seq, H]` 的向量序列（seq 个 token，每个 H 维）。**所有模态差异都被关在"输入前端"里：怎么把原始信号变成这串 H 维向量。** 进了 block 之后，文本、图像、视频、音频对 Transformer 来说都只是"一串 H 维向量"，处理逻辑完全一样。

把各种前端归成两大类：

| 类型 | 思路 | 有没有"查表" | 典型模态 |
|---|---|---|---|
| **A. 离散化 + 查表** | 先把信号变成一串**整数 ID**，再用 `W_emb [V×H]` 查表取向量（和文字完全一样） | 有 | 文本；用编解码器量化后的音频 |
| **B. 切块 + 线性投影** | 把信号切成小块，每块拉平成一个长向量，过一个 `Linear` **直接投影**到 H 维 | 无（连续投影，无离散 ID） | 图像、视频、原始音频频谱 |

各模态具体怎么做：

- **文字**：分词器 → token ID → `W_emb` 查表（A 类，前面已讲）。
- **图片（ViT 范式，B 类）**：把图切成固定大小的 **patch**（如 224×224 图切成 16×16 的小块 → 共 196 块）；每个 patch 拉平（16×16×3=768 个像素值）过一个 `Linear` 投到 H 维 → 得到 196 个"图像 token"。**没有词表、没有查表**，是连续投影。再加 **2D 位置编码**（告诉模型每块在图里的行列位置）。
- **视频（B 类，多一个时间维）**：在图片基础上加时间。要么**逐帧**切 patch 再附加"第几帧"的时序位置编码；要么用 **3D patch / tubelet**（把"几帧 × 一小块空间"当成一个时空立方体）一起切。代价是序列长度暴涨（帧数 × 每帧 patch 数），所以视频模型常配合下采样、抽帧或上下文并行 CP（见第三部分）。
- **音频**：两条路都有。① 波形 → **梅尔频谱图**，当成"图像"切 patch 投影（Whisper 用一段卷积前端做这步，B 类）；② 用神经音频编解码器（EnCodec / SoundStream）把音频**量化成离散音频 token** → 查表（走 A 类，和文字同构）。

**多模态大模型怎么把它们拼到一起**（如 LLaVA、Qwen-VL）：

```
图像 → 视觉 encoder(ViT) → 一串图像向量 ┐
                                        ├─ projector/adapter 对齐到同一个 H 空间
文本 → 分词 → 查表 → 一串文本向量      ┘
                ↓
   拼成一条序列：[文本…  <image> 图像向量…  文本…]
                ↓
        喂给同一个 Transformer 主干（block 不变）
```

- 每个模态先用各自的 encoder 把信号变成向量，再用一个小的 **projector**（通常就是一两层 Linear/MLP）把维度和语义空间对齐到 LLM 的 H；
- 然后和文本 token 向量**拼成一条序列**，用特殊分隔 token（如 `<image>`）标界，之后交给同一个主干统一处理。

> **一句话归纳**：模态适配只发生在"原始信号 → `[seq, H]` 向量序列"这一步，本质是把文字的"分词 + embedding"换成对应模态的"切块/量化 + 投影"。这也正说明第 2 节为什么把 embedding 称为"输入准备、不算层数"——它是**可按模态替换的前端**，而 block 主干始终不变。

### norm 层（LayerNorm / RMSNorm）：稳定数值，让训练能收敛

- **功能与原理（含归一化到底改了什么、为什么不干预训练、具体数值例子）**：已统一讲在第 1 节"嵌套图里的两个配角：残差 + Norm"。简言之，它归一化的是**流过的激活值（不是权重）**，把残差流累加后失控的数值尺度拉回稳定分布，是残差流上的"稳压器"。本节只看它和"层数"的关系。
- **它在哪**：norm 主要**藏在每个 block 内部**（Attention 前一个、MLP 前一个，即 Pre-Norm），是 block 的一部分 → 这部分 norm **随 block 一起被计入层数**。此外主干**最后还有一个独立的 final norm**（送进 head 之前）→ 这个**不计入层数**。
- **为什么单独的 norm 不算"一层"**：它没有权重矩阵（只有 γ、β 两个长度 H 的**向量**），是逐元素乘加而非 MatMul，参数量极小，算力可忽略。它是辅助性的"调节器"，不承担"理解/精炼"职责，所以从不单独算作一层。

> **一句话归纳**：
> - **embedding**：入口查表，token ID → 向量，**不算层数**。
> - **block 内部的 norm**：稳压器，是 block 的一部分，**随 block 计入层数**。
> - **final norm**：送 head 前的最后一次稳压，**不算层数**。
> - **head**：出口投影，向量 → 词表分数，**不算层数**。
> - 数"多少层"时，只数中间那些 `Attn+MLP` 的 block。

## 3. Transformer 层到底在做什么（从一个例子入手）

> 这是理解 Transformer 最关键的一节。前两节讲了"层是什么、怎么数"；这一节讲**层真正干活时在做什么**。不从公式入手，先看一个例子，再拆开背后的三个问题。

### 先看例子：输入"中国的首都是"，模型怎么吐出"北京"

把 N 层想象成一条流水线，每层在前一层的基础上，把这句话的理解**多精炼一小步**：

```
输入："中国的首都是"
  浅层：识别这是中文、"首都"是个名词、"中国"是个实体
    ↓（理解被一层层往下传）
  中层：建立 "中国 ←→ 首都" 的关系，把这两个词关联起来
    ↓
  深层：收敛成"该输出一个城市名，而且是中国的首都"
    ↓
  出口 head：在所有词里给"北京"打最高分 → 输出"北京"
```

没有哪一层独自答出"北京"。是 N 层每层各推进一小步、**叠起来**才把"一串词"雕成"足以预测下一个词"的状态。下面拆这条流水线的三个关键问题。

### 问题一：最后那个"出口"在思考吗？——不，head 只负责翻译

很多人以为输出答案的 head 是"大脑"。恰恰相反：

- **前面 N 层负责"想"**：逐层精炼，最后一层吐出一个向量（如 4096 个数），里面**隐含**了答案，但还是一串数字。
- **head 只负责"说"**：把这个向量乘一个大矩阵，投影到**整个词表**，每个词得一个分数(logits) → softmax 成概率 → 取最高。几乎是机械翻译，不"思考"。

> 比喻：前面 N 层像**侦探破案**（整合线索、得出结论），head 只是**把结论念出来**（翻译成具体的词）。预测准不准，全看前面 N 层把表示雕得好不好。

### 问题二：层与层怎么接力？——残差流（一条被不断增补的信息流）

别把多层想成"每层把活重做一遍"。正确的图景是**一条信息流被逐层增补**：

```
x0 (词向量，来自 embedding)
  → 层1: x1 = x0 + Δ1     ← 层1 只往流里"加"它算出的修正量 Δ1
  → 层2: x2 = x1 + Δ2     ← 层2 在 x1 基础上再加 Δ2
  → ...
  → 层N: xN = x(N-1) + ΔN
  → head: 用 xN 做预测
```

注意那个 `+`（残差连接）。每层**读当前的流 → 算一个增量 Δ → 加回流里**，信息是**累积式**前进的：

- **前一层**：把它理解到的东西写进流里，作为后一层的输入。
- **后一层**：站在前一层的肩膀上，读取需要的信息，再补充新理解。

所以"上一层 vs 下一层"不是流水线上各干各的工序，而是**共享同一条流、后层基于前层继续精炼**的接力。（这个 `+` 还有第二个作用：让梯度几乎无损地传回浅层，是深层网络训得动的关键，见问题三。）

> 💡 每个 Δ 具体怎么算出来（Attention + MLP），是更细的机制，紧接的第 4 节展开；这里先记住一句"层 = 往残差流里加一个增量"。

### 问题三：每层长得一模一样，凭什么分工不同、又是谁安排的？

观察 demo：每个 block 画出来都是 `Attention + MLP`，结构完全相同。但**"结构相同" ≠ "功能相同"**，分两层看：

- **(a) 结构（算子种类 + 连接方式）= 人设计的，每层一样、写死在代码里。** 正因如此才能用一个 `for` 循环堆 N 个相同 block，demo 也才能用一个滑块控制层数。
- **(b) 内容（权重数值）= 训练学出来的，每层各不相同。** 每层都用同样的 `Linear`，但**权重矩阵的数值各自独立、不共享**。于是涌现出"由浅入深"的分工，正好对应上面例子里的浅/中/深层：

| 位置 | 倾向学到的东西 | 在"中国的首都是"例子里 |
|------|------|------|
| 浅层（近输入） | 词形、邻近词、局部语法 | 认字、断词 |
| 中层 | 句法结构、指代、实体关系 | 建立"中国 ↔ 首都"关系 |
| 深层（近输出） | 抽象、长程推理、为预测整合 | 收敛到"中国的首都" |

**这种分工不是设计者规定的，是训练逼出来的。** 关键在于：**所有层不是各自单独训练，而是被同一个损失、一次反向传播联合更新**——

```
① 前向    x0 → 层1 → ... → 层N → head → 预测分布
② 算损失  预测 vs 真实下一个词 → loss（错了多少）
③ 反向    把 loss 的"责任"从 head 一路倒推、分摊到每层每个权重（梯度）
④ 更新    每层权重 = 旧权重 − 学习率 × 梯度
⑤ 换下一批数据，重复几十万~几百万次
```

- **责任连带**：若深层因缺某信息而预测错，反向传播会发现"是浅层没把它写进流里"，于是梯度去调浅层。**浅层被"逼着"提供深层需要的东西**——分工是梯度协商出来的，不是设计的。
- **残差让梯度好传**：问题二那个 `+` 不光累积信息，也让梯度几乎无损倒流回浅层，不因层多而梯度消失。这是深层 Transformer 训得动的工程关键。
- **海量数据**：每批数据把每层往"整体预测更准"推一点点，loss 平均意义上单调下降 = "收敛精度"。

> ⚠️ **误区**：以为"结构同构"就是"每层完全一样、可随便换顺序"。其实交换层序会让性能明显变差——各层确实承担了有顺序依赖的不同角色。（"每层完全一样"只在 ALBERT 等层间共享权重的特例里出现；GPT/LLaMA 等主流模型每层权重独立。）

> **核心一句**：没有谁规定第 3 层管句法、第 20 层管推理。是几十万次"预测错 → 反向传播 → 微调每层"，自发地把不同职责压到了不同层上。这和 CNN 同理——浅层学边缘、深层学物体，**相同的积木 + 不同的权重 → 涌现不同角色**。

## 4. 层内部细节：Attention 与 MLP 两个子层（Δ 到底怎么算）

第 3 节把每层抽象成"读残差流 → 算一个增量 Δ → 加回去"，但 **Δ 具体怎么算**？打开一个 block，它做两件大事：**Attention（横向混合信息）** 和 **MLP（各自纵向加工）**。向量进、向量出，长度始终是 H（hidden size，第 7 节详解），内容被改写。

### Attention —— 让每个 token "去看别的 token"

进来时每个 token 的向量只装着"它自己"。Attention 让它**按需把其他 token 的信息加权搬一部分到自己身上**。在一个 token 位置上：

```
我的向量 → 拆成三个角色（各乘一个权重矩阵）：
  Q (Query 我想找什么)        = 向量 × W_Q
  K (Key   我能被找到的标签)   = 向量 × W_K
  V (Value 我能提供的内容)     = 向量 × W_V
```

然后：

1. 用我的 **Q** 和句中每个 token 的 **K** 做点积 → "我该多关注谁"的分数
2. 分数过 **Softmax** → 一组加权比例（和为 1）
3. 按比例把大家的 **V** 加权求和 → 我"看完全场"的新信息
4. 再乘输出矩阵 **W_O**，加回我原来的向量（残差）

> 直觉："它 很 好吃"里，"它"的 Q 和"好吃"的 K 高度匹配 → 把"好吃"的 V 搬过来 → "它"的向量被写进"我指代的是个能吃的东西"。**这一步是 token 之间横向的信息流动。** 对应几何：把"它"这个点朝"食物"方向挪了一下。

### MLP —— 让每个 token "自己消化一下"

Attention 是横向混合，MLP 则是**每个 token 各自独立**深加工（不看别人）：

```
向量(H) --up_proj--> 放大到(4H) --非线性(SiLU/GELU)--> --down_proj--> 缩回(H)
```

先升维到 4 倍，在更高维空间用非线性做一次复杂变换，再压回原维度。直觉上是对刚收集到的信息做**提炼、组合、抽象**。

### 两个配角：残差 + Norm —— 见第 1 节

每个子层除了主角（Attention / MLP），还各挂着 **残差连接**（`新向量 = 老向量 + 增量Δ`，不断修订而非推倒重来）和 **Norm（稳压器，归一化的是激活值而非权重）**。这两个配角的完整原理、"会不会干预训练"的澄清、以及一个具体数值例子，已统一讲在**第 1 节"嵌套图里的两个配角：残差 + Norm"**，此处不再重复，只在下面的完整动作里看它们的位置。

### 合起来：一层的完整动作

```
x → Norm → Attention → +x（残差）→ Norm → MLP → +x（残差）→ 输出
          ↑横向：token 互相看            ↑纵向：各自深加工
```

堆 N 层 = 把"通气—琢磨"重复 N 轮，每轮把每个词的点朝更符合上下文的位置挪一挪，理解越来越深。

### 算子级视角：把两个子层拆成最小算子

把上面两步再拆细，就是 demo 右侧"整网算子图"里看到的**前向算子序列**（这里是不含并行切分的纯前向；每个 `Linear` 背后对应一个权重矩阵，完整清单与显存见第二部分，多卡切分见第二、三部分）：

```
Attention 子层（前向）
x [B, seq, H]
  → QKV Linear（W_Q/W_K/W_V，各 [H×H]）→ Q/K/V [B, seq, H]
  → Scaled Dot-Product Attention（无权重：QKᵀ → Softmax → 加权·V）
  → Out Proj（W_O [H×H]）
  → x + 残差 [B, seq, H]
```

```
MLP 子层（前向，SwiGLU）
x [B, seq, H]
  → up_proj（W_up [H×4H]）→ [B, seq, 4H]
  → SiLU（无权重）
  → down_proj（W_dn [4H×H]）
  → x + 残差 [B, seq, H]
```

- 末尾的 `x + 残差` 就是第 3 节"残差流"的落地：每个子层只产出一个增量、加回主流。
- 带权重的算子（QKV/Out/up/down 这些 `Linear`）才占显存；无权重的纯计算（Softmax、SiLU、残差加）不持有参数——第二部分正是用这条线索解释"为什么只有带权重的层才画成格子、才占显存"。
- 权重矩阵尺寸：`W_Q/W_K/W_V/W_O` 都约 `H×H`，`W_up/W_dn` 是 `H×4H`/`4H×H`。**这就是 hidden size 调大、参数按约平方增长的根源**（也对应 demo 里 hidden size 表单的 `Attention QKV Linear：H×H`、`MLP up_proj：H×4H`）。

> **一句话归纳**：一层 = 先让每个词互相通气（Attention 横向混合），再让每个词自己琢磨（MLP 纵向提炼），结果以增量叠回残差流。第 3 节的 Δ 就是这套算出来的；第二部分接着把这些权重矩阵的数量、显存占用、多卡切分系统摊开。

## 5. 理解了 Transformer 层，再看它和 CNN 的差异（最常见的误区）

> 现在你已经知道 Transformer 层在做什么（Attention 让 token 互相看、逐层精炼），正好可以澄清一个高频误区：卷积、池化是不是 Transformer 里的层？

> ⚠️ **核心误区**：以为卷积、池化是 Transformer 里的某种层。
> **真相：卷积/池化属于 CNN（卷积网络）；Attention/MLP 属于 Transformer。它们是两个并列的网络"家族"，不互相嵌套。Transformer 层里没有卷积核，也没有池化层。**

把它们当作两个家族里**互相替代的对应物**来看最清楚：

| | CNN（卷积网络） | Transformer |
|---|---|---|
| 核心运算 | **卷积**（卷积核滑窗） | **自注意力 Attention** |
| 基本重复单元 | 卷积块（Conv+BN+激活+池化） | Transformer block（Attn+MLP） |
| 怎么"看"数据 | 只看**局部**邻域（核覆盖的窗口） | 看**全局**所有 token 两两关联 |
| 标志性零件 | 卷积核 | Attention |
| 典型场景 | 图像（ResNet/VGG） | 序列/文本（GPT/LLaMA），也做图像(ViT) |

- **卷积核 ↔ Attention**：是各自家族里"那个标志性的运算"，平行替代关系，不是上下级。
- **卷积层、池化层只出现在 CNN 里**；标准 Transformer 层里一个都没有。
- 同一个任务通常**要么用 CNN，要么用 Transformer**，是两套并列方案。

> 比喻：卷积/池化是"柴油发动机的零件"，Attention/MLP 是"电动机的零件"。你不会在电动机里找到喷油嘴——两种引擎，各装各的零件。
>
> （例外，了解即可：Conformer、ConvNeXt 等**混合架构**会特意混搭卷积与 Attention，但那不是标准 Transformer 的默认结构。）

> 💡 第 3 节"由浅入深的分工"那段也提到 CNN 是一个道理：浅层学边缘、深层学物体——两个家族在"逐层抽象"这件事上相通，区别只在那个标志性零件（卷积 vs Attention）。

## 6. 层数怎么定 + 设多/设少的影响

### 业务上怎么确定层数

层数**不是从零推导的，是"选档 + 受约束 + 实验调"出来的**。先看标准档位：

| 量级 | 层数(典型) | 代表 |
|------|------|------|
| 小 | 6–12 | BERT-base(12)、GPT-2 small(12) |
| 中 | 24 | BERT-large、GPT-2 medium |
| 大 | 32–80 | LLaMA-7B(32)、13B(40)、70B(80) |
| 超大 | 96+ | GPT-3 175B(96) |

真正卡住层数的约束：
- **算力/显存/延迟（最硬）**：层数 × 每层参数 ≈ 显存与延迟。线上常"延迟反推层数"：定 P99 延迟上限 → 反推能跑多深。
- **数据量**：深模型要更多数据喂饱；数据少时堆层易过拟合（参考 Chinchilla：给定算力，模型大小与数据量要平衡）。
- **任务复杂度**：分类/检索/NER 等 6–12 层常够（甚至蒸馏到 2–6 层）；多步推理/长文本生成需要更深。
- **深 vs 宽权衡**：同参数量可深而窄或浅而宽。经验区间 `d_model ≈ 64 × 层数` 附近（这里的"宽"就是第 7 节的 hidden size H；层数是深、H 是宽，二者正交，按总参数预算分配）。

实际流程：定目标(任务+SLA+数据量) → 选成熟 baseline 沿用其层数 → 算力不够就降档(蒸馏/换小模型) → 在 12/24 等档位做对照实验，选性价比拐点 → 上线用蒸馏/剪枝压到满足 SLA 的最小层数。

### 案例：层数设多/设少，对 `"中国的首都是"→"北京"` 的影响

这个例子很简单，正好说明"层数不是越多越好"：

| 层数 | 效果 | 本质 |
|------|------|------|
| **太少**(1~2) | 来不及建立"中国↔首都↔北京"关系链，可能预测出泛泛的词或搞错国家 | 深度不足 → **欠拟合** |
| **刚好**(12~32) | 有足够步数走完"认字→建立关系→收敛到北京"，稳稳答对 | 容量匹配 |
| **太多**(100 去学这种简单题) | 答案不变，但浪费算力/显存、延迟高、训练更难、数据不够时**过拟合**、边际收益递减(后面的层空转) | 容量过剩 |

> **关键直觉**：层数要匹配任务的"推理步数复杂度"。
> - `"中国的首都是"` 是**一步事实检索**，少量层就够。
> - `"把所有质数相加再除以 3 余几……"` 是**多步推理**，需要更多层一步步推。
>
> 同一个模型的层数按**最难的那类任务**定上限（要能多步推理），对"北京"这种简单题就是绰绰有余、用不满。**按最复杂业务需求定上限，简单任务自然覆盖。**

## 7. hidden size：4096 维向量到底是什么，浮点数怎么装语义

前面一直说"向量""残差流"，但**每个 token 的向量到底长什么样、那几千个数字凭什么代表语义**，是初学最绕的一个坎。先把概念对齐：

- **hidden size（隐藏维度 H）= 每个 token 在模型内部被表示成的向量有多长。** H=4096 就是每个 token 用 4096 个浮点数表示。它是"模型宽度"，和"层数"（模型深度）正交：层数是流水线多长，hidden size 是流水线多宽。两者在总参数预算下相互权衡（第 6 节"深 vs 宽"：经验区间 `d_model ≈ 64 × 层数`）——同样的参数量，可以深而窄，也可以浅而宽。
- 这个向量从 embedding 查表产出（第 2 节的 `x0`），在残差流里一路被改写（第 3 节），长度自始至终是 H 不变。

### 关键结论：语义不在"某一维"，而在"整个向量的方向"

一个 token 的向量就是一串没有标签的浮点数：

```
[ 0.23, -1.07, 0.0004, 2.55, -0.88, ... ]   ← 一共 4096 个数
   ↑第0维  ↑第1维         ↑第3维   （每一维只是个数，没有人类可命名的含义）
```

> ⚠️ **核心误区**：以为"第 37 维 = 表示是不是动物"这种一维一义的对应。
> **真相：单看任何一个维度基本看不懂。语义不是装在某个维度里，而是体现在"整个 4096 维向量指向哪个方向、和别的词的向量离多近"。**

正确的图景是：把每个 token 看成 **4096 维空间里的一个点**，语义来自这些点的**相对几何关系**：

- **意思相近的词，点离得近**（夹角小）。"猫/狗/仓鼠"聚成一团，"积分/矩阵"聚成另一团。
- **某个属性 ≈ 空间里的一个方向**（一组维度的线性组合，不是单维）。经典例子：
  ```
  向量(国王) − 向量(男人) + 向量(女人) ≈ 向量(女王)
  ```
  说明"性别"大致对应一个方向；"复数""过去时""褒贬"各自也是一个方向。

所以"浮点数怎么装语义"的答案是：**单个浮点数不装语义；4096 个数共同确定了这个词在语义空间里的位置与朝向，语义来自它与其他词位置的关系。** 类比地图：一个城市的"经度"本身没意义，是"经纬度合起来 + 和别的城市的相对位置"才有意义。

### 为什么要 4096 维这么多

维度多 = 空间能容纳更多**互不干扰（近似正交）的方向**。

- 维度太少（如 2 维），所有词挤在一张纸上，"词性/时态/情感/主题"这些属性会互相冲突、没法同时分清。
- 4096 维里能塞下成千上万个互不打架的方向，模型可以**同时**用不同方向编码"词性、指代、情感、主题、句法角色……"，各管各的。

这就是 hidden size 的几何意义：**维度数 = 语义空间里能同时容纳多少条独立的"语义坐标轴"。** 维度越大 → 表示容量越大 → 潜在能编码的语义差别越多（但能否真用上取决于数据量与训练是否充分，且参数量、算力随之按约平方增长，见第 4 节与第二部分的 `H×H`、`H×4H`）。

### 向量不是固定的：同词随上下文挪位

同一个词"苹果"，在"我吃苹果"和"苹果发布会"里 embedding 查表给的初始向量一样，但经过每层加工后被改写到不同位置——一个飘向"水果"区，一个飘向"公司"区。**这正是第 4 节"层内加工"在做的事**，也是第 3 节残差流"逐层精炼"的几何说法。

> **一句话归纳**：向量 = 词在语义空间里的坐标；hidden size = 这个空间有多少维（多少条语义坐标轴）；语义藏在坐标之间的相对方向里，不在任何单个维度里。

# 第二部分 · 层、权重矩阵、算子的关系（深入层内部）

第一部分第 4 节已从"做什么"打开了一个 block（Attention/MLP 的前向算子序列）；这一部分换个角度，系统看它由哪些**权重矩阵**组成、各占多少显存、又如何被切到多张卡——这正是 demo 右侧"整网算子图"和显存估算的依据。

## 层级结构

```
Transformer Block
├── Attention 层
│   ├── 权重矩阵：W_Q [H×H], W_K [H×H], W_V [H×H], W_O [H×H]
│   └── 算子：QKV MatMul, Scaled Dot-Product, Out MatMul, all-reduce, 残差加
├── MLP 层（标准）
│   ├── 权重矩阵：W_up [H×4H], W_dn [4H×H]
│   └── 算子：up MatMul, SiLU, down MatMul, all-reduce, 残差加
├── Embedding 层
│   ├── 权重矩阵：W_emb [V×H]
│   └── 算子：Embedding lookup（gather，非 MatMul）
└── LM Head 层
    ├── 权重矩阵：W_head [H×V]（通常与 W_emb 共享）
    └── 算子：lm_head MatMul, all-gather（若 TP>1）
```

---

## 一个层有几个权重矩阵？

**0 个、1 个、多个都有可能**，取决于层的类型：

| 权重矩阵数量 | 层类型 | 说明 |
|---|---|---|
| **0 个** | Softmax、SiLU、ReLU、残差加法、Dropout | 纯计算，无可学习参数 |
| **1 个** | Embedding、LM Head | 一个大矩阵做 lookup 或投影 |
| **2 个** | 标准 MLP | W_up + W_dn |
| **3 个** | SwiGLU MLP（LLaMA 风格） | W_gate + W_up + W_dn |
| **4 个** | Attention（非融合） | W_Q + W_K + W_V + W_O |
| **4 + 2×E 个** | MoE 层 | W_router + E 个专家（每个专家 2 个矩阵） |

> LayerNorm/RMSNorm 有 γ、β 两个参数向量（长度 H），但它们是**向量不是矩阵**，对应逐元素乘加而非 MatMul，参数量极小，通常单独讨论。

> 💡 呼应第一部分第 3 节："结构相同"指的就是这张表里**算子种类和连接方式**每层一样；"功能不同"指的是这些 W_Q/W_up… 里的**数值**每层经训练而分化。

---

## 权重矩阵与算子的对应关系

基本规则：**1 权重矩阵 : 1 MatMul 算子**（前向 pass）

| 对应关系 | 场景 | 例子 |
|---|---|---|
| 1 权重 : 1 算子 | 标准情况 | W_up → up_proj MatMul |
| 多权重 : 1 算子 | 融合 kernel | W_Q/K/V 拼成 [H×3H] → 一次 fused QKV MatMul |
| 1 权重 : 多算子 | Weight tying | W_emb = W_head，前端 lookup + 后端 lm_head 各用一次 |
| 0 权重 : 1 算子 | 纯计算 | Softmax, SiLU, all-reduce, 残差加法 |

---

## Attention / MLP 的前向算子序列 —— 见第一部分第 4 节

> 这两个子层"做什么"以及"拆成哪些前向算子"（Attention：QKV Linear → Scaled Dot-Product → Out Proj；MLP：up_proj → SiLU → down_proj，含残差/Norm）已整合在**第一部分第 4 节"算子级视角"**，此处不再重复前向流程。本部分聚焦它们的**权重矩阵清单、显存、多卡切分**——前者见上面"层级结构"，切分见下面"Tensor Parallel 下的变化"。

## 一个权重矩阵在完整训练 step 中被多少个算子访问

demo 里的算子图只展示**前向** pass，但一次完整 training step 里，同一个 W 会被 4 个算子依次访问：

```
前向    output = x @ W          → 消费 W，产出激活值
反向①   grad_x = grad_out @ Wᵀ → 消费 W，产出传给上一层的梯度
反向②   grad_W = xᵀ @ grad_out → 读 x，写 grad_W（梯度累积）
优化器  W -= lr · Adam(grad_W, m, v) → 读写 W、m、v
```

> 💡 这正是第一部分第 3 节"联合训练"的微观细节：反向①把"责任"往上一层传（梯度倒流），反向②算出本层该怎么改。残差连接保证 `grad_x` 能几乎无损传回浅层。

这解释了为什么混合精度训练下每个参数需要 **16 B/param**：
- bf16 权重 2B + fp32 主权重 4B + Adam m 4B + Adam v 4B + 梯度 2B = 16B

## Tensor Parallel 下的变化

TP 把权重矩阵按列或行切分到 tp 张卡上，每张卡持有一个 shard：

- **列切（Column Parallel）**：W[:, t·k:(t+1)·k]，各卡算部分输出，最后 all-gather 拼合
- **行切（Row Parallel）**：W[t·k:(t+1)·k, :]，各卡算部分和，最后 all-reduce 求和

原本 1 个逻辑 MatMul 算子，在 TP 下变成：`tp 个并行 shard-MatMul + 1 个通信算子`

具体到第一部分第 4 节那两个子层（`/tp` = 每卡只持一份分片）：

```
Attention：QKV Linear [H×H/tp]（列切）→ Scaled Dot-Product → Out Proj [H/tp×H]（行切）→ all-reduce(仅 TP>1) → +残差
MLP：      up_proj [H×4H/tp]（列切）→ SiLU → down_proj [4H/tp×H]（行切）→ all-reduce(仅 TP>1) → +残差
```

规律：**子层里前一个 Linear 列切、后一个行切**，中间不通信，只在子层末尾 all-reduce 合并一次。

---

## MoE 层特殊性

```
x [B, seq, H]
      ↓
W_router [H × E] → gate / router（决定每个 token 去哪个专家）
      ↓  top-k dispatch
all-to-all dispatch            ← 仅 EP>1
      ↓
W_exp_i [H × 4H] → Expert_i FFN（只有被选中的 token 流经）
      ↓
all-to-all combine             ← 仅 EP>1
      ↓
x + residual [B, seq, H]
```

MoE 的 1 个专家 = 2 个 MatMul（up_proj + down_proj），但每个 token 只激活 top-k 个专家（通常 k=2），所以是**稀疏激活**。

---

## Weight Tying（W_emb = W_head）

LLaMA 等模型中 Embedding 权重和 LM Head 权重共享同一块显存：

```
W_emb [V × H]  ──── 前向入口：token ID → 向量
      ↑ 同一块显存（转置关系）
W_head[H × V]  ──── 前向出口：隐状态 → logits
```

好处：节省 V×H 参数量（约 32000×4096 ≈ 128M 参数 ≈ 0.25GB @ bf16）

在 TP 下两者都做列切分，切法对称，梯度在 all-gather 后自动合并。

## 拓扑图里"落在卡上的格子"为什么代表权重参数

中间拓扑图中，每张卡（`.wdiag-card`）里竖排的每个**格子**（`.wdiag-layer-box`）标签写的是层名（Embedding / Attn-0 / MLP-0 / Norm / Head），但它在显存与资源意义上等价于**这一层在本卡上持有的那份权重（或权重分片）**。三个原因：

1. **一张卡承载什么 = 它存了哪些权重。** 显存估算只累加各层的权重参数量：attn `4·H²/tp`（W_Q/K/V/O）、mlp `8·H²/tp`（up+down）、emb/head `V·H/tp`。"层落在哪张卡"被直接换算成"这张卡存了多少权重"。
2. **层的"身份"几乎就是它的权重。** 有权重的层（Embedding/Attn/MLP/Head）必须把权重矩阵放进某张卡显存，才占资源、才需要被并行切分；无权重的纯计算层（Softmax/SiLU/残差加/all-reduce）不占参数显存，因此**不画成格子**。
3. **TP 分片时格子标签直接写"第 t/tp 片"，切的就是权重矩阵**（列切/行切）。悬停格子弹出的明细把名字改成"…权重矩阵"，是刻意点明：格子 = 该层在本卡上持有的权重分片。

> 一句话：格子标签写的是"层名"，编码含义是"这张卡要存这一层的权重（或其一个分片）"。

# 第三部分 · 并行训练（权重如何切到多张卡）

第二部分说清了"一个 block 有哪些权重"；当模型大到单卡放不下，就要把这些权重切开放到多张卡上——这就是并行训练。

## 1. 一次训练 step 在算什么

目标：调整权重 **W**，让预测更准。每个 step 四步：

```
① 前向 forward    output = x @ W           用当前权重做预测
② 算损失 loss     比较预测与正确答案，得到误差
③ 反向 backward   grad_W = xᵀ @ grad_out   算出"W 该往哪改"
④ 优化器 optimizer W -= lr · Adam(grad_W)  真正更新权重
```

关键：同一个 W 在一个 step 里被访问 4 次（前向、反向算输入梯度、反向算权重梯度、优化器更新）。

> 💡 这是第一部分第 3 节训练循环的"单层权重视角"；那里讲的是整网联合收敛，这里聚焦单个 W。

## 2. 为什么显存爆炸 → 必须并行

混合精度训练下每个参数存 5 份东西，共 **16 B/param**：

| 项目 | 大小 | 用途 |
|---|---|---|
| bf16 权重 | 2B | 前向/反向计算 |
| fp32 主权重 | 4B | 高精度累积更新 |
| 梯度 | 2B | 反向产出的 grad_W |
| Adam m | 4B | 动量（一阶矩） |
| Adam v | 4B | 方差（二阶矩） |
| **合计** | **16B** | |

70B 模型 → 70B × 16 = **1.1 TB**，远超单卡 80GB。这就是并行训练的根本动机。

## 3. 四/五种并行（picotron 的 dp×pp×cp×tp 网格）

总卡数 World Size = DP × PP × CP × TP。每种切法切的东西不同，且**彼此正交可叠加**：

| 并行 | 切什么 | 直观理解 | 拓扑图表现 |
|---|---|---|---|
| **DP** 数据并行 | 切数据 batch | 每卡存完整模型副本，各算一部分样本 | 顶部 batch 分裂成 shard，横向多个 DP 副本 |
| **PP** 流水线并行 | 切层（深度方向） | 前几层放卡 A、后几层放卡 B，接力 | 纵向 Stage 0→1，`↓ PP` 箭头；层按连续均分分配 |
| **TP** 张量并行 | 切单个权重矩阵（宽度方向） | 一个大矩阵劈成 tp 份，各卡算一片再通信合并 | 同一行内 `↔` 连接的多卡，格子标"第 t/tp 片" |
| **CP** 上下文并行 | 切序列长度 | 长文本按 token 位置切给不同卡 | 紫色左边框的 CP 分组 |
| **EP** 专家并行 | 切 MoE 专家 | 不同专家放不同卡，token 按路由分发 | MoE 模式下专家按卡分配 + all-to-all |

> 💡 **PP 为什么按 block 整数切**：呼应第一部分第 1 节——block 是可堆叠的最小重复单元，每个 block 计算量/显存近似相同，所以按层整数切给不同 stage 最均匀。这也是"层数"以 block 为单位的工程价值。

> 💡 **CP 的典型场景＝长序列（尤其视频/长文本）**：序列越长，注意力的激活值显存随 seq 增长甚至按 `seq²` 涨，单卡放不下。把序列按 token 位置切给多卡，每卡只算自己那段，正是 CP 的用武之地。呼应第一部分第 2 节"多模态输入"——视频帧 patch 化后 `seq = 帧数 × 每帧 patch 数`，极易暴涨，是 CP 最常见的现实动机之一。

## 4. 通信规约（reduce）出现在哪——为什么不在 block 之间

初学常问：层与层之间会不会有 all-reduce / 规约？**先分清两种完全不同的"规约"：**

| | (a) 算子内部的数学求和 | (b) 分布式通信规约 |
|---|---|---|
| 例子 | Softmax 的分母求和、LayerNorm 算均值/方差 | all-reduce、reduce-scatter |
| 来自 | **Transformer 数学定义本身** | **并行切分**（张量切到多卡再合起来） |
| 单卡时 | 照样有 | **完全没有** |

你通常说的"reduce 环节"指 (b) 那种**通信规约**——它**不是 Transformer 架构产生的，是并行切分带出来的**。所以单卡跑模型时，层与层之间只有残差流的一个加法（第一部分第 3 节），**没有任何 all-reduce**。

**通信规约相对"层"到底在哪**（关键：几乎都不在 block 与 block 之间）：

| 并行 | 规约在哪 | 是不是"层间" |
|---|---|---|
| **单卡** | 没有通信规约，层间只有残差加 | — |
| **TP** | 每个**子层末尾**一次 all-reduce（Out Proj / down_proj 行切只算出"部分和"，必须求和）→ 一个 block 内 **2 次**（Attn、MLP 各 1 次） | 在 **block 内部、子层边界**，不是 block 之间 |
| **PP** | block 跨 stage 传递用**点对点 send/recv**，把激活值发给下一卡 | 是层间通信，但**是传递不是规约** |
| **DP** | 梯度 all-reduce，在**反向结束、优化器更新前**一次性跨副本规约全部参数的梯度 | 在 **step 末尾**，与"哪两层之间"无关 |

> **一句话**：规约不是 Transformer 层间的固有环节，而是并行切分的"缝合动作"——**TP 缝在子层末尾、DP 缝在 step 末尾，而真正的层间传递（PP）靠 send/recv 而非 reduce**。TP 的两次 all-reduce 见上面"Tensor Parallel 下的变化"那两行 `… → all-reduce(仅 TP>1) → +残差`。

---

## 5. 视图与业务的对应

- **格子的种类**（Attn/MLP/Emb/Head）由 PP 决定——这张卡分到哪几层
- **格子的分片标注**（第 t/tp 片）由 TP 决定——这层权重被横切给几张卡
- **格子的显存条** = 权重 × 16 B/param——这张卡装不装得下
- **横向重复的整块**由 DP 决定——同一份权重在多个数据副本上各存一份

> 并行训练的本质：决定"哪份权重、存在哪张卡、被切成几片"。格子就是这件事的可视化单元。

# 第四部分 · 性能瓶颈（计算 / 通信 / 内存三维度）

前三部分讲清了"模型由什么组成、怎么切到多卡"；这一部分回答工程上最实际的问题：**跑起来慢/放不下，瓶颈在哪个环节**。三个维度互相耦合，且常被误判。

## 1. 先建立判断框架：Roofline 与"假性算力瓶颈"

判断一个算子被**算力**还是被**内存带宽**卡住，看它的**算术强度**（每读 1 字节做多少次浮点运算）——这就是 Roofline（屋顶线）模型。

> ⚠️ **最常见误判**：以为"慢 = 算力不够"。实际上 Transformer 里**大量环节是内存带宽瓶颈**（算术强度低），堆算力没用，得靠**融合 kernel、减少 HBM 往返**。

- **算力 bound**：大矩阵乘（QKV/Out/up/down，约 `seq·H²`/层，见第一部分第 4 节）——算术强度高、跑满 Tensor Core，通常**不是**瓶颈（除非 batch 太小）。
- **带宽 bound**：逐元素/归约小算子（Softmax、LayerNorm、激活）、以及推理 decode——算术强度低，受限于从 HBM 读数据。

## 2. 计算维度（Compute）

| 典型瓶颈 | 发生环节 | 本质 |
|---|---|---|
| **Attention 的 `seq²` 复杂度** | 训练前向/反向、推理 prefill 的 `QKᵀ→Softmax→·V` | 长序列时 attention 的计算与中间分数矩阵都随 `seq²` 涨，主导开销 |
| **逐元素/归约小算子**（Softmax、LayerNorm、激活） | 每个子层 | 算术强度极低 → **带宽 bound**，不融合就被 kernel 启动 + HBM 往返拖死 |
| **推理 decode 阶段** | 自回归逐 token 生成 | GEMM 退化成 GEMV，每生成 1 token 要把**整个权重从 HBM 读一遍** → 带宽 bound，算力闲置 |
| **MoE 负载不均** | router 之后 | 热门专家被挤、冷专家空转，个别卡拖慢整体 |

- 缓解：FlashAttention（融合 attention、不落盘 `seq²` 矩阵）、kernel 融合、decode 用 MQA/GQA + 量化减少权重读取。

## 3. 通信维度（Communication）

| 典型瓶颈 | 发生环节 | 特点 |
|---|---|---|
| **TP all-reduce** | **每个子层末尾**（Attn、MLP 各 1 次，见第三部分第 4 节） | 频率高、在**关键路径**上（难重叠）；跨节点走慢速网络是灾难 → TP 一般**锁在单机内**（NVLink） |
| **DP 梯度 all-reduce** | **反向结束、优化器更新前** | 数据量大但可与反向计算**重叠**（分桶）。ZeRO/FSDP 还引入参数 all-gather + 梯度 reduce-scatter |
| **PP send/recv + 流水线气泡** | **stage 之间** | 传激活值是点对点；更痛的是 **bubble**（启动/收尾空闲），气泡占比 ≈ `(PP−1)/(microbatch 数 + PP − 1)` |
| **EP all-to-all** | MoE dispatch / combine | all-to-all 延迟敏感、扩展性差 |

> 核心矛盾：**通信很难和计算完全重叠**，暴露出来的那部分就是瓶颈；卡数越多、并行度越大，通信占比越高，最终成为扩展性天花板。这呼应第三部分第 4 节——规约不是架构固有的，是并行切分的"缝合动作"，缝得越多越贵。

## 4. 内存维度（Memory）

| 典型瓶颈 | 发生环节 | 说明 |
|---|---|---|
| **模型状态 16B/param** | 常驻 | 权重 2 + fp32 主权重 4 + 梯度 2 + Adam m/v 8（第三部分第 2 节）。70B→1.1TB → 必须 ZeRO/TP/PP 切分 |
| **激活值（activations）** | **前向存、反向用** | 随 `batch×seq×层数×H` 增长，长序列时主导；**反向时是显存峰值**（激活 + 梯度并存） |
| **Attention `seq²` 中间量** | attention 内部 | FlashAttention 之前，`seq²` 分数矩阵吃显存；长上下文最痛（第三部分 CP 的动机） |
| **KV cache（推理）** | decode 阶段 | 随 `batch×seq×层数×H` 线性涨，长上下文/高并发时主导推理显存 → PagedAttention、GQA |
| **HBM 带宽（不是容量）** | decode 每一步 | 见第 2 节，是 decode 的真正瓶颈 |

- 缓解：**激活重计算（gradient checkpointing）** 用算力换显存（前向不存、反向重算）；ZeRO/FSDP 分片模型状态；CP 切长序列。

## 5. 串起来：按"环节"看谁主导

| 环节 | 主导瓶颈 |
|---|---|
| 训练前向 | 激活显存累积 + TP all-reduce（每子层） |
| 训练反向 | **显存峰值** + 算力（反向 FLOPs ≈ 前向 2×）+ TP all-reduce |
| 梯度同步 / 优化器 | DP all-reduce（通信）+ 优化器状态显存 |
| 推理 prefill | 算力 bound（attention `seq²`，类似训练前向） |
| 推理 decode | **带宽 bound + KV cache 显存** |

> **一句话总结**：训练阶段瓶颈通常在**反向的显存峰值**与 **TP/DP 通信**；推理 prefill 是**算力（attention）**、decode 是**带宽 + KV cache**；而很多看似"算力不足"的问题，根子是**内存带宽与未融合的小算子**。

# 第五部分 · 训练 vs 推理：两种部署形态与各自的并行

> 前四部分的主线是**训练**（权重在变、要反向、要优化器）。但同一个模型还有另一个完全不同的生命阶段——**推理**（权重已固定、只做生成）。很多开源模型发布的是"推理包"，里面的"并行配置"和训练并行**不是一回事**，初学极易混淆。这一部分把两者彻底分开。

## 1. 训练 vs 推理：到底差在哪

| | **训练 (Training)** | **推理 (Inference)** |
|---|---|---|
| 在做什么 | 用海量数据**学习**，不断更新权重 W | W 已固定，**用模型回答/生成** |
| 计算 | 前向 + **反向** + 优化器更新（第三部分第 1 节四步） | **只有前向**，无反向、无优化器 |
| 每参数显存 | **16 B/param**（权重+主权重+梯度+Adam m/v，第三部分第 2 节） | 约 **2 B/param**（只有 bf16 权重）+ KV cache |
| 还需要什么 | 训练数据、loss、梯度、checkpoint、训练并行框架 | 权重 + 前向代码 + 一点采样/生成逻辑 |
| 谁来做 | 模型研发团队（如盘古团队，上千卡） | 部署方/使用者（少量卡甚至单卡） |
| 显存峰值在 | **反向**（激活+梯度并存，第四部分第 4 节） | **decode 的 KV cache**（第四部分第 4 节） |

**"推理包" = 只包含"跑模型做生成"所需文件的发布包。** 判断一个目录是不是推理包，看它**有没有反向/优化器/训练数据**：
- 有的：模型权重、配置（层数/hidden/MoE 结构）、前向实现（`model.py`）、生成脚本、推理并行配置（见下）、部署文档。
- 没有的：训练数据、loss、反向传播、优化器状态、训练启动脚本、训练并行（PP/DP/EP）配置。

> 💡 这解释了一个常见困惑：**从推理包里无法还原模型的训练并行策略**——那部分根本不在包里。能从推理包提取的只有①模型结构（config + model.py，对应第二部分的权重清单）②推理并行配置（下一节）。

## 2. 推理并行 vs 训练并行：切的目标完全不同

模型太大单卡放不下时，推理同样要**切到多张卡**，但目标和训练不一样：

| | **训练并行**（第三部分） | **推理并行** |
|---|---|---|
| 目标 | 高效**训练**：放下 16B/param 的庞大状态 + 喂海量数据 | 让模型**跑起来、跑得快**：放下权重 + 压低延迟 |
| 常用维度 | **DP × PP × CP × TP**（+MoE 的 EP）全家桶 | 主要 **TP**（张量并行），大模型/MoE 再加 **EP / DP** |
| 不需要的 | — | **DP 切数据**意义不同（推理是切请求/并发，非梯度同步）；**PP** 推理也可用但更少；**无梯度 all-reduce** |
| 通信 | TP 子层末尾 all-reduce + DP 梯度 all-reduce + PP send/recv（第三部分第 4 节） | **只剩前向那部分通信**：TP 的子层末尾 all-reduce/all-gather、EP 的 all-to-all；**没有梯度同步** |

**核心**：推理并行本质是把第三部分的并行切分**只保留前向的那一半**——因为推理没有反向、没有优化器、不做梯度规约。所以推理的 TP 切法和训练**完全一致**（列切/行切，见第二部分"Tensor Parallel 下的变化"），只是不再有 step 末尾的 DP 梯度 all-reduce。

### 一个真实例子：盘古 718B MoE 推理包的并行配置

```yaml
# runner_config/tp32.yaml —— 32 卡张量并行推理
parallel_config:
  attn_tp_size: 32    # 注意力 128 个头 → 每卡分 4 个头（W_Q/K/V/O 列切+行切）
  moe_tp_size: 32     # FFN/MoE intermediate → 每卡 1/32（up 列切、down 行切）
  embed_tp_size: 32   # 15 万词表 → 每卡 1/32（embedding/lm_head 列切）
```

- 这就是第二部分"Tensor Parallel 下的变化"在推理侧的落地：注意力按头切、FFN 按中间维切、embedding/head 按词表切，子层末尾用 all-reduce/all-gather 缝合。
- 注意配置里**没有** PP/DP/EP 的训练含义，也没有优化器/梯度——印证第 1 节"推理包不含训练并行策略"。

> **比喻**：一道超大的菜（一次前向），单个厨师（单卡）做不完 → 把食材切成 32 份，32 个厨师各做一份（TP 并行），关键步骤交换半成品（子层末尾 all-reduce），最后拼成整道菜（输出 logits）。训练则是在这之上还要"尝味道→记下哪里不对→改菜谱"（反向+优化器），那部分推理完全没有。

## 3. 一句话归纳

> - **训练包 vs 推理包**：看有没有反向/优化器/训练数据。推理包只够"拿来用"，**不含训练并行策略**。
> - **训练并行 vs 推理并行**：训练是 `DP×PP×CP×TP(+EP)` 全家桶（要扛 16B/param + 梯度同步）；推理只保留**前向那半**，以 **TP** 为主（大模型/MoE 加 EP/DP），**无梯度 all-reduce**。
> - 从一个推理包能提取的是**模型结构**（第二部分）和**推理 TP 配置**，提取不到**训练并行策略**——后者得看技术报告或训练框架（MindSpeed/Megatron）的配置。
