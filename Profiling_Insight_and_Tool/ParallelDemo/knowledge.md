# 贯穿全篇的案例：「我喜欢吃苹」→ 预测「果」

为了让后面每一节都有抓手，全文用一个**最小但完整**的例子贯穿。每节讲完概念后，都会用一个 `> 📖 对照案例` 引用块把它拉回到这个案例上，看"对这个例子而言，这一节到底在做什么"。

**案例设定**（详细逐步推演见第一部分第 8 节）：

- **输入**：`我、喜、欢、吃、苹` 共 5 个 Token（字）
- **要预测**：下一个字 `果`
- **模型**：2 层 Transformer block 的自回归模型（GPT/LLaMA 类），隐藏维度 H=4096
- **一句话**：模型把"我喜欢吃苹"这 5 个字各自表示成一条 4096 维向量，经过 2 层 block 加工，让最后一个字「苹」的向量足以在词表里给「果」打最高分。

- **残差流** "我喜欢吃苹"形成 [B,S,H]（[batch, seq, H]），形成一个不断更新的残差流，贯穿模型的每一层

> 💡 **训练机制速答**（初学最容易先问的两件事）：
> - **Q1: 训练时每个字都预测，不仅是最后一个「果」？** 对。因果 LM 用移位自监督——输入 `[BOS]我喜欢吃苹`，标签 `我喜欢吃苹果`，6 个位置各算一次交叉熵 loss 后累加，监督密度远高于只预测末字。
> - **Q2: 6 次预测是 1 个 step 还是 6 个 step？每层重复算吗？** **1 个 step，每层只算一次。** 一次前向中 6 个 token 的 `[6, H]` 张量同时流经所有层，head 一次性产出全部 6 个位置的 logits，一次反向更新全部权重。这跟推理不同——推理是逐字串行生成，每层会被反复调用多次。

# 第一部分 · Transformer 基础概念（先建直觉）

## 1. 术语：层、block、子层、算子是什么关系

"层"这个字有歧义，**取决于看的尺度**。先把尺度分清，后面一切才不乱：

> 📖 **对照案例**：处理「我喜欢吃苹」时，我们说这个模型有"2 **层**"，数的就是 2 个 **block**（每个 = Attention+MLP）。「苹」这个字的向量在每个 block 里会流经 Linear(Q/K/V/O)、Softmax 这些**算子**；而 block 内部的 Attention、MLP 各算一个**子层**。本节先把"2 层"里的"层"到底指什么说清楚。

| 说法 | 指什么 | 在 demo 里 |
|------|--------|-----------|
| **Transformer 层 / block** | 一整个 `Attention + MLP`（含 Norm、残差）的组合单元 | 滑块"Transformer 层数"数的就是它 |
| **子层 (sub-layer)** | block 内部的 Attention 子层、MLP 子层 | 原论文管 Attn 和 FFN 各叫一个 "sub-layer" |
| **MoE（专家混合）/ 专家 (expert)** | **MLP 子层的一种变体**：把"单个 MLP"换成"一个路由器 + 多个并列的专家 MLP"，每个 token 只激活其中 top-k 个专家。它**替换的是 MLP 子层这一格，不新增层** | 选 MoE 时，MLP 子层展开成 router + E 个专家 MLP |
| **路由器 (router / gate)** | MoE 里的小网络（一个 Linear + Softmax），给每个 token 打分、决定送进哪几个专家 | MoE 子层入口那个分流算子 |
| **算子 (operator)** | 最细的 `Linear`、`LayerNorm`、`Softmax` 等运算 | 右侧"整网算子图"展开的节点 |

嵌套关系（第二个子层有 **Dense / MoE 两种形态**，二选一，都仍只算"一层"）：

```
Transformer Block（= 1 层，可堆叠的最小重复单元）
├── 子层1: LayerNorm + Attention + 残差
│            └── 算子: Linear(Q) Linear(K) Linear(V) Softmax Linear(O)
└── 子层2: LayerNorm + MLP/MoE + 残差          ← 同一个"位置"，两种实现二选一
     ├─（A·Dense MLP，标准）算子: Linear  激活函数  Linear
     └─（B·MoE，稀疏变体）
            ├── 路由器 Router: Linear → Softmax → 选 top-k 个专家
            └── 专家 Expert ×E（每个结构 = 一个 MLP）
                          └── 算子: Linear  激活函数  Linear
```

**关键结论：**
- **"Transformer 层" 和 "block" 是同义词**，指 `Attn+MLP` 这个可堆叠单元，也是数"层数"的单位。网络主干就是 `for i in range(N): x = block[i](x)`。
- 说 "LLaMA-7B 32 层"、"BERT-base 12 层"，数的单位都是 **block**，不是内部算子。
- **MoE 不改变"层数"的数法。** MoE 只是把第二个子层（MLP）的内部实现从"一个 MLP"换成"路由器 + 多个专家 MLP"；block 还是那一个 block，专家是子层**内部的并列分支**，不是堆叠出来的新层。所以 "Mixtral 8×7B 32 层" 里的 32 数的仍是 block，不是专家个数。
- **专家个数（E）和层数（N）是两个正交维度**：N 是流水线多深（堆几个 block），E 是每个 MoE 子层里横向铺了几个并列专家。增大 E 是"加宽容量"而非"加深层数"。
- 模型规模、显存、并行切分（尤其 PP 流水线并行按层切）都按 block 这个整数单位算；MoE 另需"专家并行 EP"把 E 个专家分到多卡（详见后文并行部分）。

> ⚠️ **误区**：以为"层"只有一个意思。其实"多少层"说的是 block；block 内部还能再分子层、算子，那是更细的尺度，平时不计入层数。
>
> ⚠️ **MoE 专属误区**：把"8 个专家"当成"8 层"，或以为 MoE 是夹在 Attention 和 MLP 之间的"第三种子层"。其实 **MoE 就是 MLP 子层本身的替身**，站在同一个位置上；8 个专家是这一个子层内部的 8 条并列支路，每个 token 只走其中 top-k 条——不是 8 层，也不是新子层。

### MoE


1. 各卡先跑Gate（路由）计算：输入 Token 经过一个 Gate 网络，计算 Token 与各个专家的相关性得分, 给每个 token 选出 Top-K 目标专家，映射到对应设备（rank）。当然，如果TP并行，一个专家切分到2卡上，这个映射就不是映射单rank而是映射这个专家所属的完整 TP 组；

2. 本地（rank = 一张独立 GPU；一张卡内会存放多个专家）把 token按目标设备分桶打包。如果没有任何 token 要发给某张远端 rank，这个桶直接为空，通信时不会传输任何数据。

3. All-to-All Dispatch：只把「要去该设备」的 token 发过去，不发无关 token；空桶对应链路直接跳过传输，不发零向量、不填充无用占位 token，节省带宽。
4. 专家并行计算：选中的专家各自独立完成矩阵乘法运算,即各 GPU 本地执行 Expert 专家 FFN 计算。
5. All-to-All Combine：把专家结果发回 token 原始 rank
6. 将各专家的输出按 Gate 的权重进行加权求和，得到 MoE 整体输出增量\(R_{moe}\)。

> **MoE 的 ALL TO ALL**
> 第一个 All：当前 rank 需要遍历组内所有其他 rank，为每一个远端 rank 准备独立发送缓冲区（桶）。有数据就传、没数据就不传。
> 第二个 All：当前 rank 需要接收组内所有其他 rank 发来的数据片段。不管别的卡有没有发给自己，程序要预留接收通道，挨个处理每个 peer 的传入数据。

在没有 EP 的情况下，每张 GPU 都必须存储该层所有专家的完整权重。对于参数量动辄上千亿的 MoE 模型（如数百个专家），单卡显存完全无法承载。因此，EP 的核心逻辑是将 Experts 集合在第 0 维度（专家维度）进行拆分，让不同的 Rank 维护不同的专家子集。由于专家被物理隔离在不同的显卡上，原本 gate 所做的逻辑分发变成了真实跨越 GPU rank 的物理分发：

1. Dispatch (All-to-All)

在训练阶段，虽然 Transformer 是 auto regressive 的，但 Causal Mask 实现了全序列的并行 Forward。此时，每个 Rank 都同时持有大量的 Token，且每个 token 都持有不同专家的 gate 分数。于是，在每个 Rank 独立运行 Gate 算法，计算本地 Token 所需的目标专家及其所在的 Rank。由于每个 Rank 都有 Token 需要发往其他 Rank，同时也要接收来自其他所有 Rank 发来的 Token，这构成了典型的 All-to-All 通信，完成了分布式转置（Distributed Transpose）过程，将数据分布从“按序列位置对齐”重组为“按 experts 索引对齐”。

2. Expert Compute

各 Rank 并行执行本地持有的专家计算（FFN）。不同 Rank 之间的专家计算是完全独立的，无需通信同步。但是，此时的计算效率极大地取决于路由分布。如果大量 Token 涌向同一个专家（Hotspot Expert），会导致严重的负载不均衡（Load Imbalance），计算慢的 Rank 会拖累整个集群的同步速度。

3. Combine (All-to-All)

计算完成后，专家输出的局部结果（Expert Latents）需要再次通过 All-to-All 通信进行回传每个专家 Rank 将计算结果发回给该 Token 原始出发的 Rank。这确保了数据的物理分布恢复到进入 MoE 层之前的状态，以便进行后续的加权求和（Combine）、残差连接以及下一层 Attention 的并行计算。

注意到，虽然在 inference 的 Decoding 阶段，每次仅处理一个 Token，但在 Training 和 Pre-filling 阶段，高吞吐的并行计算使得 All-to-All 成为最有效、最常用的通信抽象，因此我们一般认为 EP 需要的是两次 All-to-All 通信。

### 配套结构图：整个模型从头到尾长什么样（Embed → n×Block → Norm → Head）

### 嵌套图里的两个配角：残差 + Norm

上面嵌套图里，每个子层除了主角（Attention / MLP），还各挂着两个反复出现的"配角"：**残差连接**和 **Norm（LayerNorm / RMSNorm）**。它们不混合 token、不改变语义方向，只负责"维稳"，但缺了就训不动。先在这里把它们讲透，后面（第 4 节合起来看一层的完整动作）就不再重复。

- **残差连接（那个 `+`）**：每个子层的输出不是直接覆盖原向量，而是 `新向量 = 老向量 + 增量Δ`——只往主流里"加"一笔修正，信息被**不断修订而非推倒重来**（详见第 3 节"残差流"）。它还让梯度能几乎无损地回流到浅层，是深层网络训得动的关键。

> ⚠️ **这里的"向量"到底指什么？** 就是**当前 token 在这一层此刻的隐状态向量**（长度 H，即第 7 节的 hidden size）。拿「苹」来说，就是「苹」这个字流到该子层入口时的那条 4096 维浮点数数组——它不是 embedding 刚查出来的 x₀，而是被前面各层反复改写过的"当前模样"。每个 token 各自拥有一条独立 H 维向量，残差连接对每个 token 的向量独立做 `新 = 旧 + Δ`，不同 token 互不干扰。

> 📖 **对照案例**：「苹」流过一个子层时——
>
> **什么叫"残差"？** "残差"就是子层（Attention 或 MLP）这一轮算出的**增量 Δ 本身**——一个和残差流里的向量同样长度（H=4096）的浮点数向量。"残差连接"就是把这个 Δ 逐元素加到老向量上的那个 `+` 号操作。
>
> **Δ 有正有负吗？** 当然。`新向量 = 老向量 + Δ` 里的 `+` 是**逐元素加法**：Δ 的 4096 个分量各管各的，正数分量把对应维**推高**，负数分量把对应维**压低**——不是说 4096 维齐涨齐跌，而是**各维按需升降**，合起来让整个向量朝更正确的语义方向偏移。
>
> 落到「苹」身上：Attention 认出"前面有进食动词「吃」"后，Δ 里和"水果"相关的维度被抬升（正增量），和"动词"混淆的维度被压降（负增量）。「苹」原本的字义不丢，只叠加这笔"朝能吃的水果 → 苹果方向挪一步"的修正。

#### 关键澄清：激活张量x和残差流S



我喜欢吃苹果，每一层之间更新传递的原始数据batch * 6 * 4096是主残差流 \(S\)

举例：\(S_2\)从Layer2传入Layer3

送入Layer3的Attention子层前，\(S_2\)不变，复制\(S_2\)均值方差归一化计算出激活张量：
\(x_{attn} = \text{LN}_1(S_2) \)

Attention子层完成做QKV计算，删除\(x_{attn}\)，输出残差增量：
\(R_{attn} = \text{Attention}(x_{attn}） \)

残差增量合入残差流
\(S_3 = S_2 + R_{attn} \)

送入Layer3的MoE子层前，\(S_3\)不变，复制\(S_3\)均值方差归一化计算出激活张量：
\(x_{moe} = \text{LN}_2(S_3) \)

MoE子层接收其他专家token计算结果后，按门控权重加权求和得到增量，删除\(x_{moe}\)，输出残差增量：
\(R_{moe} = \text{MoE}(x_{moe}） \)

残差增量合入残差流
\(S_3 = S_3 + R_{moe} \)

传输到下一层

**总结**：激活张量\(x\)只是为了做atten、mlp运算而均值方差归一化后的临时张量，形状也是 batch * 5 * 4096张量一样大，用完即弃，运算结果\(R\)则先后合入残差流。残差流\(S\)虽然有“流”字，但他只保留最新张量值，而不会保持每一层张量的“快照”。Attn 子层和 MoE 子层各有一套独立 LN，会被梯度更新（各自γ/β，不共享权重）。
为什么残差主干不用归一化后的数据传输？① 激活漂移/梯度爆炸；② 反向时难收敛。

#### 关键澄清：AllReduce发生在哪？

AllReduce = 多设备同维度张量求和运算 + 结果同步到全部设备。

- TP：发生在训练阶段。上一步，如果涉及TP切分，\(R_a\)+ \(R_m\)会分布在TP内不同卡上，所有卡分片 \(\tilde R\) 按通道求和，得到全局完整注意力残差 \(R_a\)

- DP：发生在更新梯度阶段。所有 DP 组本地梯度求和；平均值同步下发到每一张卡；各卡用相同梯度更新权重。


#### 关键澄清：Norm 归一化的是"数值"，不是"权重"——它不干预训练，本身就是训练的一部分

初学最容易卡在这里：**"归一化不是把参数改了吗？改参数岂不是干扰了训练？"** 这是把两样东西搞混了：

- **它归一化的对象是"激活值"，不是"权重"。** Norm 作用在**前向传播时流过网络的那个向量 x**（残差流里的中间结果），把这一串数字的尺度拉正。它**完全不碰** W_Q / W_up 这些权重矩阵——权重一个数都没动。
- **γ（逐维缩放增益）和 β（逐维平移偏置）虽然是参数**——归一化后做 `y = γ·x̂ + β`，每个分量被 γ 的对应维放大/缩小、再被 β 的对应维平移——但它们和别的权重一样是被训练"学"出来的，随反向传播一起更新，不是外部强行设定的。模型可以自己学到"这一层归一化之后要再放大还是缩小"。
- **所以非但不"干预"训练，反而是训练能进行下去的前提。** Norm 是计算图里一个普通的前向算子，梯度照常穿过它；它让数值不爆炸、让梯度稳定——是在**帮**训练收敛，不是打断它。


#### 一个具体例子：归一化把什么数变成什么数

以案例里的「苹」为例，假设残差流累加到某层时，「苹」的向量（取 4 维示意）数值已经偏大、不齐：

```
输入 x = [ 100.0, -98.0, 102.0, -101.0 ]       ← 均值≈0.75，标准差≈100（尺度很大）

第①步 归一化（减均值、除标准差）：
   x̂ = [ 0.99, -0.99, 1.01, -1.02 ]            ← 均值≈0，方差≈1（尺度被拉正）

第②步 用可学习的 γ、β 缩放平移（设 γ=[2,2,2,2]，β=[0,0,0,0]，这俩是训练学到的）：
   y = γ·x̂ + β = [ 1.98, -1.98, 2.02, -2.04 ]  ← 交给后面的 Attention/MLP
```

- **变的是数值的"尺度/分布"**：从"忽大忽小、标准差 100"变成"均值 0、方差 1 再按 γ/β 微调"。
- **没变的是"方向/相对关系"**：四个分量的正负、大小顺序基本保持——所以语义（向量指向哪）没被破坏，只是把数值调到了好训练的范围。
- γ、β 是这一层专属的两个长度 H 的向量，由训练决定"归一化之后还要不要再放大/平移一点"——这正是"参数是被学出来的，而非干预训练"的体现。

> （RMSNorm 是简化版：只除以均方根、不减均值、只有 γ 没有 β，思路相同。）

## 2. 首尾的特殊层：embedding 与 head 不计入层数

主干的 block 是同构的，但**首尾两层结构不同，且不算进"层数"**：

```
embedding（输入层，token ID → 向量）        ← 不算层数
  ├─ block 1   ┐  （内部含 Norm）
  ├─ block 2   ├─ 只有这些算 "层数"，如 LLaMA-7B = 32
  ├─ ...       │
  └─ block N   ┘
final norm（最后一层归一化）                ← 不算层数
head / lm_head（输出层，向量 → 词表 logits）  ← 不算层数
```

- demo 里用**不同颜色**区分：embedding 黄、attn 蓝、mlp 绿、norm 灰、head 紫。
- 它们结构不同于可堆叠的同构 block，所以不计入"层数"。

> 📖 **对照案例**：「我喜欢吃苹」先经 **embedding** 把 5 个字（token ID）查表成 5 条 4096 维向量——这步不算层；中间走 **2 个 block**（这才是"2 层"）；最后 **final norm + head** 把「苹」的向量投影成词表分数选出「果」——首尾这两步也都不算层数。

### embedding 层：把"token ID"翻译成"向量"

- **功能**：模型不认识文字，输入先被分词器切成一串整数 token ID（如 `"苹果"→[3621, 3637]`）。embedding 层持有一张大查找表 `W_emb [V×H]`（V=词表大小，H=隐藏维度），**用 token ID 去表里取出对应的那一行向量**——这是 lookup（gather），不是矩阵乘法。
- **产出**：`x0`，也就是第 3 节"残差流"的起点。
- **为什么不算层**：它没有 Attention/MLP 那套结构，只是一次查表；且每个 token 独立取向量，不做 token 间的交互。它是所有 block 的"输入准备"，不是可堆叠的同构计算单元。
- **部署时去哪了**：训练完成后，`W_emb` 里的向量已不再是随机数——每个字都有了语义。这份词表**本身就是模型权重的一部分**，和 `W_Q`、`W_up` 等一起保存在 checkpoint 文件（如 `pytorch_model.bin` 或 `.safetensors`）中，随模型一起部署。推理时每来一个字，就从这个已训练好的词表里取出它对应的向量。另外，很多模型（如 LLaMA）让 embedding 和 head 共享同一块权重（weight tying），也就是入口查表和出口投影用的是同一张表的两种用法——部署时它作为一份参数同时服务于输入和输出。
- **为什么新模型训练又要从随机开始**：既然词表这么有用，能不能直接复用旧模型的词表，只重新训练 block？答案是**通常不行**——词表和 block 的权重是"共生"的：训练时它们一起被梯度打磨，block 学会了"在这个词表的向量几何下去做 Attention/MLP"。如果把 A 模型的词表塞进 B 模型（block 权重是另一套随机初始化的），block 根本读不懂这些向量的"方言"，反而训得更慢甚至不收敛。所以新模型从零训练时，所有参数（包括词表）一起随机初始化、一起成长，才能保证它们是"同一套语言体系"。当然，**微调（fine-tuning）**是另一回事——那是从旧模型的**全套权重**（词表 + block 一起）出发，继续在新数据上训，此时词表当然保留，不需要从随机开始。

> 📋 **全模型权重速查表**（以下所有矩阵在**每个训练 step** 都会经历：前向被读取 → 反向被算梯度 → 优化器更新数值；各层同名矩阵独立不共享，只有 W_emb 和 W_head 是全局各一份且常互享）
>
> | 权重 | 形状 | 所属层 | 作用 | 每层独立？ |
> |------|------|--------|------|-----------|
> | **W_emb** | `[V×H]` | Embedding | token ID → 向量：用 ID 查表取出对应行的 H 维语义向量（gather，非 MatMul） | ❌ 全局唯一 |
> | **W_Q** | `[H×H]` | Attention | 生成 Query："我（当前 token）想找什么信息"的提问向量 | ✅ 每层独立 |
> | **W_K** | `[H×H]` | Attention | 生成 Key："我身上有什么标签、可被别的 token 检索到" | ✅ 每层独立 |
> | **W_V** | `[H×H]` | Attention | 生成 Value："我能给关注我的 token 提供什么实际内容" | ✅ 每层独立 |
> | **W_O** | `[H×H]` | Attention | 多头输出投影：把各头拼接的加权结果融合，压回 H 维 | ✅ 每层独立 |
> | **W_up** | `[H×4H]` | MLP | 升维投影：把 Attention 富化后的 H 维向量展开到 4H 高维空间 | ✅ 每层独立 |
> | **W_gate** | `[H×4H]` | MLP (SwiGLU) | 门控投影：决定升维后哪些维度被 SiLU 激活/抑制，实现选择性信息过滤 | ✅ 每层独立 |
> | **W_dn** | `[4H×H]` | MLP | 降维投影：把 4H 维非线性变换结果压回 H 维，产出本层 MLP 的增量 Δ | ✅ 每层独立 |
> | **W_router** | `[H×E]` | MoE | 路由打分：给每个 token 对 E 个专家打分，选出 top-k 个专家处理该 token | ✅ 每层独立 |
> | **W_exp_i** | `[H×4H]` | MoE 专家 (×E个) | 专家 FFN：第 i 个专家的权重（含 up+gate+down），仅被路由选中的 token 流经 | ✅ 每层独立 |
> | **W_head** | `[H×V]` | Head | 向量 → logits：把最终隐状态乘 `[H×V]` 投影到词表维度，每个词得一个分数 | ❌ 全局唯一 |
> | **γ, β** | `[H]` (向量) | Norm | 归一化后逐维缩放(γ)和平移(β)，让模型自己学"归一化后要再放大/缩小多少" | ✅ 每层独立 |
>
> > 💡 **"每 step 更新"含义**：每个训练 step，loss 对所有权重矩阵求梯度，优化器把每个矩阵的数值微调一次。W_emb 和 W_head 虽然全局只有一份，但它们的梯度来自所有层的累积贡献——比如 W_emb 的梯度既来自第一层 Attention 传回的信号，也来自深层 MLP 传回的信号，最终合在一起更新一次。详见第二部分和第三部分第 1 节。
>
> ⚠️ **W_emb 是查表，为什么也算"权重"？** "查表"只是操作方式（取出某一行），但那张表本身——`W_emb [V×H]`——是 15 万个 × 4096 个**可学习的浮点数参数**，和 `W_Q`、`W_up` 一样初始随机、一样被梯度更新。正因为它每个 step 都在变，同一个字「苹」在不同训练阶段的查表结果才不一样——从随机噪声逐渐变成有语义的向量（见第 8.1 节"阶段 1 → 阶段 2"）。它和 `W_Q` 的唯一区别是使用方式：`W_Q` 被矩阵乘，`W_emb` 被按行取——但都是可训练参数，都保存在 checkpoint 里。


### 多模态输入：图片 / 视频 / 音频怎么变成向量序列

上面只讲了文字。但 Transformer 主干（block）**根本不关心模态**——它只吃一个形状为 `[seq, H]` 的向量序列（seq 个 token，每个 H 维）。**所有模态差异都被关在"输入前端"里：怎么把原始信号变成这串 H 维向量。** 进了 block 之后，文本、图像、视频、音频对 Transformer 来说都只是"一串 H 维向量"，处理逻辑完全一样。

把各种前端归成两大类：

| 类型 | 思路 | 有没有"查表" | 典型模态 |
|---|---|---|---|
| **A. 离散化 + 查表** | 先把信号变成一串**整数 ID**，再用 `W_emb [V×H]` 查表取向量（和文字完全一样） | 有 | 文本；用编解码器量化后的音频 |
| **B. 切块 + 线性投影** | 把信号切成小块，每块拉平成一个长向量，过一个 `Linear` **直接投影**到 H 维 | 无（连续投影，无离散 ID） | 图像、视频、原始音频频谱 |

各模态具体怎么做：

- **文字**：分词器 → token ID → `W_emb` 查表（A 类，前面已讲）。
- **图片（ViT 范式，B 类）**：把图切成固定大小的 **patch**（如 224×224 图切成 16×16 的小块 → 共 196 块）；每个 patch 拉平（16×16×3=768 个像素值）过一个 `Linear` 投到 H 维 → 得到 196 个"图像 token"。**没有词表、没有查表**，是连续投影。再加 **2D 位置编码**（告诉模型每块在图里的行列位置）。
- **视频（B 类，多一个时间维）**：在图片基础上加时间。要么**逐帧**切 patch 再附加"第几帧"的时序位置编码；要么用 **3D patch / tubelet**（把"几帧 × 一小块空间"当成一个时空立方体）一起切。代价是序列长度暴涨（帧数 × 每帧 patch 数），所以视频模型常配合下采样、抽帧或上下文并行 CP（见第三部分）。
- **音频**：两条路都有。① 波形 → **梅尔频谱图**，当成"图像"切 patch 投影（Whisper 用一段卷积前端做这步，B 类）；② 用神经音频编解码器（EnCodec / SoundStream）把音频**量化成离散音频 token** → 查表（走 A 类，和文字同构）。

**多模态大模型怎么把它们拼到一起**（如 LLaVA、Qwen-VL）：

```
图像 → 视觉 encoder(ViT) → 一串图像向量 ┐
                                        ├─ projector/adapter 对齐到同一个 H 空间
文本 → 分词 → 查表 → 一串文本向量      ┘
                ↓
   拼成一条序列：[文本…  <image> 图像向量…  文本…]
                ↓
        喂给同一个 Transformer 主干（block 不变）
```

- 每个模态先用各自的 encoder 把信号变成向量，再用一个小的 **projector**（通常就是一两层 Linear/MLP）把维度和语义空间对齐到 LLM 的 H；
- 然后和文本 token 向量**拼成一条序列**，用特殊分隔 token（如 `<image>`）标界，之后交给同一个主干统一处理。

> **一句话归纳**：模态适配只发生在"原始信号 → `[seq, H]` 向量序列"这一步，本质是把文字的"分词 + embedding"换成对应模态的"切块/量化 + 投影"。这也正说明第 2 节为什么把 embedding 称为"输入准备、不算层数"——它是**可按模态替换的前端**，而 block 主干始终不变。

> 💡 **多模态把张量撑多大？维度数不变，只撑大 `seq`**：流经各层的激活 `x` 永远是三维 `[batch, seq, H]`（前端已把图/视频/音的空间时间结构全压进 `seq`，H 由模型定、不随模态变）。变的只是 `seq` 的**数值**：文本「我喜欢吃苹」seq=5、图片(ViT)≈196、高清图/视频 seq 可达几千~几万。x 的显存随 seq **线性**涨；更狠的是 Attention 内部 `[batch,heads,seq,seq]` 分数矩阵按 **seq²** 涨（seq=4096 时一层即 ~1GB）——这正是长序列(尤其视频)必须上 **CP 上下文并行**的原因（第三部分第 3 节）。

### norm 层（LayerNorm / RMSNorm）：稳定数值，让训练能收敛

- **功能与原理**：已统一讲在第 1 节"嵌套图里的两个配角：残差 + Norm"。简言之，它归一化的是**流过的激活值（不是权重）**，把残差流累加后失控的数值尺度拉回稳定分布，是残差流上的"稳压器"。本节只看它和"层数"的关系。
- **它在哪**：norm 主要**藏在每个 block 内部**（Attention 前一个、MLP 前一个，即 Pre-Norm），是 block 的一部分 → 这部分 norm **随 block 一起被计入层数**。此外主干**最后还有一个独立的 final norm**（送进 head 之前）→ 这个**不计入层数**。
- **为什么单独的 norm 不算"一层"**：它没有权重矩阵（只有 γ、β 两个长度 H 的**向量**），是逐元素乘加而非 MatMul，参数量极小，算力可忽略。它是辅助性的"调节器"，不承担"理解/精炼"职责，所以从不单独算作一层。
- **LayerNorm vs RMSNorm**：LayerNorm 减均值 + 除标准差，有 γ 和 β 两个参数；RMSNorm 只除均方根、不减均值、只有 γ 无 β——计算更省，效果接近，LLaMA/DeepSeek 等主流模型已普遍采用 RMSNorm。


### LM Head 层：把向量映射，或翻译成词表分数

- **功能**：前面 N 层逐层精炼后，最后一层吐出一个向量（如 4096 个数），里面**隐含**了答案，但还是一串浮点数字。head 的任务是把这个向量乘一个大矩阵 `W_head [H×V]`，投影到**整个词表**，每个词（词表的词，不是我喜欢吃苹这里每个词）得一个分数(logits，比如果:9.2，梨:3.1，饭:-2.5，山:-5.8…)， → softmax 成概率（比如果≈0.9978，梨≈0.0022，其余趋近 0） → 取最高。几乎是机械翻译，不"思考"。

注意，Softmax 不属于模型网络里任何层，它是独立的后处理算子，在 LM Head 之后、损失 / 采样之前执行。

**尾部几个步骤的衔接** 残差流 → LM Head（矩阵乘，输出分片 logits）→ 到Head外了 → 独立 All-Gather 通信（Logits All-Gather：把所有 GPU 的分片 logits 沿词表维度拼接，得到完整 [B,S,V]） → 完整全局 logits → softmax / 损失。

- **为什么Head不算transformer层**：它没有 Attention/MLP 那套结构，只是一次 MatMul + Softmax；不参与"理解/精炼"，只是把最终的隐状态"翻译"成人类可读的词。
- **直观比喻**：前面 N 层像**侦探破案**（整合线索、得出结论），head 只是**把结论念出来**（翻译成具体的词）。预测准不准，全看前面 N 层把表示雕得好不好，head 不背锅。

> 📖 **对照案例**：自回归模型预测下一个字时，head **只看最后一个 token 的最终向量**（即「苹」走完所有 block 后的向量），不看「我」「喜」「欢」「吃」。但这些字的贡献并没有丢——在每层 block 的 Attention 里，「苹」逐一看了前面的「我」「喜」「欢」「吃」，把它们的语义（尤其是「吃」是进食动词）搬到了自己身上。所以 head 拿到的虽然是「苹」这一个向量，它里面其实浓缩了前面 4 个字的全部上下文。
>
> 训练初期，前面 N 层还没训好，Attention 搬过来的信息不对、「苹」的向量指向不准——head 乘 `W_head` 后「果」才 0.03、「桃」0.15、「蕉」0.12。Loss 巨大的信号顺着「苹」这条线往前回溯：梯度不仅修 head 和 MLP，还会穿过 Attention 的连接，去修「吃」→「苹」的关联、甚至「喜」→「欢」的搭配——前面每个字都间接为预测「果」出了力，错了也一起挨罚。几十万轮后收敛，Attention 练就了精准搬运上下文的能力，「苹」的向量终于指向正确方向→「果」飙到 0.85。head 做的事始终没变——乘矩阵、softmax、取最高分，它只是把前面 N 层"协作雕出来的答案"忠实地报出来。
>
> ⚠️ **那「苹」在下一条语料里会带着"水果味"吗？下一个 step 的向量又从随机开始？** 不会从随机开始——被丢弃的不是 embedding 权重，而是**前向传播时流经各层的临时激活值**。打个比方：
>
> - **`W_emb` 是一本"字典"**（模型参数），训练中不断被修改。step1 训完「我喜欢吃苹→果」后，字典里「苹」这一页已经被梯度微调过，带上了一点"水果/苹果"方向的偏移。这本字典**跨 step 持久保存**。
> - **每次前向 = 翻开字典查字，得到初始向量，然后流经各层加工。** 这条加工流水线上产出的中间结果（每个 block 出口的 `5×4096` 张量）就是"临时激活值"——反向传播用完后就从显存里释放，**不跨 step 保留**。
> - **下一个 step 来的是新语料「苹字很难写」**：重新翻开同一本字典（`W_emb` 已经带着 step1 的更新），查出「苹」的向量依然有"水果味"痕迹。但 Attention 看到的是「字/很难写」这个新上下文，会在残差流里把临时向量朝"汉字话题"方向改写；step2 的 Loss 反向同样会把字典里「苹」这一页往新方向微调。
>
> 所以"丢弃"的是流水线上的中间结果，不是字典本身。千万轮下来，字典里每个字的向量被无数次微调，收敛到在各种上下文里都"好用"的状态——最终朝哪偏，由每轮的 Attention+MLP 根据实时上下文决定。

> **一句话归纳**：
> - **embedding**：入口查表，token ID → 向量，**不算层数**。
> - **block 内部的 norm**：稳压器，是 block 的一部分，**随 block 计入层数**。
> - **final norm**：送 head 前的最后一次稳压，**不算层数**。
> - **head**：出口投影，向量 → 词表分数，**不算层数**。
> - 数"多少层"时，只数中间那些 `Attn+MLP` 的 block。

## 3. Transformer 层到底在做什么（从一个例子入手）

> 这是理解 Transformer 最关键的一节。前两节讲了"层是什么、怎么数"；这一节讲**层真正干活时在做什么**。不从公式入手，先看一个例子，再拆开背后的三个问题。

> 📖 **对照案例**：下面这条流水线讲的就是本案例——浅层 block 认出这 5 个字、识别「吃」是动词；深层 block 把「吃 ↔ 苹」关联起来、收敛到"该补一个能吃的水果后缀"；最后 head 在词表里给「果」打最高分。没有哪一层单独答出「果」，是 2 层各推进一小步、叠起来的。

### 先看例子：输入"我喜欢吃苹"，模型怎么吐出"果"

把 N 层想象成一条流水线，每层在前一层的基础上，把这句话的理解**多精炼一小步**：

```
输入："我喜欢吃苹"
  浅层：识别这是中文、"吃"是个进食动词、"苹"是个字
    ↓（理解被一层层往下传）
  中层：建立 "吃 ←→ 苹" 的动宾关系，把这两个字关联起来
    ↓
  深层：收敛成"该补一个能吃的水果后缀，凑成『苹果』"
    ↓
  出口 head：在所有字里给"果"打最高分 → 输出"果"
```

没有哪一层独自答出"果"。是 N 层每层各推进一小步、**叠起来**才把"一串字"雕成"足以预测下一个字"的状态。head 层（见第 2 节）只是把最终结果翻译成词表分数，不参与"想"的过程。下面拆这条流水线的两个关键问题。

### 问题一：层与层怎么接力？——残差流（Residual Stream 残差流里的中间结果）一条被不断增补的信息流）

别把多层想成"每层把活重做一遍"。正确的图景是**一条信息流被逐层增补**：

```
x0 (词向量，来自 embedding)
  → 层1: x1 = x0 + Δ1     ← 层1 只往流里"加"它算出的修正量 Δ1
  → 层2: x2 = x1 + Δ2     ← 层2 在 x1 基础上再加 Δ2
  → ...
  → 层N: xN = x(N-1) + ΔN
  → head: 用 xN 做预测
```

注意那个 `+`（残差连接）。每层**读当前的流 → 算一个增量 Δ → 加回流里**，信息是**累积式**前进的：

- **前一层**：把它理解到的东西写进流里，作为后一层的输入。
- **后一层**：站在前一层的肩膀上，读取需要的信息，再补充新理解。

所以"上一层 vs 下一层"不是流水线上各干各的工序，而是**共享同一条流、后层基于前层继续精炼**的接力。（这个 `+` 还有第二个作用：让梯度几乎无损地传回浅层，是深层网络训得动的关键，见问题二。）
残差只是残差流里往里加的增加，残差流说那么大，最终交付也只是最后一层的结果（就是“我喜欢吃苹”5个字最后的浮点值），最终送入 LM Head 生成 logits。

> 💡 每个 Δ 具体怎么算出来（Attention + MLP），是更细的机制，紧接的第 4 节展开；这里先记住一句"层 = 往残差流里加一个增量"。

### 问题二：每层长得一模一样，凭什么分工不同、又是谁安排的？

观察 demo：每个 block 画出来都是 `Attention + MLP`，结构完全相同。但**"结构相同" ≠ "功能相同"**，分两层看：

- **(a) 结构（算子种类 + 连接方式）= 人设计的，每层一样、写死在代码里。** 正因如此才能用一个 `for` 循环堆 N 个相同 block，demo 也才能用一个滑块控制层数。
- **(b) 内容（权重数值）= 训练学出来的，每层各不相同。** 每层都用同样的 `Linear`，但**权重矩阵的数值各自独立、不共享**。于是涌现出"由浅入深"的分工，正好对应上面例子里的浅/中/深层：

| 位置 | 倾向学到的东西 | 在"我喜欢吃苹"例子里 |
|------|------|------|
| 浅层（近输入） | 词形、邻近词、局部语法 | 认字、断词，识别"吃"是动词 |
| 中层 | 句法结构、指代、实体关系 | 建立"吃 ↔ 苹"动宾关系 |
| 深层（近输出） | 抽象、长程推理、为预测整合 | 收敛到"该补水果后缀『果』" |

**这种分工不是设计者规定的，是训练逼出来的。** 关键在于：**所有层不是各自单独训练，而是被同一个损失、一次反向传播联合更新**——

```
① 前向    x0 → 层1 → ... → 层N → head → 预测分布
② 算损失  预测 vs 真实下一个词 → loss（错了多少）
③ 反向    把 loss 的"责任"从 head 一路倒推、分摊到每层每个权重（梯度）
④ 更新    每层权重 = 旧权重 − 学习率 × 梯度
⑤ 换下一批数据，重复几十万~几百万次
```

- **责任连带**：若深层因缺某信息而预测错，反向传播会发现"是浅层没把它写进流里"，于是梯度去调浅层。**浅层被"逼着"提供深层需要的东西**——分工是梯度协商出来的，不是设计的。
- **残差让梯度好传**：问题二那个 `+` 不光累积信息，也让梯度几乎无损倒流回浅层，不因层多而梯度消失。这是深层 Transformer 训得动的工程关键。
- **海量数据**：每批数据把每层往"整体预测更准"推一点点，loss 平均意义上单调下降 = "收敛精度"。

> ⚠️ **误区**：以为"结构同构"就是"每层完全一样、可随便换顺序"。其实交换层序会让性能明显变差——各层确实承担了有顺序依赖的不同角色。（"每层完全一样"只在 ALBERT 等层间共享权重的特例里出现；GPT/LLaMA 等主流模型每层权重独立。）

> **核心一句**：没有谁规定第 3 层管句法、第 20 层管推理。是几十万次"预测错 → 反向传播 → 微调每层"，自发地把不同职责压到了不同层上。这和 CNN 同理——浅层学边缘、深层学物体，**相同的积木 + 不同的权重 → 涌现不同角色**。

## 4. 层内部细节：Attention 与 MLP 两个子层（Δ 到底怎么算）

第 3 节把每层抽象成"读残差流 → 算一个增量 Δ → 加回去"，但 **Δ 具体怎么算**？打开一个 block，它做两件大事：**Attention（横向混合信息）** 和 **MLP（各自纵向加工）**。向量进、向量出，长度始终是 H（hidden size，第 7 节详解），内容被改写。

> 📖 **对照案例**：「苹」的向量增量 Δ 怎么来？**Attention**（横向）让「苹」去看前面的「吃」，把"前面有个进食动词"搬到自己身上；**MLP**（纵向）让「苹」自己把这条信息提炼成"我大概率要接『果』"。两步算出的 Δ 加回「苹」的残差流，就是它在这一层的更新。

### Attention —— 让每个 token "去看别的 token"

进来时每个 token 的向量只装着"它自己"。Attention 让它**按需把其他 token 的信息加权搬一部分到自己身上**。在一个 token 位置上：

```
我的向量 → 拆成三个角色（各乘一个权重矩阵）：
  Q (Query 我想找什么)        = 向量 × W_Q
  K (Key   我能被找到的标签)   = 向量 × W_K
  V (Value 我能提供的内容)     = 向量 × W_V
```

然后：

1. 用我的 **Q** 和句中每个 token 的 **K** 做点积 → "我该多关注谁"的分数
2. 分数过 **Softmax** → 一组加权比例（和为 1）
3. 按比例把大家的 **V** 加权求和 → 我"看完全场"的新信息
4. 再乘输出矩阵 **W_O**，加回我原来的向量（残差）

> 直觉："苹果 很 好吃"里，"苹果"的 Q 和"好吃"的 K 高度匹配 → 把"好吃"的 V 搬过来 → "苹果"的向量被写进"我是个能吃、好吃的东西"。**这一步是 token 之间横向的信息流动。** 对应几何：把"苹果"这个点朝"食物"方向挪了一下。

### MLP —— 让每个 token "自己消化一下"

Attention 是横向混合，MLP 则是**每个 token 各自独立**深加工（不看别人）：

```
向量(H) --up_proj--> 放大到(4H) --非线性(SiLU/GELU)--> --down_proj--> 缩回(H)
```

先升维到 4 倍，在更高维空间用非线性做一次复杂变换，再压回原维度。直觉上是对刚收集到的信息做**提炼、组合、抽象**。

> ⚠️ **两个容易踩的坑：**
>
> **坑一：除了「苹」，别的字也会被 MLP 处理吗？** 会。MLP 对**每一个 token 都做**，无一例外。「我」「喜」「欢」「吃」「苹」——5 个字各自独立经过同一个 MLP。比如「吃」的向量进了 MLP 后，会被提炼成"这是个进食动词、后面大概率接食物宾语"；「喜」的向量被提炼成"这是个正向心理词、常与『欢』搭配"。
>
> **坑二：既然「苹」过 MLP 就能趋近『果』，那还要 Attention 干什么？** 关键在这里： 如果从 embedding 出来就让「苹」直接进 MLP，「苹」还是孤零零的一个字向量，MLP 并不知道前面出现过「吃」——它只能根据「苹」本身的信息做变换，永远炼不出"前面有个进食动词"这条线索。是 **Attention 先帮「苹」把「吃」的信息搬过来**（「苹」的向量被写进了"前面有人在吃东西"的上下文），然后 MLP 再在这个**已经被 Attention 富化过的向量**上深加工，才能把"进食动词 + 苹 → 苹果 → 下一个字是果"这条逻辑链打通。
>
> 同理，「喜」也需要 Attention 先帮它看到后面的「欢」（"喜"的 Q 和"欢"的 K 高分匹配），"喜"的向量被写进"我后面紧跟着一个高频搭配字『欢』"的上下文；然后 MLP 在这个富化后的向量上深加工，把"喜"提炼成一个**更精准的"信息提供者"**——当下一层的 Attention 中别的 token（比如某个更远的字）看向「喜」时，能从它身上读到更丰富的语义，而不仅仅是"这是个正面的字"。两个子层缺一不可：**Attention 负责"收集上下文"，MLP 负责"基于收集到的信息做推理"**。所以，MLP 的顺序在 Attention 之后。
>
> **坑三：升到 4H 维凭什么就"知道"该预测「果」了？** 并不是升维这个动作本身有什么魔力——**魔力全在权重里，是训练"塞"进去的。** 但要先澄清一个关键时序：上面描述的"进食动词+苹→果"推理规则，是模型**训练收敛之后**的状态，不是刚初始化就有的。
>
> **训练刚开始时**，`up_proj` 和 `down_proj` 里的数值是随机初始化的（和 embedding 查表刚诞生时一样，见第 8.1 节"阶段 1"）。此时 MLP 产出的 Δ 基本是随机噪声，「苹」过完 MLP 后向量乱飘，预测大概率是错的。
>
> **是反向传播在一点点"雕刻"它们。** 每次前向预测错 → loss 高 → 梯度从 head 一路回传到 MLP 的 `up_proj` 和 `down_proj` → 优化器把这两个矩阵的数值朝"下次少犯错"的方向微调一点点。千万轮下来，两块权重矩阵被反复打磨，最终形成了一套**分布式"if-then"记忆**：当输入向量里同时出现"进食动词"和"苹类字符"这两组特征时，up_proj 会把这两组特征放大、投射到 4H 空间的某些特定神经元上；非线性（SiLU）像一个门控开关，把这些神经元激活；然后 down_proj 把这些被激活的神经元映射回 H 维，产出的增量 Δ 恰好朝"果"的方向推了一步。整个过程等价于 `Δ = down_proj @ SiLU(up_proj @ x)`——每一步都是纯数学运算，但三块组件（up_proj 的权重、down_proj 的权重、非线性）合在一起，就构成了一个被训练刻进权重里的"推理规则"。

> 💡 **FFN 总结**：上面讲的 MLP，学名就叫 **FFN（Feed Forward Network，前馈网络）**——Transformer 每层里不带时序、不做注意力、只单独加工单个 token 向量的全连接网络。MoE 的共享专家和路由专家，底层单元全是 FFN，本质都是同一套 `升维 → 激活 → 降维` 结构。FFN 输入一个 token 的 H 维向量，输出**同样 H 维的新向量**——维度不变，往里面写入词汇含义、语法逻辑、事实知识等。对比：**Attention** 让 token 互相看（横向混合上下文），**FFN** 单独深化每个 token 自身的语义知识。

### 两个配角：残差 + Norm —— 见第 1 节

每个子层除了主角（Attention / MLP），还各挂着 **残差连接**（`新向量 = 老向量 + 增量Δ`，不断修订而非推倒重来）和 **Norm（稳压器，归一化的是激活值而非权重）**。这两个配角的完整原理、"会不会干预训练"的澄清、以及一个具体数值例子，已统一讲在**第 1 节"嵌套图里的两个配角：残差 + Norm"**，此处不再重复，只在下面的完整动作里看它们的位置。

### 合起来：一层的完整动作

```
x → Norm → Attention → +x（残差）→ Norm → MLP → +x（残差）→ 输出
          ↑横向：token 互相看            ↑纵向：各自深加工
```

堆 N 层 = 把"通气—琢磨"重复 N 轮，每轮把每个词的点朝更符合上下文的位置挪一挪，理解越来越深。

### 算子级视角：把两个子层拆成最小算子

把上面两步再拆细，就是 demo 右侧"整网算子图"里看到的**前向算子序列**（这里是不含并行切分的纯前向；每个 `Linear` 背后对应一个权重矩阵，完整清单与显存见第二部分，多卡切分见第二、三部分）：

```
Attention 子层（前向）
x [B, seq, H]
  → QKV Linear（W_Q/W_K/W_V，各 [H×H]）→ Q/K/V [B, seq, H]
  → Scaled Dot-Product Attention（无权重：QKᵀ → Softmax → 加权·V）
  → Out Proj（W_O [H×H]）
  → x + 残差 [B, seq, H]
```

```
MLP 子层（前向，SwiGLU）
x [B, seq, H]
  → up_proj（W_up [H×4H]）→ [B, seq, 4H]
  → SiLU（无权重）
  → down_proj（W_dn [4H×H]）
  → x + 残差 [B, seq, H]
```

- 末尾的 `x + 残差` 就是第 3 节"残差流"的落地：每个子层只产出一个增量、加回主流。
- 带权重的算子（QKV/Out/up/down 这些 `Linear`）才占显存；无权重的纯计算（Softmax、SiLU、残差加）不持有参数——第二部分正是用这条线索解释"为什么只有带权重的层才画成格子、才占显存"。
- 权重矩阵尺寸：`W_Q/W_K/W_V/W_O` 都约 `H×H`，`W_up/W_dn` 是 `H×4H`/`4H×H`。**这就是 hidden size 调大、参数按约平方增长的根源**（也对应 demo 里 hidden size 表单的 `Attention QKV Linear：H×H`、`MLP up_proj：H×4H`）。

> **一句话归纳**：一层 = 先让每个词互相通气（Attention 横向混合），再让每个词自己琢磨（MLP 纵向提炼），结果以增量叠回残差流。第 3 节的 Δ 就是这套算出来的；第二部分接着把这些权重矩阵的数量、显存占用、多卡切分系统摊开。

## 5. 理解了 Transformer 层，再看它和 CNN 的差异（最常见的误区）

> 现在你已经知道 Transformer 层在做什么（Attention 让 token 互相看、逐层精炼），正好可以澄清一个高频误区：卷积、池化是不是 Transformer 里的层？

> ⚠️ **核心误区**：以为卷积、池化是 Transformer 里的某种层。
> **真相：卷积/池化属于 CNN（卷积网络）；Attention/MLP 属于 Transformer。它们是两个并列的网络"家族"，不互相嵌套。Transformer 层里没有卷积核，也没有池化层。**

> 📖 **对照案例**：处理「我喜欢吃苹」用的是 **Attention**（让「苹」看「吃」）——这是 Transformer 家族的零件。若换成 CNN，就会用**卷积核**在字序列上滑窗看局部，不会有"两两 token 全局关联"。本案例自始至终没有卷积、没有池化。

把它们当作两个家族里**互相替代的对应物**来看最清楚：

| | CNN（卷积网络） | Transformer |
|---|---|---|
| 核心运算 | **卷积**（卷积核滑窗） | **自注意力 Attention** |
| 基本重复单元 | 卷积块（Conv+BN+激活+池化） | Transformer block（Attn+MLP） |
| 怎么"看"数据 | 只看**局部**邻域（核覆盖的窗口） | 看**全局**所有 token 两两关联 |
| 标志性零件 | 卷积核 | Attention |
| 典型场景 | 图像（ResNet/VGG） | 序列/文本（GPT/LLaMA），也做图像(ViT) |

- **卷积核 ↔ Attention**：是各自家族里"那个标志性的运算"，平行替代关系，不是上下级。
- **卷积层、池化层只出现在 CNN 里**；标准 Transformer 层里一个都没有。
- 同一个任务通常**要么用 CNN，要么用 Transformer**，是两套并列方案。

> 比喻：卷积/池化是"柴油发动机的零件"，Attention/MLP 是"电动机的零件"。你不会在电动机里找到喷油嘴——两种引擎，各装各的零件。
>
> （例外，了解即可：Conformer、ConvNeXt 等**混合架构**会特意混搭卷积与 Attention，但那不是标准 Transformer 的默认结构。）

> 💡 第 3 节"由浅入深的分工"那段也提到 CNN 是一个道理：浅层学边缘、深层学物体——两个家族在"逐层抽象"这件事上相通，区别只在那个标志性零件（卷积 vs Attention）。

## 6. 层数怎么定 + 设多/设少的影响

> 📖 **对照案例**：「我喜欢吃苹」→「果」是**一步事实/搭配补全**，2 层就绰绰有余。层数太少（如 1 层）可能来不及把「吃 ↔ 苹」关联起来、预测出别的字；堆到 100 层则纯属浪费——答案还是「果」，只是更慢更费显存。层数要匹配任务的"推理步数复杂度"。

### 业务上怎么确定层数

层数**不是从零推导的，是"选档 + 受约束 + 实验调"出来的**。先看标准档位：

| 量级 | 层数(典型) | 代表 |
|------|------|------|
| 小 | 6–12 | BERT-base(12)、GPT-2 small(12) |
| 中 | 24 | BERT-large、GPT-2 medium |
| 大 | 32–80 | LLaMA-7B(32)、13B(40)、70B(80) |
| 超大 | 96+ | GPT-3 175B(96) |

真正卡住层数的约束：
- **算力/显存/延迟（最硬）**：层数 × 每层参数 ≈ 显存与延迟。线上常"延迟反推层数"：定 P99 延迟上限 → 反推能跑多深。
- **数据量**：深模型要更多数据喂饱；数据少时堆层易过拟合（参考 Chinchilla：给定算力，模型大小与数据量要平衡）。
- **任务复杂度**：分类/检索/NER 等 6–12 层常够（甚至蒸馏到 2–6 层）；多步推理/长文本生成需要更深。
- **深 vs 宽权衡**：同参数量可深而窄或浅而宽。经验区间 `d_model ≈ 64 × 层数` 附近（这里的"宽"就是第 7 节的 hidden size H；层数是深、H 是宽，二者正交，按总参数预算分配）。

实际流程：定目标(任务+SLA+数据量) → 选成熟 baseline 沿用其层数 → 算力不够就降档(蒸馏/换小模型) → 在 12/24 等档位做对照实验，选性价比拐点 → 上线用蒸馏/剪枝压到满足 SLA 的最小层数。

### 案例：层数设多/设少，对 `"我喜欢吃苹"→"果"` 的影响

这个例子很简单，正好说明"层数不是越多越好"：

| 层数 | 效果 | 本质 |
|------|------|------|
| **太少**(1~2) | 来不及建立"吃↔苹↔果"关系链，可能预测出泛泛的字或搞错搭配 | 深度不足 → **欠拟合** |
| **刚好**(12~32) | 有足够步数走完"认字→建立关系→收敛到果"，稳稳答对 | 容量匹配 |
| **太多**(100 去学这种简单题) | 答案不变，但浪费算力/显存、延迟高、训练更难、数据不够时**过拟合**、边际收益递减(后面的层空转) | 容量过剩 |

> **关键直觉**：层数要匹配任务的"推理步数复杂度"。
> - `"我喜欢吃苹"` 补全成 `"果"` 是**一步搭配补全**，少量层就够。
> - `"把所有质数相加再除以 3 余几……"` 是**多步推理**，需要更多层一步步推。
>
> 同一个模型的层数按**最难的那类任务**定上限（要能多步推理），对"果"这种简单题就是绰绰有余、用不满。**按最复杂业务需求定上限，简单任务自然覆盖。**

## 7. hidden size：4096 维向量到底是什么，浮点数怎么装语义

前面一直说"向量""残差流"，但**每个 token 的向量到底长什么样、那几千个数字凭什么代表语义**，是初学最绕的一个坎。先把概念对齐：

> 📖 **对照案例**：H=4096 就是「苹」这个字被表示成 **4096 个浮点数**。「苹」的"水果含义"不在第几维，而在这 4096 维合起来指向的方向——它离「果」「桃」这些字的向量近，离「我」「喜」远。本案例每个字都是这样一条 4096 维向量。

- **hidden size（隐藏维度 H）= 每个 token 在模型内部被表示成的向量有多长。** H=4096 就是每个 token 用 4096 个浮点数表示。它是"模型宽度"，和"层数"（模型深度）正交：层数是流水线多长，hidden size 是流水线多宽。两者在总参数预算下相互权衡（第 6 节"深 vs 宽"：经验区间 `d_model ≈ 64 × 层数`）——同样的参数量，可以深而窄，也可以浅而宽。
- 这个向量从 embedding 查表产出（第 2 节的 `x0`），在残差流里一路被改写（第 3 节），长度自始至终是 H 不变。

### 关键结论：语义不在"某一维"，而在"整个向量的方向"

一个 token 的向量就是一串没有标签的浮点数：

```
[ 0.23, -1.07, 0.0004, 2.55, -0.88, ... ]   ← 一共 4096 个数
   ↑第0维  ↑第1维         ↑第3维   （每一维只是个数，没有人类可命名的含义）
```

> ⚠️ **核心误区**：以为"第 37 维 = 表示是不是动物"这种一维一义的对应。
> **真相：单看任何一个维度基本看不懂。语义不是装在某个维度里，而是体现在"整个 4096 维向量指向哪个方向、和别的词的向量离多近"。**

正确的图景是：把每个 token 看成 **4096 维空间里的一个点**，语义来自这些点的**相对几何关系**：

- **意思相近的词，点离得近**（夹角小）。"猫/狗/仓鼠"聚成一团，"积分/矩阵"聚成另一团。
- **某个属性 ≈ 空间里的一个方向**（一组维度的线性组合，不是单维）。经典例子：
  ```
  向量(国王) − 向量(男人) + 向量(女人) ≈ 向量(女王)
  ```
  说明"性别"大致对应一个方向；"复数""过去时""褒贬"各自也是一个方向。

所以"浮点数怎么装语义"的答案是：**单个浮点数不装语义；4096 个数共同确定了这个词在语义空间里的位置与朝向，语义来自它与其他词位置的关系。** 类比地图：一个城市的"经度"本身没意义，是"经纬度合起来 + 和别的城市的相对位置"才有意义。

### 为什么要 4096 维这么多

维度多 = 空间能容纳更多**互不干扰（近似正交）的方向**。

- 维度太少（如 2 维），所有词挤在一张纸上，"词性/时态/情感/主题"这些属性会互相冲突、没法同时分清。
- 4096 维里能塞下成千上万个互不打架的方向，模型可以**同时**用不同方向编码"词性、指代、情感、主题、句法角色……"，各管各的。

这就是 hidden size 的几何意义：**维度数 = 语义空间里能同时容纳多少条独立的"语义坐标轴"。** 维度越大 → 表示容量越大 → 潜在能编码的语义差别越多（但能否真用上取决于数据量与训练是否充分，且参数量、算力随之按约平方增长，见第 4 节与第二部分的 `H×H`、`H×4H`）。

### 向量不是固定的：同词随上下文挪位

同一个词"苹果"，在"我吃苹果"和"苹果发布会"里 embedding 查表给的初始向量一样，但经过每层加工后被改写到不同位置——一个飘向"水果"区，一个飘向"公司"区。**这正是第 4 节"层内加工"在做的事**，也是第 3 节残差流"逐层精炼"的几何说法。

> **一句话归纳**：向量 = 词在语义空间里的坐标；hidden size = 这个空间有多少维（多少条语义坐标轴）；语义藏在坐标之间的相对方向里，不在任何单个维度里。

## 8. 案例串讲：输入「我喜欢吃苹」→ 预测「果」（一次完整前向 + 反向）

> 前 7 节把概念拆开讲透了；这一节用一个最小可跟手算的例子，把 embedding（第 2、7 节）、前向（第 3、4 节）、反向与训练循环（第 3 节）串成一条完整链路，看一句话怎么被两层 Transformer 雕成「下一个字是果」。

> 📖 **本节就是贯穿全篇那个案例的完整版**：前面每个 `> 📖 对照案例` 引用块都是这条链路的一个切片，这里从头到尾走一遍（embedding → 2 层前向 → 算 Loss → 反向更新）。

### 8.0 案例设定
1. 任务：自回归大模型（LLaMA/GPT 类 Decoder）
2. 隐藏维度 `H = d_model = 4096`：每个 Token 对应一条长度 4096 的浮点向量（即第 7 节的 hidden size）
3. 模型结构：2 层 Transformer block（方便看清逐层流转）
4. 输入序列：`T0=我、T1=喜、T2=欢、T3=吃、T4=苹`，共 5 个 Token
5. 目标标签：下一个字 `GT=果`，用交叉熵 Loss 计算预测误差
6. 张量统一尺寸：`[序列长度5, 4096]`，5 个字各占一条独立 4096 维浮点数组

### 8.1 4096 维向量到底是什么？单个浮点数如何承载语义

**向量本质**
每个 Token 对应 **4096 个独立浮点数**（浮点数即小数：比如「喜」在某一维真实存储的 BF16 值是个浮点数 0.8125），范围大致在 `[-1.5, 1.5]`。呼应第 7 节「语义不在某一维，而在整个向量的方向」——下面的「通道」只是为了直觉示意，真实语义是多维线性组合而非单维独占：

- **通道组 A（32、108、567、1209 维）：正向情绪方向**
  训练完成后「喜」这里数值 ≈ +0.8~1.2（高激活）；「怒」≈ -0.9（抑制）；「苹」≈ 0（无情绪）
- **通道组 B（211、780 维）：心理动词方向**
  「喜」数值为正，代表可作动词；物品名词此处接近 0
- **通道组 C（3001、3666 维）：搭配「欢」方向**
  「喜」该方向数值偏高，和「欢」向量叠加后匹配度拉满

剩余几千维分别编码字形、褒贬、语境、使用频率、多义场景等细碎信息。

**关键疑问：词嵌入查表是随机数，还是自带情绪特征？** 分两个阶段，完全区分开。

**阶段 1：模型刚初始化，未开始训练**

- *初始化嵌入矩阵*：进 embedding 层查模型厂家的词表，矩阵形状为 `W_emb [V×H]`（即第 2 节的查找表），全部由高斯随机生成浮点数。「我 喜 欢 吃 苹」5 个字查表后输出 `5×4096` 张量；这个张量形状在前向传播时一直在 block 之间传递（就是第 3 节的残差流 `x`）。
- 查表的意义是**获取行编号**，而不是查「喜」是什么意思。比如 Qwen-7B 内部「喜」固定 `ID=3621`，查表永远取第 3621 行。行号只为让「喜」在每层更新时和别的字区分、不混淆。这一行此时是 4096 个随机浮点数，**没有任何语义**。
- *临时特征张量 vs 模型权重*：这个 `5×4096` 张量不是权重，只是流过 block 的激活值（第 1 节强调 Norm 归一化的也正是这种激活值）。每过一层被改写、传到下一层，上一层不保存。
- *每层 block 权重完全独立*：呼应第 3 节「结构同构、数值各异」——每层都有自己独立的 `W_Q/W_K/W_V/W_O [H×H]`、`W_up/W_dn`（详见第二部分权重清单），互不共享。
- *Embedding/LM Head 是全局共享参数*：词表查表权重全局唯一，反向时统一更新「喜」对应的那一行；LM Head 很多模型与 embedding 权重共享（weight tying，见第二部分）。
- *嵌入向量跨 batch/epoch 持续更新*：「喜」对应的 4096 行浮点数在整个训练里持续保存、迭代——
  - 初始化：`喜 = [0.1, -0.8, 0.6 ...]`
  - Batch1 训练后：`喜 = [0.2, -0.7, 0.65 ...]`
  - 之后不管「喜」出现在哪句话，都直接读当前值继续优化；上万轮后向量稳定成型，自带正向情绪语义。
  - 只有整个 batch 跑完前向 + 反向、算出梯度后，优化器才改写这一行（第 3 节训练循环的第④步）；不会自动退回随机值。但**下一个模型**重新训练时仍从随机初始化开始。

**阶段 2：千万轮训练收敛后**
海量文本的 Loss 梯度持续修改「喜」对应的 4096 个浮点数：见「喜悦/喜事」→ 拉高正向情绪方向；见「不喜」→ 压低；高频「喜欢」→ 拉高搭配方向。训练完成后查表直接读出一套固定浮点组合，**向量本身自带该字基础语义**。

### 8.2 前向传播完整链路（我喜欢吃苹 → 预测果）
整体：词嵌入 H₀ → 第 1 层 block → H₁ → 第 2 层 block → H₂ → 取末尾 Token 预测 → 算 Loss。每层做的「读残差流 → 算增量 Δ → 加回」正是第 3、4 节的机制。

**步骤 1：生成初始嵌入 H₀[5,4096]**
每个字查表得到 4096 维向量，叠加位置编码：
- `H0[0]「我」`：人称相关方向激活
- `H0[1]「喜」`：训练后自带正向情绪、动词方向高值
- `H0[2]「欢」`：和「喜」配套情绪方向
- `H0[3]「吃」`：进食动作、期待宾语方向激活
- `H0[4]「苹」`：轻度水果方向激活，缺少“果”后缀特征

**步骤 2：第一层 block 加工（Pre-Norm + 双重残差）**

*2.1 Attention 子层（学习字与字的关联，第 4 节的横向混合）*
1. 残差 shortcut：`res1 = H0`，完整复制 5 条原始向量
2. Norm 标准化（第 1 节稳压器）
3. QKV 投影（`W_Q/W_K/W_V`，4096→4096）算 Token 相关性：「喜」↔「欢」互相匹配强化搭配；「吃」匹配「苹」建立动宾关系
4. 输出融合上下文的 `AttnOut[5,4096]`，再过 `W_O`
5. 残差相加 `x_attn = res1 + AttnOut`：不丢原生语义，只叠加搭配关系增量，「苹」新增“前面有进食动词”特征

*2.2 MLP 子层（各 token 自己深加工，第 4 节的纵向提炼）*
1. 残差 shortcut：`res2 = x_attn`
2. Norm，`up_proj` 升维 → SiLU → `down_proj` 回落 4096，细化各方向激活：拉高「吃」“等待食物宾语”、轻微强化「苹」水果属性
3. 残差相加得第一层输出 `H₁ = res2 + MlpOut`

> 第一层小结：完整保留 5 个 Token 的 4096 维特征，注意力建立句内关联、MLP 细化语义，整条 `5×4096` 无压缩传入第二层。

**步骤 3：第二层 block 加工（深化全局语义）**
输入即 H₁，流程一致：残差保留第一层特征 → 注意力再放大「吃+苹」动宾链路、大幅拉高「苹」的“水果待补全后缀”方向 → MLP 进一步强化「苹」和“果”的隐式关联 → 残差相加得最终表征 `H₂[5,4096]`。`H₂[4]` 就是两层后的「苹」专属向量，蕴藏“苹果”补全信息。

**步骤 4：预测文字 + 计算 Loss（head 只翻译，不思考，见第 3 节问题一）**
1. 取末位 Token 向量：`h_last = H₂[4]`
2. LM Head 线性投影 4096 → 词表大小，输出每个字的 logits
3. 训练初期模型大概率给“桃、蕉”高分，和真实标签「果」有差距
4. 交叉熵 Loss 生成标量误差，衡量预测出错的程度

### 8.3 反向传播：Loss 如何逐层更新 4096 维向量与权重
核心结论（呼应第 3 节问题二「责任连带」+ 第二部分「一个 W 被 4 个算子访问」）：每层 block 参数独立、梯度独立；残差连接提供无衰减梯度直通通道，避免梯度消失。误差根源是「苹」的特征没能推出「果」，梯度从 head 往前逐层回传。

1. **先更新第二层权重**：`∂Loss/∂H₂` 作用于末位「苹」向量 → 第二层 MLP 梯度调整方向映射，让下次处理「苹」更强激活“果”；第二层注意力梯度强化「吃」对「苹」的关注；残差直通梯度（导数恒为 1）无衰减传到 H₁。
2. **流入第一层**：`∂Loss/∂H₁` 反向走第一层 → MLP 梯度让「吃」原生自带“等待水果后缀”；注意力梯度强化「吃-苹」动宾匹配；残差继续把梯度直通到最初的 H₀。
3. **改写词嵌入**：梯度直接改写 `W_emb` 中「苹」那一整行 4096 个浮点数，微调水果相关方向（即第二部分「embedding 反向统一更新整行」的具体一例）。

> **直观训练效果**：本轮预测错 → Loss 大 → 强梯度修正所有层级权重；下一轮再算「我喜欢吃苹」，两层输出的 4096 维向量更倾向输出“果”，Loss 持续下降。这正是第 3 节「几十万次 预测错→反向→微调」把分工压到不同层上的微观一例。

# 第二部分 · 层、权重矩阵、算子的关系（深入层内部）

第一部分第 4 节已从"做什么"打开了一个 block（Attention/MLP 的前向算子序列）；这一部分换个角度，系统看它由哪些**权重矩阵**组成、各占多少显存、又如何被切到多张卡——这正是 demo 右侧"整网算子图"和显存估算的依据。

> 📖 **对照案例**：上一部分跟着「苹」的向量走完了一层；这一部分换个角度，看每个 block 里到底有哪些**权重矩阵**（W_Q/K/V/O、W_up/W_dn…）在加工「苹」，它们各占多少显存、又怎么切到多张卡。

## 层级结构

> 📖 **对照案例**：「苹」向量在一个 block 里依次被这些权重加工——Attention 的 W_Q/K/V/O 让它看「吃」，MLP 的 W_up/W_dn 让它深加工；开头 embedding 的 W_emb 负责查表，结尾 head 的 W_head 负责选出「果」。

```
Transformer Block
├── Attention 层
│   ├── 权重矩阵：W_Q [H×H], W_K [H×H], W_V [H×H], W_O [H×H]
│   └── 算子：QKV MatMul, Scaled Dot-Product, Out MatMul, all-reduce, 残差加
├── MLP 层（标准）
│   ├── 权重矩阵：W_up [H×4H], W_dn [4H×H]
│   └── 算子：up MatMul, SiLU, down MatMul, all-reduce, 残差加
├── Embedding 层
│   ├── 权重矩阵：W_emb [V×H]
│   └── 算子：Embedding lookup（gather，非 MatMul）
└── LM Head 层
    ├── 权重矩阵：W_head [H×V]（通常与 W_emb 共享）
    └── 算子：lm_head MatMul, all-gather（若 TP>1）
```

## 一个层有几个权重矩阵？

> 📖 **对照案例**：本案例每个 block 的 Attention 有 4 个权重矩阵（W_Q/K/V/O），MLP 有 2 个（标准）或 3 个（SwiGLU）；而加工「苹」时经过的 Softmax、SiLU、残差加都是 **0 权重**的纯计算——不占显存，所以后面拓扑图里不画格子。

**0 个、1 个、多个都有可能**，取决于层的类型：

| 权重矩阵数量 | 层类型 | 说明 |
|---|---|---|
| **0 个** | Softmax、SiLU、ReLU、残差加法、Dropout | 纯计算，无可学习参数 |
| **1 个** | Embedding、LM Head | 一个大矩阵做 lookup 或投影 |
| **2 个** | 标准 MLP | W_up + W_dn |
| **3 个** | SwiGLU MLP（LLaMA 风格） | W_gate + W_up + W_dn |
| **4 个** | Attention（非融合） | W_Q + W_K + W_V + W_O |
| **4 + 2×E 个** | MoE 层 | W_router + E 个专家（每个专家 2 个矩阵） |

> LayerNorm/RMSNorm 有 γ、β 两个参数向量（长度 H），但它们是**向量不是矩阵**，对应逐元素乘加而非 MatMul，参数量极小，通常单独讨论。

> 💡 呼应第一部分第 3 节："结构相同"指的就是这张表里**算子种类和连接方式**每层一样；"功能不同"指的是这些 W_Q/W_up… 里的**数值**每层经训练而分化。

## 权重矩阵与算子的对应关系

> 📖 **对照案例**：给「苹」算 Q/K/V 时，若 W_Q/K/V 被拼成一个大矩阵一次算完，就是"多权重:1 算子"的融合 kernel；开头查「苹」和结尾选「果」若用同一块权重（weight tying），就是"1 权重:多算子"。

基本规则：**1 权重矩阵 : 1 MatMul 算子**（前向 pass）

| 对应关系 | 场景 | 例子 |
|---|---|---|
| 1 权重 : 1 算子 | 标准情况 | W_up → up_proj MatMul |
| 多权重 : 1 算子 | 融合 kernel | W_Q/K/V 拼成 [H×3H] → 一次 fused QKV MatMul |
| 1 权重 : 多算子 | Weight tying | W_emb = W_head，前端 lookup + 后端 lm_head 各用一次 |
| 0 权重 : 1 算子 | 纯计算 | Softmax, SiLU, all-reduce, 残差加法 |

## 一个权重矩阵在完整训练 step 中被多少个算子访问

> 📖 **对照案例**：加工「苹」用的 W_Q，在一个训练 step 里被访问 4 次——前向用它算 Q、反向用它算传给上一层的梯度、反向算它自己该改多少、优化器再更新它。这正是每个参数要存 16 B 的来由。

demo 里的算子图只展示**前向** pass，但一次完整 training step 里，同一个 W 会被 4 个算子依次访问：

```
前向    output = x @ W          → 消费 W，产出激活值
反向①   grad_x = grad_out @ Wᵀ → 消费 W，产出传给上一层的梯度
反向②   grad_W = xᵀ @ grad_out → 读 x，写 grad_W（梯度累积）
优化器  W -= lr · Adam(grad_W, m, v) → 读写 W、m、v
```

> 💡 这正是第一部分第 3 节"联合训练"的微观细节：反向①把"责任"往上一层传（梯度倒流），反向②算出本层该怎么改。残差连接保证 `grad_x` 能几乎无损传回浅层。

这解释了为什么混合精度训练下每个参数需要 **16 B/param**：
- bf16 权重 2B + fp32 主权重 4B + Adam m 4B + Adam v 4B + 梯度 2B = 16B

## MoE 层特殊性

> 📖 **对照案例**：若把 MLP 换成 MoE，「苹」会先经 router 被分给 top-k 个专家（比如负责"食物/水果"的专家），只有那几个专家加工它，其余专家对「苹」空转——这就是稀疏激活。

```
x [B, seq, H]
      ↓
W_router [H × E] → gate / router（决定每个 token 去哪个专家）
      ↓  top-k dispatch
all-to-all dispatch            ← 仅 EP>1
      ↓
W_exp_i [H × 4H] → Expert_i FFN（只有被选中的 token 流经）
      ↓
all-to-all combine             ← 仅 EP>1
      ↓
x + residual [B, seq, H]
```

MoE 的 1 个专家 = 2 个 MatMul（up_proj + down_proj），但每个 token 只激活 top-k 个专家（k 是模型设计决定的超参数：Mixtral 用 k=2，DeepSeek-V3 用 k=8；Router 打分后取分数最高的前 k 个），所以是**稀疏激活**。

### MoE 的共享专家（Shared Expert）

> 📖 **对照案例**：若 MoE 带共享专家，「苹」除了走 router 选中的专家，还会**强制多走一路共享专家**——这路不经过路由、所有 token 都走。两路并行计算后相加：`Output = SharedExpert(x) + Σ(路由专家输出)`。

**流程**：每个 token 同时走两路——共享专家提供通用基底（必走），路由专家提供领域增强（top-k 稀疏）。它俩不是"先训共享再训路由"的先后关系，而是**同一趟前向里并联计算、同一次反向里一起更新**——共享专家自己也在被训练。

共享专家解决传统 MoE 三个天生缺陷：

- **消除知识冗余**：基础语法、主谓宾、标点等通用知识不必每个路由专家重复学一遍，共享专家单独承包，路由专家只专攻细分领域。
- **兜底防崩**：路由打分全低或冷门 token 无匹配专家时，共享专家提供稠密基础表征，避免生成乱码、loss NaN。
- **缓解路由失衡**：共享专家吸收通用计算后，路由专家流量天然更均匀，负载均衡 loss 权重可大幅降低。

**在并行中的分布**：共享专家是稠密 FFN，跟随所属层走 PP/TP/DP，**绝不参与 EP 拆分**——PP 把它绑定在本层所在 stage、TP 沿维度切分权重、DP 每副本完整复制，EP 仅路由专家走 all-to-all。一句话：**每层自带独立共享专家，跟层走、可 TP 切、可 DP 拷，绝不走 EP。**

## DEMO拓扑图里"落在卡上的格子"为什么代表权重参数

> 📖 **对照案例**：拓扑图里写着"Attn-0""MLP-0"的格子，含义是"这张卡存了加工「苹」时这一层的权重（或其分片）"。无权重的 Softmax/残差加不画格子，因为它们不占显存。

中间拓扑图中，每张卡（`.wdiag-card`）里竖排的每个**格子**（`.wdiag-layer-box`）标签写的是层名（Embedding / Attn-0 / MLP-0 / Norm / Head），但它在显存与资源意义上等价于**这一层在本卡上持有的那份权重（或权重分片）**。三个原因：

1. **一张卡承载什么 = 它存了哪些权重。** 显存估算只累加各层的权重参数量：attn `4·H²/tp`（W_Q/K/V/O）、mlp `8·H²/tp`（up+down）、emb/head `V·H/tp`。"层落在哪张卡"被直接换算成"这张卡存了多少权重"。
2. **层的"身份"几乎就是它的权重。** 有权重的层（Embedding/Attn/MLP/Head）必须把权重矩阵放进某张卡显存，才占资源、才需要被并行切分；无权重的纯计算层（Softmax/SiLU/残差加/all-reduce）不占参数显存，因此**不画成格子**。
3. **TP 分片时格子标签直接写"第 t/tp 片"，切的就是权重矩阵**（列切/行切）。悬停格子弹出的明细把名字改成"…权重矩阵"，是刻意点明：格子 = 该层在本卡上持有的权重分片。

> 一句话：格子标签写的是"层名"，编码含义是"这张卡要存这一层的权重（或其一个分片）"。

# 第三部分 · 并行训练（权重如何切到多张卡）

第二部分说清了"一个 block 有哪些权重"；当模型大到单卡放不下，就要把这些权重切开放到多张卡上——这就是并行训练。

> 📖 **对照案例**：「我喜欢吃苹」这个 2 层小模型单卡放得下。但若换成 70B 大模型，加工「苹」用的那些权重单卡根本装不下，就得把它们切到多张卡——本部分讲怎么切。

## 1. 一次训练 step 在算什么

> 📖 **对照案例**：喂进「我喜欢吃苹」、目标「果」，一个 step 就是：前向预测（初期可能先猜成「桃」）→ 算 Loss（和「果」的差距）→ 反向算每个权重该怎么改 → 优化器更新。下次再喂这句，更可能输出「果」。

目标：调整权重 **W**，让预测更准。每个 step 四步：

```
① 前向 forward    output = x @ W           用当前权重做预测
② 算损失 loss     比较预测与正确答案，得到误差
③ 反向 backward   grad_W = xᵀ @ grad_out   算出"W 该往哪改"
④ 优化器 optimizer W -= lr · Adam(grad_W)  真正更新权重
```

关键：同一个 W 在一个 step 里被访问 4 次（前向、反向算输入梯度、反向算权重梯度、优化器更新）。

> 💡 这是第一部分第 3 节训练循环的"单层权重视角"；那里讲的是整网联合收敛，这里聚焦单个 W。

## 2. 为什么显存爆炸 → 必须并行

> 📖 **对照案例**：本案例 2 层小模型无所谓；但若加工「苹」的是 70B 权重，乘 16 B/param = 1.1 TB，单卡 80GB 根本放不下——这才逼出并行。

混合精度训练下每个参数存 5 份东西，共 **16 B/param**：

| 项目 | 大小 | 用途 |
|---|---|---|
| bf16 权重 | 2B | 前向/反向计算 |
| fp32 主权重 | 4B | 高精度累积更新 |
| 梯度 | 2B | 反向产出的 grad_W |
| Adam m | 4B | 动量（一阶矩） |
| Adam v | 4B | 方差（二阶矩） |
| **合计** | **16B** | |

70B 模型 → 70B × 16 = **1.1 TB**，远超单卡 80GB。这就是并行训练的根本动机。

## 3. 四/五种并行（picotron 的 dp×pp×cp×tp 网格）

> 📖 **对照案例**：把"加工「我喜欢吃苹」"这件事拆给多卡的不同方式——**DP**：多份模型副本各算一批不同句子；**PP**：第 1 层放卡 A、第 2 层放卡 B 接力；**TP**：把加工「苹」的某个权重矩阵劈成几片；**CP**：把"我喜欢吃苹"这 5 个 token 按位置切给不同卡（序列很长时才需要）。

总卡数 World Size = DP × PP × CP × TP。每种切法切的东西不同，且**彼此正交可叠加**：

| 并行 | 切什么 | 直观理解 | 拓扑图表现 |
|---|---|---|---|
| **DP** 数据并行 | 切数据 batch | 每卡存完整模型副本，各算一部分样本 | 顶部 batch 分裂成 shard，横向多个 DP 副本 |
| **PP** 流水线并行 | 切层（深度方向） | 前几层放卡 A、后几层放卡 B，接力 | 纵向 Stage 0→1，`↓ PP` 箭头；层按连续均分分配 |
| **TP** 张量并行 | 切单个权重矩阵（宽度方向） | 一个大矩阵劈成 tp 份，各卡算一片再通信合并 | 同一行内 `↔` 连接的多卡，格子标"第 t/tp 片" |
| **CP** 上下文并行 | 切token序列长度。虽然可以用超长文本滑动截断做训练数据预处理控制长度，但是截断后，单条样本窗口 W 如果还是太大，单卡显存不够用，就需要进一步用CP | 假设输入长句：我、喜、欢、吃、苹、果、今、天、很、甜、每、次、吃、都、开心，总长度 L=15，CP 并行度 = 3（3 张 GPU）：rank0：我、喜、欢、吃、苹；rank1：果、今、天、很、甜；rank2：每、次、吃、都、开心 | 紫色左边框的 CP 分组 |
| **EP** 专家并行 | 切 MoE 专家 | 不同专家放不同卡，token 按路由分发 | MoE 模式下专家按卡分配 + all-to-all |

> 💡 **PP 为什么按 block 整数切**：呼应第一部分第 1 节——block 是可堆叠的最小重复单元，每个 block 计算量/显存近似相同，所以按层整数切给不同 stage 最均匀。这也是"层数"以 block 为单位的工程价值。

> 💡 **CP 的典型场景＝长序列（尤其视频/长文本）**：序列越长，注意力的激活值显存随 seq 增长甚至按 `seq²` 涨，单卡放不下。把序列按 token 位置切给多卡，每卡只算自己那段，正是 CP 的用武之地。呼应第一部分第 2 节"多模态输入"——视频帧 patch 化后 `seq = 帧数 × 每帧 patch 数`，极易暴涨，是 CP 最常见的现实动机之一。

## 4. 通信规约（reduce）出现在哪——为什么不在 block 之间

> 📖 **对照案例**：单卡跑「我喜欢吃苹」时，上下 2 层之间只有残差加，**没有任何 all-reduce**。只有把「苹」的权重切到多卡（TP）才在每个子层末尾冒出一次 all-reduce 来缝合；多副本训练（DP）才在 step 末尾同步梯度。规约不是 Transformer 层间的固有环节，而是出现切分后所需的"缝合动作"。



**通信规约相对"层"到底在哪**（关键：几乎都不在 block 与 block 之间）：

| 并行 | 规约在哪 | 是不是"层间" |
|---|---|---|
| **单卡** | 没有通信规约，层间只有残差加 | — |
| **TP** | 每个**子层末尾**一次 all-reduce（Out Proj / down_proj 行切只算出"部分和"，必须求和）→ 一个 block 内 **2 次**（Attn、MLP 各 1 次） | 在 **block 内部、子层边界**，不是 block 之间 |
| **PP** | block 跨 stage 传递用**点对点 send/recv**，把激活值发给下一卡 | 是层间通信，但**是传递不是规约** |
| **DP** | 梯度 all-reduce，在**反向结束、优化器更新前**一次性跨副本规约全部参数的梯度 | 在 **step 末尾**，与"哪两层之间"无关 |

---
### PP stage 之间到底 send/recv 了什么——具象到「我喜欢吃苹」

上一段说 PP "靠 send/recv 传递"，那**这一发到底发的是什么**？初学最容易猜成"传 token"或"传 token + 一份额外的残差"。**都不对。两张卡之间传的，是边界 block 出口那条「残差流」，形状 `[batch, seq, H]` 本身**——既不含原始 token ID，也没有"x 之外再附一份残差"。

把案例的 2 层模型按 PP=2 切成两段（每段一台卡）：

```
                              ┌─ 卡A send 出去 / 卡B recv 进来：就是这个 x ──┐
 Stage 0（卡A）: Embedding → block 1  ──────[ x: batch×5×4096 ]──────►  block 2 → Norm → Head  Stage 1（卡B）
                  ↑ token ID 只活到这                                       ↑ 卡B 从不碰 token ID，只接着加工 x
```

> 📖 **对照案例（先看 1 个句子这一片）**：跨卡传的那个 `x`，**先只看「我喜欢吃苹」这一句,就是一张 5 行 × 4096 列的浮点数表**——每个字占一行，每行是它**流到卡A出口（block 1 算完）时此刻的 4096 维向量**：
>
> ```
> x（batch 里的第0个句子）=
>       [ 我 的当前向量 :  0.21, -0.88,  1.30, …  (共4096个数) ]   ← 第0行
>       [ 喜 的当前向量 : -0.40,  0.77,  0.05, …            ]   ← 第1行
>       [ 欢 的当前向量 :  0.12,  0.66, -0.91, …            ]   ← 第2行
>       [ 吃 的当前向量 :  0.88, -0.10,  0.34, …            ]   ← 第3行
>       [ 苹 的当前向量 :  0.05,  1.12, -0.20, …            ]   ← 第4行  ← 预测下一字只靠这一行
>       └──────────────── 这一片形状 [seq=5, H=4096] ────────────────┘
> ```
>
> 关键是看清「苹」那一行**此刻是什么状态**：它已经不是 embedding 刚查出来的孤立字向量了——在卡A的 block 1 里，Attention 已经帮它把前面「吃」的"进食动词"信息搬了进来（第一部分第 4 节），MLP 也消化过一轮。但 block 2 才负责的"收敛到『该接果』"这一步**还没发生**（block 2 在卡B上）。所以传过去的 `x`，是**「加工到一半、刚走完前半段流水线」的隐状态快照**：前半段的理解已经累加进去了，后半段的理解还没加。卡B 接过这张表，从 block 2 继续往上加 `Δ`，最后让「苹」那一行（第4行）足以在 Head 处给「果」打最高分。
>
> ### 那 `[batch, seq, H]` 里的 batch 在这个案例里具体是什么
>
> 上面那张 `5×4096` 的表，其实只是**整个张量的一"层"切片**——它对应 **batch 里的 1 个句子**。`batch`（批大小）= **这一个 step 里被叠在一起、同时塞进模型并行处理的「独立句子」有多少条**。前面为了讲清楚,一直只画「我喜欢吃苹」这一句,等于**默认 batch=1**;真实训练为了喂饱 GPU、让梯度更稳,一个 step 通常一次塞几十~几千句。
>
> 假设这个 step 的 batch=4，那除了「我喜欢吃苹」，同批还有另外 3 句各自独立的训练样本（每句也都在预测自己的下一个字）：
>
> ```
> batch 维（4 个互不相干的句子，叠成一摞）：
>   句子0:  我 喜 欢 吃 苹   → 要预测「果」      ← 上面那张 5×4096 表就是这一片
>   句子1:  今 天 天 气 真   → 要预测「好」
>   句子2:  他 正 在 看 电   → 要预测「视」
>   句子3:  明 天 就 要 放   → 要预测「假」
>
> 整个激活张量 x 的形状 = [batch=4, seq=5, H=4096]
>                          └ 4 张 5×4096 的表摞在一起，共 4×5×4096 个浮点数 ┘
> ```
>
> - **batch 这一维上的 4 个句子彼此完全独立、互不看对方**：Attention 只在**同一句内部**的 token 间发生（「苹」看的是同句的「吃」，绝不会去看句子1 的「天气」）。叠 batch 纯粹是为了"一次多算几句、把 GPU 喂满 + 梯度按多句平均更稳"，不产生跨句的信息混合。
> - **PP 传的就是这个完整的 `[4,5,4096]`**：卡A 把 4 个句子各自算完 block 1 的结果**一整摞**send 给卡B,不是一句一句发。
> - **batch 和 DP 的关系**：如果再开数据并行 DP，就是把这 4 句的大 batch **再切成几份分给不同副本**（如 DP=2 → 每副本各算 2 句），每副本内部的 PP 仍按上面方式传自己那半批的激活（见第 3 节 DP 行、第三部分 DP 梯度 all-reduce）。
> - （真实工程里同批句子长短不一,会用 padding 补齐到同一个 seq,或按长度分桶；这里为讲清楚让 4 句都取 seq=5。）

为什么说它"既不是 token、也不是单独的残差"：

| 你可能以为传的是 | 实际情况 |
|---|---|
| **原始 token ID**（我=101、喜=102…的整数） | ❌ token ID 一进 Embedding 就被查表换成向量了，**只活在 Stage 0 入口**；卡B 拿到的全是已加工的 `H` 维浮点向量，从不见原始字 |
| **token 向量 + 另附一份"残差"** | ❌ 没有两份东西。传的就是**残差流 `x` 这一份**——它本身已经是 `x = x0 + Δ(block1)` 累加后的"当前完整隐状态"（第一部分第 3 节），不是"基础向量再外挂一个残差量" |
| **当前 block 这一层算出的增量 Δ** | ❌ 传的是**累加后的完整 `x`**，不是单层增量 Δ。Δ 是层内部的中间量，加进 `x` 后就并入主流，不单独跨卡传 |

前向、反向各传一次，方向相反：

- **前向**：Stage 0 把出口激活 `x [batch,5,4096]`（batch=4 时即 `[4,5,4096]`）**send** 给 Stage 1，当作 block 2 的输入。
- **反向**：Stage 1 把"对这个 `x` 的梯度" `dL/dx`（**同样形状 `[batch,5,4096]`**）**send** 回 Stage 0，让前半段继续往浅层回传、更新 block 1 与 Embedding 的权重。

> ⚠️ **误区**：以为 PP 像 TP/DP 那样要 all-reduce 把数据"合起来"。不是——PP 是**点对点单向投递**（卡A→卡B 一根管子），只是把一段算完的激活原样递给下一段，**没有跨卡求和**（见上表 PP 行）。所以 PP 的通信量 ≈ 边界激活的大小 `batch×seq×H×字节数`，和"切了几片权重"无关。

> ⚠️ **别误会 `x` 只在 stage 之间才传**：`x` 是 Transformer **每一对相邻层之间都要交接**的东西（第一部分第 3 节残差流，单卡时照样发生）。**层间传 `x` 是 Transformer 本来就有的（每层都传）；PP 只是把"碰巧跨在两张卡之间的那一次层间传递"从免费的显存传递，换成了要走网络/NVLink 的 send/recv。** 同卡内的层间交接仍是免费显存传递；跨卡通信只长在 `PP-1` 条 stage 缝上（如 80 层、PP=4 → 只有 3 次 send/recv，其余 76 次都在卡内免费传）。

> **一句话归纳**：PP stage 之间传的是**边界 block 出口的残差流激活 `x [batch,seq,H]`**——具象成案例就是那张「5 字 × 4096 维、加工到一半」的浮点数表；前向递 `x`、反向递它的梯度 `dL/dx`，靠 send/recv 点对点投递，不含 token、不做规约。

# 第五部分 · 训练 vs 推理：两种部署形态与各自的并行

> 前四部分的主线是**训练**（权重在变、要反向、要优化器）。但同一个模型还有另一个完全不同的生命阶段——**推理**（权重已固定、只做生成）。很多开源模型发布的是"推理包"，里面的"并行配置"和训练并行**不是一回事**，初学极易混淆。这一部分把两者彻底分开。

> 📖 **对照案例**：前面讲的都是"用「我喜欢吃苹」**训练**模型（权重在变）"。但模型训好后，拿它来真正**生成**出「果」是另一种形态——推理。两者的并行配置不是一回事。

## 1. 训练 vs 推理：到底差在哪

> 📖 **对照案例**：**训练**时喂「我喜欢吃苹」+ 标签「果」，要前向+反向+优化器、每参数 16 B；**推理**时只喂「我喜欢吃苹」，只做前向生成出「果」，每参数约 2 B + KV cache。

| | **训练 (Training)** | **推理 (Inference)** |
|---|---|---|
| 在做什么 | 用海量数据**学习**，不断更新权重 W | W 已固定，**用模型回答/生成** |
| 计算 | 前向 + **反向** + 优化器更新（第三部分第 1 节四步） | **只有前向**，无反向、无优化器 |
| 每参数显存 | **16 B/param**（权重+主权重+梯度+Adam m/v，第三部分第 2 节） | 约 **2 B/param**（只有 bf16 权重）+ KV cache |
| 还需要什么 | 训练数据、loss、梯度、checkpoint、训练并行框架 | 权重 + 前向代码 + 一点采样/生成逻辑 |
| 谁来做 | 模型研发团队（如盘古团队，上千卡） | 部署方/使用者（少量卡甚至单卡） |
| 显存峰值在 | **反向**（激活+梯度并存，第四部分第 4 节） | **decode 的 KV cache**（第四部分第 4 节） |

**"推理包" = 只包含"跑模型做生成"所需文件的发布包。** 判断一个目录是不是推理包，看它**有没有反向/优化器/训练数据**：
- 有的：模型权重、配置（层数/hidden/MoE 结构）、前向实现（`model.py`）、生成脚本、推理并行配置（见下）、部署文档。
- 没有的：训练数据、loss、反向传播、优化器状态、训练启动脚本、训练并行（PP/DP/EP）配置。

> 💡 这解释了一个常见困惑：**从推理包里无法还原模型的训练并行策略**——那部分根本不在包里。能从推理包提取的只有①模型结构（config + model.py，对应第二部分的权重清单）②推理并行配置（下一节）。

## 2. 推理并行 vs 训练并行：切的目标完全不同

> 📖 **对照案例**：推理生成「果」只保留"前向那一半"切分——以 TP 为主，子层末尾 all-reduce 照旧，但**没有**反向、没有梯度 all-reduce、没有优化器状态。

模型太大单卡放不下时，推理同样要**切到多张卡**，但目标和训练不一样：

| | **训练并行**（第三部分） | **推理并行** |
|---|---|---|
| 目标 | 高效**训练**：放下 16B/param 的庞大状态 + 喂海量数据 | 让模型**跑起来、跑得快**：放下权重 + 压低延迟 |
| 常用维度 | **DP × PP × CP × TP**（+MoE 的 EP）全家桶 | 主要 **TP**（张量并行），大模型/MoE 再加 **EP / DP** |
| 不需要的 | — | **DP 切数据**意义不同（推理是切请求/并发，非梯度同步）；**PP** 推理也可用但更少；**无梯度 all-reduce** |
| 通信 | TP 子层末尾 all-reduce + DP 梯度 all-reduce + PP send/recv（第三部分第 4 节） | **只剩前向那部分通信**：TP 的子层末尾 all-reduce/all-gather、EP 的 all-to-all；**没有梯度同步** |

**核心**：推理并行本质是把第三部分的并行切分**只保留前向的那一半**——因为推理没有反向、没有优化器、不做梯度规约。所以推理的 TP 切法和训练**完全一致**（列切/行切，见第二部分"Tensor Parallel 下的变化"），只是不再有 step 末尾的 DP 梯度 all-reduce。

### 一个真实例子：盘古 718B MoE 推理包的并行配置

```yaml
# runner_config/tp32.yaml —— 32 卡张量并行推理
parallel_config:
  attn_tp_size: 32    # 注意力 128 个头 → 每卡分 4 个头（W_Q/K/V/O 列切+行切）
  moe_tp_size: 32     # FFN/MoE intermediate → 每卡 1/32（up 列切、down 行切）
  embed_tp_size: 32   # 15 万词表 → 每卡 1/32（embedding/lm_head 列切）
```

- 这就是第二部分"Tensor Parallel 下的变化"在推理侧的落地：注意力按头切、FFN 按中间维切、embedding/head 按词表切，子层末尾用 all-reduce/all-gather 缝合。
- 注意配置里**没有** PP/DP/EP 的训练含义，也没有优化器/梯度——印证第 1 节"推理包不含训练并行策略"。

> **比喻**：一道超大的菜（一次前向），单个厨师（单卡）做不完 → 把食材切成 32 份，32 个厨师各做一份（TP 并行），关键步骤交换半成品（子层末尾 all-reduce），最后拼成整道菜（输出 logits）。训练则是在这之上还要"尝味道→记下哪里不对→改菜谱"（反向+优化器），那部分推理完全没有。

## 3. 一句话归纳

> - **训练包 vs 推理包**：看有没有反向/优化器/训练数据。推理包只够"拿来用"，**不含训练并行策略**。
> - **训练并行 vs 推理并行**：训练是 `DP×PP×CP×TP(+EP)` 全家桶（要扛 16B/param + 梯度同步）；推理只保留**前向那半**，以 **TP** 为主（大模型/MoE 加 EP/DP），**无梯度 all-reduce**。
> - 从一个推理包能提取的是**模型结构**（第二部分）和**推理 TP 配置**，提取不到**训练并行策略**——后者得看技术报告或训练框架（MindSpeed/Megatron）的配置。
