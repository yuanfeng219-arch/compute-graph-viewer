Section 01 · 硬件原生哲学

"Hardware-Native"：从硬件真相反推系统软件

hw-native-sys 不是从 SDK 接口往下做设计，而是从 Ascend 芯片的物理 tile 真相往上做设计。本节回答两个问题：这种思想从哪里来？最终的层级架构是怎么一步步反推出来的？

一、思想来源：MLsys 编译器社区的十年路径，遇上 NPU 物理现实

"Hardware-Native" 不是一个新概念——它在 AI 编译器社区里已经被验证、被反复重写。hw-native-sys 把社区累积的方法论带到 NPU 生态，但拒绝继续套用 GPU 中心的抽象。下面三条路径，是这个哲学的三条直接前驱：

① 编译器路径

代表：Apache TVM、MLIR、IREE。
贡献：用统一多层 IR 把"模型"和"硬件"接起来；自动调度、张量化、图改写。

→ 教会社区"硬件感知 IR"应该长什么样。

unified IRauto-schedule

② DSL 路径

代表：Triton、TileLang、Halide。
贡献：把"按线程编程"提升成"按 tile / block 数据流编程"，让算子作者直接表达切分。

→ 验证了 tile 抽象在 DNN 算子上是高生产率的。

tile DSLblock ptr

③ 硬件现实：NPU ≠ GPU

代表：Ascend AIC（Cube）+ AIV（Vector）+ AICPU 三类核心。
关键差异：它的最小单元不是 SIMT 线程，而是一组矩阵 / 向量 / 控制三种异构核协作。

→ "warp / SM" 心智在这里水土不服。

tile-first非 SIMT
组织来源：hw-native-sys 的主要贡献者带着 Apache TVM、MLIR 等编译器社区的工作经验进入 NPU 生态——他们已经知道"统一 IR + 张量化 + 自动调度"能走多远，也知道这条路在 GPU 中心抽象下会卡在哪里。把"硬件感知 IR"的方法论搬到 Ascend，但拒绝继续用 SIMT 抽象做表层包装——这就是 "Hardware-Native" 哲学的字面来源。

二、架构反推：从硬件真相到 7 层栈

如果 tile 是 Ascend 的硬件真相，那么所有上层抽象都必须以 tile 为枢纽。下面 6 步是从硬件向上的反向构造——每一条推论都是上一条的逻辑后果。这解释了为什么 hw-native-sys 长成现在这个 6 仓库 7 层的形状，而不是别的形状。

起点
Ascend 硬件真相：tile 是天然单元，AIC + AIV 异构协作
推论 1
tile 必须是 IR 一等公民 → 多层 IR：Tensor → Tile → Block → Execution
推论 2
需虚拟 ISA 锁住跨代差异 → PTO（pto-isa）
推论 3
DSL 必须能显式声明层级与角色 → @pl.function（pypto）
推论 4

终点

这套架构不是"集成多种已有思想"的拼装，而是"承认硬件真相后必然如此"。每个仓库都是某一推论的物质化，缺一条推论就缺一个仓库。
组织自我陈述："Our philosophy is 'Hardware-Native': we believe that to achieve peak performance and efficiency, system software—from compilers to kernels—must be designed with a deep awareness of the underlying hardware architecture. We aim to bridge the gap between rapidly evolving AI models and the diverse landscape of emerging NPU/GPU accelerators."

Section 03 · PTO 虚拟 ISA

PTO：tile 维度的 PTX，把代际差异挡在硬件背后

PTO（Parallel Tile Operation）是 Ascend CANN 定义的一套 tile 级虚拟指令集。它的角色和 NVIDIA PTX 类似——做一层"虚拟 ISA"承担向上稳定、向下适配，让算子作者写一次就能跨代，把寄存器分配、pipe 映射这些代际差异留给后端。

原文定位："Parallel Tile Operation (PTO) is a virtual ISA for tile-oriented programming defined by Ascend CANN... uses a higher-level tile programming model to bridge implementation differences across generations."

计算指令

matmul、vector op、reduce、softmax 等 tile 级别的算术单元，覆盖 AIC（Cube）和 AIV（Vector）两类核心。

tensor.cmptensor.cmpsL0 tiling

搬运指令

GM ↔ UB / SRAM 之间的 tile 搬运，含 tile_shape 物理布局提示，把短跨步 TLOAD 合并成一次连续 burst。

A2/A3 burst 512BA5 burst 128B

通信指令

跨核 ring：TPUSH / TPOP / TFREE 六指令；跨节点 SDMA / RoCE / URMA / CCU 通过 complete_in_future 抽象成异步 token。

32 cross-core flags

三程序模型：一次部署 = 三个产物

每个 PTO kernel 在 Ascend 上展开为三个协作的程序产物，分别跑在不同硬件单元上，通过定义的 API 互相调用。这是 simpler 运行时调度的最小单位。

Section 04 · PyPTO DSL

PyPTO：把分层调度写成 Python 装饰器

pypto 是 hw-native-sys 的 Python 入口。它的设计假设是：tile 是一等公民——shape、mask、pipeline、scope 都应在 IR 中显式建模，而不是被隐藏在 runtime cleverness 后面。开发者用 @pl.function 声明一个函数运行在哪个层级、扮演什么角色，编译器据此选择降级路径。

层级与角色（@pl.function）

函数同时声明 层级（CHIP / CORE_GROUP / CORE…）和 角色（ORCHESTRATOR / SCHEDULER / WORKER），IR 据此分配资源、染色（WHITE 控制 / GREEN AIV / RED AIC）。

orchestrator on chip
@pl.function(level=pl.Level.CHIP, role=pl.Role.ORCHESTRATOR) def matmul_kernel(A, B, C): with pl.at(level=pl.Level.CORE_GROUP, optimization=pl.chunked_loop_optimizer): ...

四种循环语义

pl.range 顺序、pl.parallel 数据并行、pl.pipeline 流水化、pl.spmd SPMD。同一份代码用 MPMD（不同核执行不同程序体）方式实际部署，IR 自动按角色拆分。

range parallel pipeline spmd MPMD 调度

多层 IR：自顶向下的下沉路径

① Tensor Graph

算法工程师可见层。整张计算图、shape、dtype、devices——还没有 tile 概念。

② Tile Graph

性能工程师可见层。tensor 被切成 tile，tile_shape 物理布局开始起作用。

③ Block Graph

多核协作层。tile 被分配给 AIC / AIV，TPUSH/TPOP 通信、ring buffer 与 scope 出现。

④ Execution Graph

运行时直接消费的 DAG：tensormap、ringbuffer 状态机、async hardware 完成事件全部展开。

原文："Through a multi-level intermediate representation (IR) system, it compiles AI model applications built via APIs from high-level Tensor graphs step by step into hardware instructions."
tile_shape 物理布局：把 16 次短 burst 合并成 1 次
Without tile_shape: 16 × 16B 短跨步 burst ← 远低于 A2/A3 偏好的 512B With tile_shape=(TM, TN): 1 × 256B 连续 burst ← 接近 burst length 上限 T = pl.gm_tensor(shape=(M, N), dtype=pl.fp8, tile_shape=(TM, TN))
Section 05 · Simpler 运行时

Orchestrator-Scheduler-Worker：一套协议从 L2 递归到 L7

simpler 是 hw-native-sys 的运行时层，承担把 Execution Graph 真正放到硅上跑。它的关键设计抉择是：每一个层级（chip、host、cluster、global）都使用 同一套 Orchestrator-Scheduler-Worker 三元组。这意味着单核内的微调度和万卡级的全局协调用的是同一个心智模型，只是规模不同。

Orchestrator

构图者。把 Execution Graph 切成本层级可消费的子任务 DAG，向下提交。

DAG 提交

Scheduler

调度者。维护 ready queue、ring buffer、依赖关系、refcount，决定谁可以开始。

ready / refcount

Worker

执行者。在被分配的物理资源上运行任务体；跑完通知 scheduler，或挂在 complete_in_future 上等异步硬件。

run / wait async

complete_in_future：函数返回 ≠ 任务完成

硬件单元（SDMA、RoCE、URMA、CCU）天然异步：发射后核心就空了，但实际数据搬运还没结束。simpler 把这一类硬件统一抽象成 "延迟完成任务"——worker 函数 return 后核心立即可被 scheduler 复用，但这个任务在 SDMA / RoCE / URMA / CCU 的 CQ entry 或 notification counter 触发前不会标记 COMPLETE。下游任务通过同一个 token 等待。

关键 invariant："A complete_in_future task's function return releases the core but does not complete the task." 这一条让一份调度逻辑同时覆盖单核内的 pipe 重叠和万卡间的 collective 同步——异步硬件不再是特例。
异步硬件统一抽象（伪代码）
tag = pto2_send_request_entry(SDMA, rq_id, &desc) pto2_save_expected_completion(SDMA, cq_id, tag, A) # Worker A 已 return，core 释放给 Scheduler # 下游 Worker B 仍然必须 wait 在 tag 上 worker_b: wait_completion(tag) consume(buffer) // 数据真就绪了

Section 06 · Lingqu / 灵渠层级

L7 → L0：把全球部署逐级缩放到一个 Core-Group

simpler 与 pypto 共享一套 Lingqu（灵渠）8 级机器层级，把硬件世界从跨数据中心一路缩放到一个 AIV/AIC/AICPU 组合。每一级都是一个 Orchestrator-Scheduler-Worker 实例；上一级的 Worker 会成为下一级的 Orchestrator。下面先用一张总览图建立递归关系，再用详表和网络边界图展开 8 个层级。

三类网络：从 Scale Up 到 DCN

8 个层级跨越的距离不一样，对应的网络也不一样。理解 Lingqu 必须先分清这三种带宽量级——它们决定哪一种并行策略可以在哪一级用。

Scale Up（SU）

Pod 内紧耦合总线，例如统一总线 UB。带宽最高、延迟最低，可以做细粒度通信。

3.2 TbpsL4 内

→ 适合 EP（Expert Parallel）、TP（Tensor Parallel）。

Scale Out（SO）

Cluster 内 Pod 之间的 CLOS 拓扑。带宽低于 SU，但仍是高速光互联。

16,384 端口L5–L6

→ 适合 DP、PP（Pipeline Parallel）跨 pod。

DCN

Data Center Network。跨集群、跨数据中心的长距离网络，延迟显著更高。

跨 DCL7

→ 仅适合异步复制、checkpoint sync、灾备。

放大 L0：IR 染色其实是 L0 内视图

前面 8 张图描述的是层与层之间的递归调度。但当任务最终下发到一个 Core-Group（L0）后，里面还要再分发一次：哪些指令是控制流（在 AICPU 上）、哪些走向量单元（AIV）、哪些走 Cube 矩阵（AIC）。pypto IR 用三种颜色把这种"L0 内部的再分发"直接画在源码视图上——所以 IR 染色不是与 L0-L7 并列的概念，而是在显微镜里看 L0 时才浮现的一层。

Section 07 · Tile 数据流

TPUSH / TPOP / sharded_tensor：把张量通信变成显式语义

硬件原生的代价是通信不能再是隐式 collective——它必须出现在 IR 中。hw-native-sys 把通信拆成三层：核间用 6 指令 ring 协议，多节点用 sharded_tensor，整个图用 scope-depth 多层 ring 控制峰值显存。三层都有显式 API 和 IR 表示，没有隐藏路径。

① 核间：TPUSH / TPOP / TFREE 6 指令

AIC ↔ AIV 跨核环（每集群 32 flag）
six instructions
tpush_to_aiv tpop_from_aiv tfree_from_aiv tpush_to_aic tpop_from_aic tfree_from_aic flags: P2C「data ready」 + C2P「space free」 共 32 个 / cluster slots: SLOT_NUM = 8（单向） 或 4 + 4（双向） modes: 1:1 / 1:2 ；TileSplitAxis ∈ {NO_SPLIT, UP_DOWN, LEFT_RIGHT} A5 优化: consumer-SRAM aliasing → 零拷贝

② 多节点：sharded_tensor

逻辑视图

看起来是一个全局 tensor，shape 与单卡完全相同。算法代码无须感知物理分布。

原文："Logically, a sharded_tensor represents a single global tensor."

物理布局

实际存储 = shard_shape × rank_shape，分散在 rank_num 个 rank 上。layout 操作必须额外携带 rank_index，构造时通过阻塞式 all-to-all barrier。

实验 APIubmem 优先

③ 多层 ring stack + 显式 pl.free

单层全局 ring 在长上下文场景下会撑爆显存。pypto 用 scope-depth 索引的多层 ring替代它：每个 scope 进入时新增一层 ring，离开时自然释放。开发者还可以用 pl.free(tensor) 显式提前归还某个 tensor 的引用，强制 scope-exit refcount 提前生效，进一步压低 peak。

scope 化 ring stack 示意
scope_depth = 0 ← 全局 ring（仅持久 tensor） scope_depth = 1 ← prefill 阶段 ring scope_depth = 2 ← 单层 attention ring tensor t1, t2, t3 pl.free(t1) // 提前归还，refcount 立刻 -1 scope exit → 释放整层 ring
关键设计：核间通信、多节点 sharding、生命周期都是 显式 IR 节点而不是 runtime cleverness。这让 profiler 能直接告诉你"这条 TPUSH 卡在哪个 flag 上"和"这一层 ring 把峰值推高了多少 GiB"。

Section 09 · 核心结论

结论：硬件原生的差异化不是兼容，是 tile 抽象贯通

hw-native-sys 的差异化不在于"比 CUDA 更像 CUDA"，而是把 tile 维度从 DSL、IR、虚拟 ISA、运行时一路打穿，再把同一套 Orchestrator-Scheduler-Worker 协议从 L0 递归到 L7。它的代价是开发者需要理解 tile、scope、async 等一手概念；回报是源码-IR-硬件之间的解释链不依赖任何闭源工具就能贯通。

01
把多层 IR 工具化提供 ptobc dump、IR 可视化、scope-depth ring 占用图，让开发者直接看到自己代码降到了哪一层。
