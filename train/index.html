0 · 学习地图

先记住三句话

大模型不是神秘黑盒，它是一套“把文字切成 token，再用大量矩阵计算预测下一个 token”的系统。 做饭类比：文字先被切成小食材，后厨流水线每一步都加工一点，最后判断下一道该上什么。

训练是在更新参数，推理是在使用已有参数。训练像反复试菜并修改菜谱，推理像按已经定稿的菜谱正式出餐。 做饭类比：训练时厨师会改配方；推理时厨师不改配方，只按配方做菜。

真正的大模型工程不只会调用 model.generate()，还要理解 tokenizer、权重格式、数据预处理、并行策略、KV Cache、量化和部署配置。 做饭类比：会按菜谱做一道菜还不够，开中央厨房还要懂采购、切配、分工、库存、设备和出餐速度。

本页依据： 本地 /Users/yin/gitcode 内的 MindSpeed-LLM、cann-recipes-infer、Qwen7B source 资料，以及 patterns/model-graphviz 中的 Qwen7B / DeepSeek V3.2 架构图。

类比只是入口，真正写代码和读文档时要回到准确术语。下面的表可以当作第一天的速查表。 做饭类比：先用厨房理解概念，但进真实厨房后还是要会说灶台、刀工、火候和出餐节奏。

2 · 模型基础对象

结构、权重、数据、框架的区别

初学大模型时，最容易把“模型”这个词混成一团。模型结构、模型权重、训练数据和运行框架分别解决不同问题。 做饭类比：不要把厨房平面图、菜谱克数、食材和厨具混为一谈；它们都重要，但职责不同。

对象准确含义第一天要会看什么做饭类比
模型结构层数、hidden size、attention heads、MLP/MoE 等模块连接方式。config.json、架构图、modeling_*.py厨房布局和工位顺序
模型权重训练得到的参数数值，决定模型具体能力和输出风格。.safetensors、权重索引、checkpoint每道菜的具体配方克数
数据训练或微调用的文本、指令对话、领域语料，以及预处理后的 token 文件。JSON/JSONL、prompt 模板、.bin/.idx食材和已经切好的备菜
运行框架负责加载权重、调度设备、执行训练或推理的工具链。MindSpeed-LLM、Transformers、cann-recipes-infer、YAML 配置炉灶、锅具、排班系统和出餐流程

Qwen7B 适合先建立结构、权重、数据、框架之间的对应关系：它的结构相对标准，文件也更容易对应。DeepSeek V3.2 则能看到这些概念在 MoE、并行和长上下文工程里怎样放大。 做饭类比：先在标准厨房分清图纸、配方、食材和工具，再去看大型中央厨房才不会迷路。

3 · 文本怎么进入模型

Tokenizer、词表与 token id

模型不能直接吃中文、英文或代码字符串。第一步必须由 tokenizer 把文本切成 token，再把 token 映射成整数 id。 做饭类比：客人说的是自然语言菜单，厨房先要把它拆成标准菜品编号和备菜单。

词表是 tokenizer 能识别的 token 集合。Qwen7B 使用约 15 万级大词表，本地 README 写 151851，本地 config.json 的 vocab_size 是 151936，常见原因是工程配置里会包含补齐或特殊 token。 做饭类比：词表像厨房编码表，真实系统里会给特殊菜品、空位和内部标记也留编号。

tokenizer 和权重必须匹配。换 tokenizer 相当于把同一串数字解释成另一套餐品编号，模型会读不懂。 做饭类比：同样的 151643，在 A 厨房可能是结束标记，在 B 厨房可能是别的菜，不能混用。

4 · 模型结构

模型的“层”是什么？

层是一组按固定顺序执行的计算模块。输入文本先被 tokenizer 变成 token id，再进入 Embedding 层变成向量；随后向量依次经过很多个 Transformer Block；最后进入 LM Head，变成“下一个 token 是谁”的概率。 做饭类比：先把菜单切成编号，再把编号变成食材，经过很多加工工位，最后从出菜口决定下一道菜。

平时说 Qwen7B 有 num_hidden_layers: 32，通常指 32 个 Transformer Block。这个数字一般不把最前面的 Embedding 和最后的 LM Head 算进去。 做饭类比：32 层不是 32 本菜谱，而是同一道菜连续经过 32 个加工工位。

一个 Transformer Block 不是单个矩阵，而是 Norm、Attention、残差连接、MLP/FFN 等小模块的组合。Attention 让 token 看上下文重点，MLP 对每个 token 的表示做进一步加工，Norm 稳定数值，残差连接保留原信息并叠加新变化。 做饭类比：一个工位不是只有一口锅，还包含称重、看订单、翻炒、保留原味和校准火候。

模块做什么类比
Embedding把 token id 变成向量把菜名编号换成真实食材
Attention让当前 token 关注上下文重点看整张订单，找关键备注
MLP / FFN对 token 表示做非线性加工调味、翻炒、让味道更复杂
Norm稳定数值范围校准火候和称重
Residual把输入绕路加回输出保留原味，再叠加新调味
LM Head映射到词表概率根据半成品判断下一道该出什么

5 · 训练在训练什么

预训练、SFT、LoRA 与 checkpoint

预训练最常见目标是 next token prediction：给模型一段前文，让它预测下一个 token。训练时计算预测分布和真实目标之间的 loss，再通过反向传播更新参数。 做饭类比：模型先猜下一步怎么做，猜错就调整菜谱里的比例，反复很多次后厨艺变稳。

指令微调 SFT 让模型学会“看到指令后按任务回答”。MindSpeed-LLM 文档里覆盖单样本微调、多样本 pack 微调和多轮对话微调。 做饭类比：预训练是学通用厨艺，SFT 是学会看客人点单并按指定格式出餐。

LoRA 会冻结大部分原始权重，只训练少量低秩矩阵。它不是从零训练新模型，而是在已有模型旁边加一组可学习的小模块。 做饭类比：不重写整本菜谱，只在关键页贴便签，便签很小但能改变口味和出餐格式。

6 · 推理在推什么

Prefill、Decode、KV Cache 与生成参数

推理不会更新权重。它的目标是加载已有权重，根据 prompt 生成输出。整个过程通常分为 Prefill 和 Decode 两个阶段。 做饭类比：推理时厨师不改菜谱，只看订单并正式做菜。

Prefill 阶段一次性处理完整 prompt，建立上下文状态和 KV Cache。Decode 阶段基于已有上下文，每一步生成一个或少量新 token。 做饭类比：Prefill 是先完整读菜单和备注，Decode 是一道一道继续出菜。

KV Cache 缓存 Attention 中的 Key/Value，避免每生成一个 token 都重新计算所有历史。长上下文、多并发推理时，KV Cache 的显存管理会直接影响速度和可服务请求数量。 做饭类比：把已经处理好的中间材料放保温盒，下一步直接复用，不用每次从洗菜开始。

参数 / 指标含义类比
max_new_tokens最多生成多少新 token最多出几道菜
top_k只在概率最高的 k 个候选里选只从前 k 个热门菜里选
top_p选累计概率达到 p 的候选集合只从足够可能的菜里选
temperature控制随机性温度高更敢创新，低更保守
TTFTTime To First Token，首 token 延迟第一道菜多久上桌
TPOTTime Per Output Token，每个 token 耗时后续每道菜的出餐间隔

7 · 案例一：Dense Transformer

Qwen7B：适合第一阶段学习的标准样例

Qwen7B 是 70 亿参数规模的 Dense Transformer。它适合用来学习模型目录、配置文件、tokenizer、基础推理和 Dense Transformer 的主干结构。 做饭类比：Qwen7B 像标准厨房，所有订单基本都走同一条完整流水线，结构比大型中央厨房更容易入门。

Qwen7B README 和配置里能看到 RoPE、SwiGLU、RMSNorm、Flash Attention 等现代 LLM 常见组件。第一天不用把数学细节推完，但要知道它们分别属于位置编码、MLP 激活、归一化和高效注意力计算。 做饭类比：先知道这些是定位菜品、调味方式、校准火候和高效锅具，后面再研究每个工具的细节。

8 · 案例二：MoE + DSA

DeepSeek V3.2：适合学习大规模工程复杂度

本页将 dpsk 3.2 对应为本地 DeepSeek-V3.2-Exp 资料。它是 671B 级别 MoE 模型，适合学习 MoE、MLA、DSA、MTP、TP、PP、EP、CP、量化和长上下文推理。 做饭类比：DeepSeek V3.2 像大型中央厨房，不只是一条流水线，而是有分菜员、专家厨师、长菜单处理和多节点协作。

模块DeepSeek V3.2 里要看的点

9 · 权重与格式转换

为什么同一个模型还要转权重

模型结构回答“厨房长什么样”，模型权重回答“每个参数具体是多少”。同样结构配上不同权重，能力和输出风格会完全不同。 做饭类比：厨房平面图相同，不代表菜谱配方相同；调料克数不同，味道就不同。

HuggingFace 权重是通用生态常见格式，Megatron-Mcore 权重适合 MindSpeed-LLM 分布式训练，推理框架权重可能还会按并行、量化或设备要求重新切分和重排。 做饭类比：公开菜谱、大厨房内部菜谱、压缩版出餐菜谱，内容相关但格式不同。

权重转换最容易出错的是并行参数不一致、tokenizer 路径不匹配、base/chat 模型混用、LoRA 权重没有和 base 权重一起正确加载。 做饭类比：菜谱、菜品编号、厨房设备和便签必须对应，错一个就可能出错菜。

10 · 数据预处理

为什么要生成 .bin 和 .idx

训练时直接反复读取 JSON、CSV 或 Parquet 并实时 tokenize 会慢。MindSpeed-LLM 的流程会提前把原始数据 tokenize，生成 .bin 保存 token 内容，生成 .idx 保存索引和样本边界。 做饭类比：营业前先洗菜、切菜、分盒编号，出餐时直接拿盒子，不要每单都重新切。

预训练数据常见字段是 text；指令微调会把 instruction、input、response、system、history 等字段映射成模型需要的 prompt 和 labels。 做饭类比：预训练像读大量菜谱文本，指令微调像一张张“客人点单和标准答复”的训练单。

DATA_PATH 通常填输出前缀，而不是随便某个 .bin 文件。这个细节是很多初学者第一次跑训练脚本时会踩的坑。 做饭类比：交给厨房的是“这一批菜盒的编号前缀”，不是只拿其中一个盒子。

11 · 大模型为什么要并行

DP、TP、PP、CP、EP 怎么分工

大模型太大，单卡通常放不下或算不快，所以训练和推理会把计算、参数、层、数据、上下文或专家拆到多张卡、多节点上协作。 做饭类比：一桌万人宴席不可能一个人完成，需要采购、切配、热厨、冷菜、甜品和传菜一起协作。

并行方式拆什么做饭类比
DP多份模型副本处理不同 batch，再同步梯度多个厨师做不同订单后同步经验
TP把大矩阵乘法按维度切到多卡一口大锅分成几口锅同时炒
PP把不同模型层放到不同设备前菜、主菜、甜点按流水线传递
CP长序列上下文按 token 维度分片超长菜单分给多人读
EPMoE 的 expert 分布到不同设备不同菜系专家分布在不同厨房
ZeRO / FSDP参数、梯度、优化器状态分片保存菜谱、笔记和库存分散保管

DeepSeek V3.2 这类 MoE 模型会组合使用多种并行方式。比如 TP 拆矩阵，PP 拆层，EP 拆专家，CP 面向长上下文；这些都属于协同训练或协同推理。 做饭类比：同一道菜可以跨多个工位完成，但所有工位仍然围绕同一张订单工作。

12 · 拆分训练

模型可以拆分训练吗？

工程上可以拆分。TP、PP、DP、EP、CP、ZeRO/FSDP 都是在把一个大模型拆到多设备上共同训练，它们仍然处于同一个模型、同一个 loss、同一次反向传播里。 做饭类比：中央厨房把一道菜分给多个工位协作，但最后仍然按同一张订单和同一套质检标准出菜。

语义上不能随便拆。不能简单地把前 10 层和后 10 层各自单独训练，再拼起来期待效果稳定。后面的层依赖前面的表示，前面的层也要根据最终 loss 反向传播回来的梯度调整。 做饭类比：不能让一个人只练切菜，另一个人只练调味，最后随机拼班就保证整道菜好吃。

合理的“局部训练”包括继续预训练、全参微调、冻结部分层、LoRA/Adapter、QLoRA、蒸馏、MoE 专家和路由优化。它们不是随意拆层拼接，而是在明确训练目标和约束下进行。 做饭类比：可以只训练甜品组或给原菜谱贴便签，但仍要用最终菜品质量来验收。

方式训练什么适合情况
全量预训练从头训练全部参数海量数据和大规模集群，成本最高
继续预训练已有模型上继续学习新语料行业语料、代码、法律、医学等领域增强
全参微调让全部参数参与微调资源较多、任务变化较大
冻结部分层固定一部分参数，只训练部分层低成本实验，但能力上限有限
LoRA / Adapter冻结 base，只训练小模块初学者和业务微调最常见
蒸馏用大模型输出教小模型想部署更小、更快模型

14 · 第一天到第五阶段

初学者学习路线

第 1 阶段只学会看懂模型目录：README、config、tokenizer、generation_config、权重索引分别做什么。目标是能解释 Qwen7B 的层数、隐藏维度、头数、词表和上下文长度。 做饭类比：先看厨房地图、菜谱封面、食材编码表和出餐规则，不急着开火。

第 2 阶段学一次基础推理：理解 AutoTokenizer、AutoModelForCausalLM、generate()、max_new_tokens、top_p、temperature。 做饭类比：先按已有菜谱做一道小菜，观察第一道多久上、后面出餐是否稳定。

第 3 阶段学训练流程，但先不追求跑 671B：环境准备、权重转换、数据预处理、预训练、SFT、LoRA、checkpoint 保存和最小推理验证。 做饭类比：先练小规模试菜流程，不要第一天就承接万人宴席。

第 4 阶段学并行和 DeepSeek V3.2 脚本：看懂 TP、PP、DP、CP、EP、num-experts、router topk、DSA、Lightning Indexer、Sparse Flash Attention。 做饭类比：开始研究中央厨房怎样把一道复杂订单分给多个工位和专家组。

第 5 阶段学推理部署与性能优化：区分 BF16、FP8、INT8、W8A8C8，理解 TTFT、TPOT、吞吐、KV Cache、Paged Attention 和 YAML 配置。 做饭类比：不只要做得对，还要上菜快、占用少、多人点餐时不乱。

15 · 读脚本时应该抓什么重点

先看参数属于哪一类，再看它影响哪一步

初学者读训练或推理脚本时，不要从第一行到最后一行硬啃。先把参数归类：路径、模型结构、并行、精度/量化、数据、checkpoint、启动方式和功能开关。 做饭类比：看中央厨房排班表时，先分清哪些是食材地址，哪些是工位数量，哪些是锅具规格，哪些是出餐流程。

参数类别常见名字应该问的问题
路径MODEL_PATH、LOAD_PATH、SAVE_PATH权重、tokenizer、输出 checkpoint 是否指向同一套模型？
模型结构num-layers、hidden-size、num-attention-heads脚本参数是否和 config、架构图一致？
并行tensor-model-parallel-size、pipeline-model-parallel-size、expert-model-parallel-size并行切分是否和权重格式、卡数、节点数一致？
数据DATA_PATH、seq-length、micro-batch-size数据是否已经预处理，batch 和序列长度是否会 OOM？
精度/量化bf16、fp8、w8a8、quant加载路径、YAML 配置和硬件能力是否匹配？
功能开关enable-dsa-indexer、use-sparse-flash-attn打开的优化是否有对应模型结构、依赖和数据路径？

读脚本的目标不是背参数，而是能解释“这个参数影响哪张图里的哪一块”。例如 TP 影响矩阵并行，EP 影响 MoE experts，CP 影响长上下文切分，DSA 影响稀疏注意力路径。 做饭类比：学习者不需要背整张排班表，但要知道每个岗位动了会影响哪道菜、哪口锅、哪组厨师。

16 · 用做饭类比串起完整流程

从原始文本到推理服务是一条流水线

大模型工程不是单点知识，而是一条连续链路：准备原始文本，tokenizer 编码，数据预处理，加载或转换权重，训练或微调，保存 checkpoint，再加载到推理框架执行 Prefill/Decode。 做饭类比：采购食材、切配、按菜谱试菜、保存定稿菜谱、正式营业出餐，是同一条厨房流程。

如果流程出错，先按链路定位：tokenizer 是否匹配，数据前缀是否正确，权重是否完整，转换并行参数是否一致，推理配置是否对应量化和设备。 做饭类比：菜不对味时，先查食材编号、备菜盒、菜谱版本、工位分工和锅具设置，不要只盯最后出餐口。

17 · Checklist

训练和推理前先检查这些

训练前

  • CANN / torch_npu / PyTorch / MindSpeed-LLM 版本匹配。
  • 模型权重完整，tokenizer 路径与模型匹配。
  • HF 到 Megatron-Mcore 转换参数和训练脚本 TP / PP / EP 一致。
  • 数据已预处理成 .bin / .idx，DATA_PATH 填前缀。
  • micro-batch-size、global-batch-size、seq-length 不会导致 OOM。

推理前

  • 模型权重、tokenizer、prompt template 匹配。
  • 并行参数与权重切分一致。
  • max_new_tokens 不要过大，长上下文要估算 KV Cache。
  • 量化权重路径与 YAML 中 dtype / quant mode 对应。
  • 多节点推理时 IP、rank、world size 一致。

18 · 常见误区

第一天最容易误解的地方

7B 不等于普通电脑一定轻松跑。显存、精度、量化、上下文长度和推理框架都会影响能不能跑、跑得多快。 做饭类比：小餐厅也需要炉灶和食材，不是菜名短就一定容易做。

下载权重不等于能训练。还需要环境、tokenizer、数据预处理、权重格式转换、并行参数和足够硬件。 做饭类比：有菜谱不等于能开店，还要厨房设备、备菜和员工排班。

LoRA 不等于完整训练。它是参数高效微调，适合学习任务格式、领域风格或小范围适配，但不能低成本获得全新基础能力。 做饭类比：贴便签能微调菜品风味，但不能把完全不会做饭的人变成大厨。

量化不只是“变快”。它通常能省显存、提吞吐、降时延，但可能影响效果，必须结合评测和业务样例验证。 做饭类比：把菜谱压缩成速记版会更快，但不能漏掉关键步骤。

DeepSeek V3.2 脚本不能随便照抄跑。脚本是大规模集群参考，需要按硬件、节点数、网络、权重路径、数据路径、并行策略和版本调整。 做饭类比：中央厨房排班表不能直接拿到家用厨房执行。

模型可以工程拆分训练，但不能把每层各自独立训练后随便拼起来。TP、PP、DP、EP、CP 是协同训练同一个模型，不是把模型语义切碎。 做饭类比：流水线可以分工，但所有工位仍然服务同一道菜和同一个质检标准。

19 · 结论

先用 Qwen 建地基，再用 DeepSeek 学工程复杂度

初学者第一阶段应该用 Qwen7B 建立 Dense Transformer、tokenizer、config、权重、推理参数和基础训练流程的认知。它仍然不是“随便本机全量训练”的对象，但结构更适合作为第一张地图。 做饭类比：先在标准厨房把切菜、调味、出餐练顺，再去看中央厨房。

DeepSeek V3.2 适合第二阶段学习 MoE、MLA、DSA、MTP、并行、量化和长上下文推理。它的价值不是让初学者第一天直接全量训练，而是帮助学习者理解真实大模型工程为什么复杂。 做饭类比：万人宴席不适合第一天上手承包，但它的厨房图纸很适合用来理解大规模协作。

判断自己是否真正入门，可以看三件事：能解释模型的层是什么；能说清训练、微调、推理分别更新什么或不更新什么；能读懂脚本里路径、数据、并行、权重和量化参数各自影响哪里。 做饭类比：能说清厨房布局、菜谱怎么改、正式出餐怎么跑，就已经跨过第一道门槛。
