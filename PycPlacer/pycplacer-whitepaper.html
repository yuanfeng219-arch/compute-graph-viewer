Scope · 在 PTO 总地图中的位置

H-Anchor 是算法原型层：验证分层锚点能否服务 placement

本白皮书讨论的是一个 placement 研究原型，不是完整商业 EDA，也不是运行时算子工具。它在总体 scope 中承接 HNSW 的分层思想，落到“weighted graph / cells → 初步二维布局坐标”的算法验证。

解决什么问题

传统 force-directed placement 在大规模图上容易受 O(N²) 计算、全量重排和确定性不足限制。H-Anchor 用分层 anchor、空间抑制和局部传播降低搜索空间。

Background · 背景

VLSI 布局：规模与速度的两难

现代芯片设计动辄包含数万个逻辑单元，布局质量直接决定芯片延迟、功耗与良率。而传统算法在规模扩大时遭遇严重的性能与工程瓶颈。

核心矛盾：布局优化本质是 N 体问题——排斥力计算复杂度为 O(N²)，3,300 个单元即意味着超过 1,000 万次成对计算。纯 Python 实现耗时 42 秒，每次参数调整都需等待，严重阻碍设计探索。

速度瓶颈

大规模电路（3000+ 单元）布局耗时以十秒级计，每次参数调整都需等待完整重新布局，无法满足交互式设计探索需求。

42s / runPure Python

确定性缺失

并行化通常引入浮点累加顺序不一致，导致多次运行产生不同结果，破坏可复现的工程工作流与 CI/CD 管线。

结果不一致调试困难

增量支持不足

修改少量单元后，传统工具需重新执行完整布局流程。即使只调整 5 个单元，也没有「只重新优化局部」的路径。

Core Innovation · 分层锚点架构

分层选择：PageRank × 空间抑制

H-Anchor 的命名来源于其核心创新：用受 HNSW 图启发的分层锚点结构，将 N 体布局问题分解为「由稀到密、自上而下」的层次优化问题。

H-Anchor · 分层结构主架构图
蓝色大圆 = 顶层全局锚点（100 个），紫色中圆 = 中层区域锚点，灰色小点 = 底层全量单元；蓝色虚线箭头表示自上而下的位置投影方向。橙色圆圈示意锚点选择时的空间抑制范围，防止锚点聚集。

PageRank 中心性评分

节点重要性 = 度数中心性 × 0.4 + PageRank × 0.6，迭代 20 次收敛，阻尼系数 0.85。中心性高的节点优先成为锚点，保证覆盖整个连接图的关键枢纽。

scoring 公式
score = α·degree + β·pagerank α = 0.4, β = 0.6, damping = 0.85 iterations = 20 ← 20 次迭代内收敛

空间抑制选择

按得分降序排列所有节点，逐个选为锚点，同时将其邻域标记为「已覆盖」并跳过，防止高分节点全部聚集在图的同一稠密区域。

build_layers 伪码
sort nodes by score desc for node in sorted: if node not covered: select as anchor mark neighborhood as covered until len(anchors) == top_layer_size
关键公式 — PageRank 迭代：
pr_new[i] = (1 − damping) / N + damping × Σ( pr[neighbor] / degree[neighbor] )
damping = 0.85，N = 节点总数，迭代 20 次后与度数中心性以 α=0.4 / β=0.6 加权合并为最终锚点评分。

Incremental · 增量更新

BFS 传播：局部改动，全局稳定

增量更新是 H-Anchor 区别于同类工具的关键工程特性。当少量单元位置变化时，通过 BFS 扩展影响区域并施加衰减，在 ~0.02 秒内完成局部优化，HPWL 变化仅 ±5%。

Findings · 结论与建议

三角突破：速度 · 确定性 · 增量

H-Anchor 通过分层锚点 + 空间抑制 + BFS 衰减传播的组合，首次在单一开源实现中同时满足高速度、确定性和增量更新三个工程约束。

核心洞察：H-Anchor 通过三个机制各自击破三角：(1) 层次化结构将 O(N²) 排斥计算降至层内局部范围 → 速度；(2) schedule(static) 在并行中保持浮点顺序 → 确定性；(3) BFS 衰减将增量更新复杂度降至 O(radius² · k) → 增量支持。三角不再需要妥协。
