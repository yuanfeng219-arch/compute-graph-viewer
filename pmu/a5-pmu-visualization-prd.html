Product Requirements Document

A5 PMU 数据可视化 PRD

面向算子开发者与性能优化专家，设计一套嵌入现有泳道图的 A5 PMU 诊断工作台。产品目标不是展示 counter 列表，而是帮助用户快速判断异常集中位置、dominant counter、硬件路径与下一步采集方向。

1. 背景与问题

当前 PyPTO 通过 PROF_PMU_EVENT_TYPE 在一次运行中选择一个 PMU counter group 采集。A5 / dav_3510 底层最多支持 10 个 counter slot，但实际展示数量取决于所选 group 的有效事件数，0x0 占位不应暴露给用户。

问题 1：counter 可读性低

用户看到一组 counter 值后，很难直接判断哪条 lane、哪个 Wrap、哪个 task 最该优化。

问题 2：group 语义差异大

Group 2 适合 pipeline balance，但 group 6 更像 conflict / issue 诊断，不能用同一种图表强行表达。

问题 3：缺少定位闭环

异常 counter 需要继续跳到源码、CALL、Block Graph、硬件路径与 peer task，否则只能停留在数值观察。

产品判断：PMU group 不是“字段集合”，而是一组不同诊断镜头。切换 group 后，应切换到更符合该 group 语义的诊断工作台。

2. 目标与范围

2.1 产品目标

  1. 帮助用户在泳道图上快速识别异常 lane、Wrap 与 task。
  2. 帮助用户判断当前 PMU group 是否能解释 total_cycle。
  3. 帮助用户理解 dominant counter 对应的硬件路径和优化方向。
  4. 支持从异常 counter 一跳定位到源码、CALL、Block Graph、硬件路径。
  5. 为多次运行对比、专家解释、BufferID / URMA / CCU 直出展示预留入口。

2.2 首版范围

范围项 要求 优先级
架构切换 支持芯片型号切换入口。首版重点支持 A5 / dav_3510；旧架构后续复用框架接入。 MUST
Group 切换 支持 group 1 / 2 / 4 / 5 / 6 / 7 / 8。切换 group 后刷新泳道图 PMU 叠加层、诊断摘要与 detail panel。 MUST
A5 Mix 展示 打开 Mix 模式后，泳道内部展示分段与状态，支持同 Wrap 内 AIC / AIV0 / AIV1 对照。 MUST
一跳定位 异常 counter 可跳转到源码片段、CALL、Block Graph、硬件路径。 MUST
关键 PMU 事件标记 可在时间轴上标记关键事件，并支持点击查看触发原因、影响范围、定位入口。 SHOULD
多次运行对比 预留入口与数据模型，不作为首版完整交付。 FUTURE

2.3 暂不做

  • 不在首版对 dav_2201 与 A5 / dav_3510 做差异化教育解释。
  • 不把 0x0 占位 counter 展示给用户。
  • 不把所有 group 强行统一成同一种 stacked bar 或同一张字段表。
  • 不在缺少真实采集数据时伪装成 L1 真实结果；示例数据必须标注为演示或推导数据。

3. 用户与场景

算子开发者

关注某个算子、某段源码、某个 Wrap 为什么慢。需要从泳道图直接跳到源码、CALL、Block Graph，并获得下一步该采什么 group 的建议。

性能优化专家

关注瓶颈归因、跨 lane balance、硬件路径压力、cache / buffer / issue 类问题。需要快速定位全局最该看的异常，而不是逐 task 翻表。

核心用户故事

编号 用户故事 成功结果
US-01 作为性能优化专家，我想一眼看出 A5 上 AIC、AIV0、AIV1 谁占流水更多。 Group 2 下可看到 lane balance summary 与 Wrap 内 C/V 对照。
US-02 作为算子开发者，我想知道某个 long task 是否被当前 group 解释。 task detail 显示 total_cycle、可解释指标、gap / severity / efficiency 与结论。
US-03 作为性能优化专家，我想从异常 counter 追溯到硬件结构和相关 peer task。 detail panel 提供硬件路径、同 Wrap peer task、源码 / CALL / Block Graph 入口。
US-04 作为算子开发者，我想知道当前 group 解释不了时下一次该采哪个 group。 系统给出下一次采集建议，并说明建议依据。

4.2 页面布局

5. 功能需求

ID 需求 详细说明 优先级
FR-01 芯片型号切换 用户可切换芯片架构。切换后刷新泳道图、PMU group 列表、counter map、诊断解释和空状态。首版默认 A5 / dav_3510。 MUST
FR-02 PMU group 诊断工作台 每个 group 对应独立诊断工作台，不只是替换 counter 字段。工作台需定义默认摘要、counter 分类、task 表达、硬件路径和下一步建议。 MUST
FR-03 单次运行 group 约束提示 明确提示一次运行只能采一个 PMU group。若用户切到未采集 group，展示“无本 group 数据”状态和重新采集命令建议。 MUST
FR-04 有效 counter 展示 只展示当前 group 的有效 counter。0x0 占位不展示，不参与排序、计算和图例。 MUST
FR-05 Lane summary 按当前 group 语义聚合 AIC / AIV0 / AIV1：Group 2 显示 pipeline share / coverage，Group 8 显示 hit rate / miss / victim，Group 6 显示 conflict severity。 MUST
FR-06 Wrap-grouped swimlane 所有 task 放在对应 AIC / AIV lane 上，不为每个 task 单独创建 task row。Wrap 内应能横向比较 C / V 的开始、结束、等待和拖尾。 MUST
FR-07 Task 内部 PMU 表达 task 外框宽度对应 total_cycle，内部按当前 group 展示 composition、ratio 或 severity。Group 2 可展示 cycle-like counter composition 与 gap badge。 MUST
FR-08 A5 Mix 模式 开启后，展示泳道内部的分段与状态，包括等待、计算、访存、重叠或 uncovered 等状态。支持同 Wrap 内 C 与 V 对照。 MUST
FR-09 关键 PMU marker 系统基于阈值、排序、dominant counter 和 critical path 自动生成 marker。用户可点击 marker 查看触发规则、影响 task、相关 lane、硬件路径和定位入口。 SHOULD
FR-10 异常 task 排序 Immediate attention 区展示当前 group 下最该关注的 task / Wrap / lane。排序依据随 group 变化，不强行使用统一分数。 MUST
FR-11 一跳定位链路 点击源码、CALL、Block Graph、硬件路径时必须打开对应内容或聚焦对应面板，不允许只改变按钮状态。 MUST
FR-12 下一次采集建议 当前 group 不能解释异常时，提示建议采集的 group。例如 long AIC task + low cube busy 可建议转看 group 2 / 4 / 5 / 6。 SHOULD
FR-13 预留高级入口 预留多次运行对比、完整专家解释、BufferID / URMA / CCU 直出展示入口，但首版可为 disabled / coming soon 状态。 FUTURE

6. Group 诊断工作台

Group 工作台 有效项与默认表达 首要回答
1 Cube Arithmetic View cube_fp_instr_busy、cube_int_instr_busy；突出 AIC lane，展示 FP / INT 比例。 AIC / Cube 算术指令是否是真瓶颈？
2 Pipeline Balance View pmu_idc_aic_vec_busy_o、cube_instr_busy、scalar_instr_busy、mte1_instr_busy、mte2_instr_busy、mte3_instr_busy、icache_req、icache_miss、pmu_fix_instr_busy；展示 pipeline composition 与 gap。 AIC / AIV0 / AIV1 谁占流水，谁拖尾？
4 Memory Access Flow View main read/write、AIV ext rd/wr UB、vec rd/wr UB ACC、L1 rd/wr；按 main、UB/ACC、L1 分类。 数据是否在 main / UB / L1 间产生过大访问压力？
5 Cube L0 Path View L0A / L0B / L0C / FixPipe-L0C 分解，AIC lane 为主。 AIC 核内 L0 / L0C / FixPipe 路径是否异常？
6 Conflict / Issue View stu_pmu_wctl_ub_cflt、ldu_pmu_ib_ub_cflt、pmu_idc_aic_vec_instr_vf_busy_o、idu_pmu_ins_iss_cnt；不做 clc_cycle 求和主视角。 长 task 是否由冲突、issue 或 VF pressure 解释？
7 Local Buffer Pressure View ACC/UB read、ACC/UB write、MTE read/write、Fix write UB；AIV lane 优先。 AIV / UB / ACC / MTE 本地访问是否拖慢？
8 L2 Efficiency View AR/AW hit、miss、victim ratio；展示 read hit rate、write hit rate 与 victim。 L2 hit / miss / victim 是否说明复用失败或离散访问问题？

Group 2 指标规则

指标 规则 产品呈现
total_cycle task 在泳道图时间轴上的真实跨度。 作为 task 外框宽度和所有 PMU 解释基准。
clc_cycle cycle-like counter 聚合值，不包含 icache_req、icache_miss。 展示 pipeline coverage 或 overlap，但不单独作为好坏判断。
clc_cycle < total_cycle 疑似 uncovered / wait / idle / sync / uncounted stall。 红色或琥珀色 wait suspect badge。
clc_cycle ≈ total_cycle 当前 group 对 task 生命周期解释较充分。 绿色 accounted badge。
clc_cycle > total_cycle 可能是 pipeline overlap，也可能是 non-exclusive counter double-count。 紫色 overlap / double-count suspect badge，不表述为“更好”。

7. 交互设计

7.1 主流程

  1. 用户打开泳道图，默认进入当前 trace 对应的芯片型号与已采 PMU group。
  2. 系统展示诊断摘要：当前 group 是否可解释主要耗时、critical lane / Wrap / task、dominant counter。
  3. 用户在泳道图上点击异常 task 或 marker。
  4. 右侧 detail panel 展示 counter breakdown、gap / severity / efficiency、peer task、硬件路径。
  5. 用户点击源码 / CALL / Block Graph / 硬件路径，系统打开对应内容。
  6. 若当前 group 解释不足，系统给出下一次采集建议。

7.2 关键 PMU marker 机制

Marker 类型 生成条件 点击后显示
Critical task task 位于 Wrap critical path，且 total_cycle 或 group-specific score 排名前列。 task detail、dominant counter、peer task 对比、定位入口。
Wait suspect Group 2 下 clc_cycle < total_cycle 且 gap ratio 超阈值。 uncovered 时间段、同 Wrap peer task、下一次采集建议。
Overlap suspect Group 2 下 clc_cycle > total_cycle 且 overlap ratio 超阈值。 counter 非互斥说明、重叠 pipe、不要直接判断为更优的提示。
Hardware pressure dominant counter 指向某条硬件路径，且 pressure / severity 排名前列。 硬件路径高亮、相关 task 列表、可能优化方向。
Cache efficiency Group 8 下 read / write miss 或 victim 超阈值。 L2 hit rate、miss / victim breakdown、layout / stride / TileShape 排查入口。

7.3 定位链路

源码

打开对应源码片段并聚焦行号。缺少行号时聚焦算子或 callsite。

CALL

打开 CALL 视图，定位到 callOpMagic 对应节点。

Block Graph

打开 Block Graph，定位到 rootHash / leafHash 相关节点。

硬件路径

打开硬件路径面板，高亮 dominant counter 映射到的 Cube / MTE / UB / L0 / L2 / FixPipe / issue 结构。

Peer task

聚焦同 Wrap 内相关 AIC / AIV task，并显示 start / end / wait / dominant counter 对比。

采集建议

展示建议采集 group、原因和可复制的环境变量提示，例如 PROF_PMU_EVENT_TYPE=2。

8. 视觉设计

对象 视觉规则 原因
泳道 AIC、AIV0、AIV1 固定为独立 lane。task 不拆成独立 row，task 名称显示在 bar 内。 便于横向比较同一时间轴上的真实执行关系。
Task bar 外框表示 total_cycle，内部用当前 group 的语义分段或 ratio 表达。 同时保留宏观时间跨度与微观 PMU 解释。
Counter 颜色 按语义族群固定颜色：Compute、MTE、Memory、Buffer、Conflict、Cache、FixPipe 分别使用稳定色。 跨 group 保持认知稳定，但不要求所有 group 图形一致。
异常 badge Wait / uncovered 用红或琥珀，accounted 用绿色，overlap / double-count suspect 用紫色。 避免把 clc_cycle > total_cycle 误表达为绝对正向。
Marker 使用小型图标与 hover tooltip，避免遮挡 task 名称与时间轴。 关键事件可见，但不破坏泳道阅读。
Detail panel 先展示结论，再展示证据：结论、dominant counter、关联路径、原始 counter。 符合专家快速定位的阅读顺序。

9.3 派生指标

指标 适用 group 说明
clc_cycle / gap / gap_ratio Group 2 只聚合 cycle-like counter，不包含事件计数。
pressure_score Group 1 / 4 / 5 / 7 结合 task duration、dominant counter 占比和 lane / Wrap 排名。
conflict_severity Group 6 结合 conflict counter、issue density 与 total_cycle，不做 cycle 求和主视角。
read_hit_rate / write_hit_rate / victim_ratio Group 8 用于 L2 efficiency 判断和 cache churn 识别。

10. 状态与异常

状态 触发条件 界面反馈
未采集当前 group 用户切换到当前 run 未包含的 PMU group。 展示空状态、说明一次运行只能采一个 group，并给出重新采集建议。
counter 缺失 CSV 中缺少当前 group 应有字段。 隐藏缺失字段，提示数据不完整，不参与派生指标计算。
join 失败 PMU task identity 无法匹配泳道 task。 保留未匹配列表，允许用户查看 raw row，不在泳道上错误挂载。
定位信息缺失 缺少源码、CALL、Block Graph 或硬件路径映射。 对应入口 disabled，并说明缺少的 key，例如 leafHash。
A5 Mix 关系缺失 无法识别 Wrap 或 C/V peer 关系。 降级为 lane-level 展示，不生成 peer task 诊断结论。

11. 成功指标

定位效率

用户从打开泳道图到定位首个异常 task 的时间下降。

解释完成率

用户能判断当前 group 是否解释 total_cycle 的任务比例提升。

一跳定位使用率

源码、CALL、Block Graph、硬件路径入口的点击率与成功打开率。

下一次采集有效率

用户按系统建议采集新 group 后，能获得更明确诊断结论的比例。

12. 验收标准

ID 验收项 标准
AC-01 Group 切换 切换 group 1 / 2 / 4 / 5 / 6 / 7 / 8 后，工作台标题、摘要、counter 图例、task 内部表达、detail panel 均按 group 语义变化。
AC-02 泳道布局 AIC / AIV task 均落在对应 lane 上，不为每个 task 创建独立 task row，task 名称在 bar 内可读。
AC-03 Group 2 gap 能区分 clc_cycle < total_cycle、≈、> 三种状态，并正确排除 icache_req、icache_miss。
AC-04 A5 Mix 开启 Mix 模式后，同 Wrap 内 AIC / AIV0 / AIV1 可比较开始、结束、等待、计算分段与 critical lane。
AC-05 定位链路 点击源码、CALL、Block Graph、硬件路径后必须打开或聚焦对应内容，不允许无反馈。
AC-06 无效 counter 0x0 占位不出现在表格、图例、marker、排序、派生指标中。
AC-07 未采集 group 切换到未采集 group 时展示清晰空状态和下一次采集建议，不展示误导性空图。
AC-08 预留入口 多次运行对比、专家解释、BufferID / URMA / CCU 入口存在且状态明确，首版不要求完整可用。
下一步建议：用真实或接近真实的 tilefwk_prof_pmu.csv 与 merged_swimlane.json 验证 join schema，再把 group-specific diagnosis rules 固化为可配置映射。
