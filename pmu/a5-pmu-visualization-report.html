Design Report · A5 PMU Visualization

A5 PMU 数据可视化设计分析与洞察

本文整理当前围绕 A5 / dav_3510 PMU 数据可视化形成的产品判断、指标口径、诊断工作台设计与交互框架。核心结论：PMU group 不应被当作“字段集合”，而应被设计成不同的性能诊断镜头。

目标用户：算子开发者 目标用户：性能优化专家 首要场景：嵌入泳道图 当前聚焦：A5 / dav_3510

1. 核心产品判断

A5 PMU 可视化的目标不是把 counter 列出来，而是帮助用户判断：当前 group 能否解释 total_cycle，异常集中在哪条 lane / 哪个 Wrap / 哪个 task，dominant counter 对应哪条硬件路径，以及下一次应该采哪个 group。
从表格转向诊断

单 task counter 表只能回答“有什么值”；诊断工作台要回答“先看哪里、为什么看、下一步去哪”。

从 task 转向 Wrap

A5 Mix 场景下，同一个 Wrap 内的 AIC / AIV0 / AIV1 关系比单个 task 更关键。

从 counter 转向硬件路径

异常 counter 需要映射到 Cube、MTE、UB、L0、L2、FixPipe、issue / conflict 等硬件结构。

2. 关键指标口径

total_cycle、clc_cycle 与 gap

指标 含义 使用方式
total_cycle task 在泳道图时间轴上的真实跨度，近似 wall-clock duration。 作为所有 PMU 解释的外框和基准。
clc_cycle 对 cycle-like counter 聚合后的微观 pipeline 观察值。 只适合用于包含 cycle-like counter 的 group，例如 group 2；不应混入 event-count counter。
gap = clc_cycle - total_cycle 微观 PMU 聚合与真实时间跨度之间的差异。 作为 evidence，而不是直接作为好坏判断。
gap_ratio gap / total_cycle。 用于跨 task 排序和 badge 展示。
clc_cycle < total_cycle

可能表示 group counter 未覆盖完整 task 生命周期，疑似等待、空转、同步、未统计 stall 或非 pipeline 时间。

clc_cycle ≈ total_cycle

PMU pipeline 覆盖接近 wall-clock，可继续查看 dominant counter 是否集中。

clc_cycle > total_cycle

可能表示 pipeline 重叠，也可能是非互斥 counter 求和带来的 double-count 风险，不能直接判断为更好。

重要边界：像 icache_req、icache_miss 更像事件计数，不应进入 clc_cycle 求和。

3. PMU Group 是诊断镜头

Group 默认工作台 首要回答
1 Cube Arithmetic View AIC / Cube 算术指令是否是真瓶颈？
2 Pipeline Balance View AIC / AIV0 / AIV1 谁占流水，谁拖尾？
4 Memory Access Flow View 数据是否在 main / UB / L1 间产生过大访问压力？
5 Cube L0 Path View AIC 核内 L0 / L0C / FixPipe 路径是否异常？
6 Conflict / Issue View 长 task 是否由冲突、issue 或 VF pressure 解释？
7 Local Buffer Pressure View AIV / UB / ACC / MTE 本地访问是否拖慢？
8 L2 Efficiency View L2 hit / miss / victim 是否说明复用失败或离散访问问题？

4. 各 Group 的用户目标与默认视图

Group 1 · Cube Arithmetic Instruction Pressure

主要关注 cube_fp_instr_busy 与 cube_int_instr_busy。它不适合做完整 C/V balance，更适合判断 AIC / Cube 算术是否拖尾。

  • 找出 Cube FP / INT 指令 busy 最高的 AIC task。
  • 识别 long AIC task + high cube busy，作为 C 侧计算瓶颈。
  • 识别 long AIC task + low cube busy，提示转看 group 2 / 4 / 5 / 6。

Group 2 · Pipeline Balance / Utilization

最适合做 A5 Mix pipeline balance。task 外框展示 total_cycle，内部展示 cycle-like counter composition；icache_req / icache_miss 独立展示。

  • 一眼比较 AIC / AIV0 / AIV1 的 pipeline 占比。
  • 识别 uncovered / accounted / overlap。
  • 标出 critical Wrap 和 critical lane。
  • 从 dominant counter 追溯到硬件 pipe 和 peer task。

Group 4 · Main Memory + AIV/UB/L1 Access Flow

更像数据流访问视角，尤其对 AIV / UB / L1 和 main memory 访问敏感。

  • 找出主存读写压力最高的 task。
  • 找出 AIV UB read / write 最高的 task。
  • 判断 C/V 之间是否存在过多中间数据交换。

Group 5 · Cube L0 / L0C / FixPipe-adjacent Path

AIC / Cube 核内路径深钻，重点是 L0A / L0B / L0C 和 FixPipe 读 L0C。

  • 判断 L0A / L0B 输入是否均衡。
  • 判断 L0C read / write 是否异常高。
  • 追溯 fixp_rd_l0c_instr 到 FixPipe / output path。

Group 6 · UB Conflict / Issue / VF Pressure

不是常规利用率视角，而是冲突、issue 和 VF pressure 视角。

  • 识别 UB write-control conflict 高的 task。
  • 识别 load / IB / UB conflict 高的 task。
  • 区分 pipeline 没忙起来是等待、冲突，还是 issue 限制。

Group 7 · ACC / UB / MTE Local Access Pressure

重点是 ACC / UB 本地访问，以及 MTE 与 Vector / FixPipe 在本地 buffer 上的读写压力。

  • 找出 UB / ACC 读写最重的 AIV task。
  • 区分 Vector compute-heavy 和 UB/MTE access-heavy。
  • 追溯到 UB -> Vector -> UB 或 ACC/UB -> MTE -> GM/L2 路径。

Group 8 · L2 Hit / Miss / Victim Efficiency

用于 L2 efficiency，区分读侧 AR 与写侧 AW 的 hit / miss / victim。

  • 计算 read hit rate、write hit rate。
  • 找出 read miss / write miss / victim 最高的 task。
  • 识别高 victim，提示 cache churn / 复用失败。

5. 统一交互框架

选择 PMU group 切换诊断工作台 查看 lane / Wrap / task 热点 点击异常 task 查看 dominant counter 追溯硬件 pipe / peer task 获得下一次采集建议
区域 设计职责
顶部 group selector 切换诊断镜头，不只是切 counter 字段。
Immediate attention 给出当前 group 下最该先看的 lane / Wrap / task。
Lane summary 按 group 语义聚合 lane pressure、coverage、severity 或 efficiency。
Wrap-grouped swimlane 保留真实 total_cycle 时间跨度，同时在 task 内部展示当前 group 的 counter composition。
Wrap diagnosis 判断 C / V 谁拖尾、哪个 Wrap 破坏 balance、哪个 Wrap 复用效率差。
Detail panel 展示 task 解释、counter breakdown、硬件 pipe、peer task、源码锚点和下一次采集建议。

6. 高价值诊断目标补充

Pipeline saturation vs starvation

高 counter + 长 task 可能是 saturation；长 task + counter 不高更像 starvation / wait / uncovered。

Critical path 不等于最大 counter

Wrap 中最晚结束的 task 才决定用户感知的瓶颈，counter 最大的 task 不一定在关键路径上。

Group 无法解释也是结论

如果当前 group 的 dominant counter 不能解释 long task，应明确建议下一次采集哪个 group。

Peer task 是追溯入口

A5 Mix 里异常 task 往往要回看同 Wrap 内 C / V peer 的 start/end、wait 和 dominant counter。

7. 后续建议

后续应优先补齐数据契约与诊断解释链路，让工作台从可视化展示进一步变成可执行的定位工具。

下一步建议：用真实或更接近真实的 tilefwk_prof_pmu.csv + merged_swimlane.json 设计数据 join schema，明确 task identity、wrapId、event-hint.rootHash、callOpMagic、leafHash 的联动字段。
