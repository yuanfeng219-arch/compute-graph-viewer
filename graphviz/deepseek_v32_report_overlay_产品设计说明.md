# DeepSeek V3.2 Report Overlay — 产品设计说明

> 文件：`deepseek_v32_report_overlay_demo.html`
> 评审日期：2026-05-22（数据层与产品形态分析补充于同日）
> 范围：单文件 demo 评审 + 产品设计补充说明 + 产品形态分析 + 优化建议

---

## 1. 产品定位

把**静态的 NPU 性能分析报告**和**模型架构图**两个原本割裂的产物，叠加到同一个画布上，让性能工程师在「看模型结构」时直接看到「这一段慢在哪、为什么慢、下一步看什么」。

一句话：**架构图是地图，性能报告是路况，这个 demo 把路况画到了地图上。**

### 输入物
| 物料 | 来源 | 角色 |
|---|---|---|
| 模型架构图 | `deepseek_v32_source_graph.html`（Graphviz/D3 渲染，88 节点可折叠树） | 底图，iframe 嵌入 |
| 性能报告 A | `ds32 report/report (5).html`（NPU 性能分析报告：事实指标、每步拆分、Top-10 算子） | 数据源 |
| 性能报告 B | `ds32 report/ds3.2_report (9).html`（多流时序图，548 kernel 记录） | 数据源 |
| 叠加层 | demo 本身 | 把 A/B 的结论手工对齐到架构图节点上 |

---

## 2. 目标用户与核心场景

- **用户**：做大模型推理性能调优的工程师 / 算子工程师。
- **场景**：拿到一份 profiling 报告，结论是「P0 HOST_BOUND，Free Time 34.4%」。光看报告只有一堆算子名和百分比，对不上「这是模型的哪一段」。这个工具让他从架构图上的 `Decoder Block` 节点直接读到归因、证据、算子排名和下一步动作。
- **核心价值**：把「报告结论」翻译成「架构语境下的可定位、可下钻的诊断」。

## 3. 术语表（给非工程背景读者）

> 这份文档里的专业词，用生活类比解释。

### 硬件与执行

| 术语 | 是什么 | 生活类比 |
|---|---|---|
| **Profiling / 性能分析** | 给程序运行过程做一次「全程录像 + 计时」，记录每一步花了多久、在等什么 | 给一场马拉松装上沿途摄像头和秒表，赛后逐段回放 |
| **NPU / 昇腾 910B** | 专门用来算 AI 的芯片（和显卡 GPU 同类），这里是华为昇腾 | 专做一类菜的「中央厨房」，比通用厨房快很多 |
| **Device / 设备端** | 指 NPU 芯片本身——真正做计算的地方 | 后厨灶台 |
| **Host / 主机端** | 指 CPU——负责给芯片「派活、喂数据」的指挥方 | 前台配菜员，把食材一份份递进后厨 |
| **Step / 步** | 模型完整跑一轮推理叫一步；报告里看了第 10–19 步 | 流水线跑完一整轮；这次观察了其中连续 10 轮 |
| **Stream / 流** | 芯片上多条并行的「执行队列」，多个流可同时干活 | 超市开了 13 个收银台同时结账，每个台就是一条流 |
| **Kernel / 核函数** | 芯片上执行的最小计算任务，一步里有几十上百个 kernel | 菜谱里的一道道具体工序（切、炒、装盘） |
| **Operator / 算子** | 神经网络里的一类运算，如 MatMul（矩阵乘）；一个算子运行时拆成多个 kernel | 「炒」这个动作；具体炒哪盘菜就是 kernel |

### 性能诊断

| 术语 | 是什么 | 生活类比 |
|---|---|---|
| **Warmup step / 预热步** | 头几步还没进入稳定状态，数据不准，要单独排除（报告里是 Step 10） | 汽车冷启动的头一公里，不算进油耗统计 |
| **Free Time / 空闲时间** | 芯片闲着、没活干的时间占比（demo 里 34.4%） | 后厨灶台空烧，厨师在等配菜 |
| **HOST_BOUND** | 瓶颈在主机端——CPU 喂数据太慢，把芯片饿着了 | 厨师手很快，但配菜员跟不上，后厨频繁停工 |
| **Underfeed / 喂不满** | 设备没被持续喂满任务 | 灶台一次只放了半锅，本可以放满 |
| **Communication / 通信、CommNO** | 多张卡之间互相传数据；CommNO 指「没和计算重叠」的那部分通信（纯等待） | 几个厨房互相借料；借料时如果大家都停下来等，就是浪费 |
| **Overlap / 重叠** | 通信和计算同时进行，互不耽误（越高越好） | 一边等借的料、一边继续炒别的菜 |
| **Wall time / Op time / Wait time** | 墙上时间=实际经过的总时长；Op=真正在算的时间；Wait=在等待的时间 | 一段路程：总耗时 / 真正在开车 / 堵在红灯 |
| **Wait anchor / False hotspot（假热点）** | 某算子看起来很耗时，其实它只是在「等」上游，不是真的在算 | 队伍最后一个人站了很久，但他没干活，只是在等前面 |
| **Cube utilization / MFU** | 芯片算力的利用率，越高说明算得越满 | 灶台火力开到了百分之多少 |
| **P0 / P1 / P2** | 问题优先级，P0 最紧急 | 急诊分级：红、黄、绿 |

### 模型结构

| 术语 | 是什么 | 生活类比 |
|---|---|---|
| **Transformer / Decoder Block** | 大模型的主体，由一层层结构重复堆叠（这里重复 27 层） | 一栋楼，27 层结构几乎一样 |
| **MLA / Attention（注意力）** | 模型「读取上下文、决定关注哪些词」的机制；MLA 是 DeepSeek 的优化版 | 读一段话时，眼睛重点扫哪几个关键词 |
| **Sparse attention / top-k** | 不看全部上下文，只挑最相关的 k 个，省算力 | 复习时只看划重点的几页，不通读全书 |
| **MoE / Router / Expert（专家）** | 混合专家：一个问题不是全员处理，而是由 Router 路由给少数几个「专家」子网络 | 医院分诊台把病人转给对口的专科医生，而非全院会诊 |
| **Embedding / LM Head / Logits** | Embedding=把输入文字转成数字（入口）；LM Head/Logits=算出下一个词的概率（出口） | 入口的翻译柜台 / 出口的结算台 |
| **Speculative / MTP / spec_token_verify** | 投机解码：先快速「猜」几个词，再回头验证对不对，对了就省时间 | 抢答：先猜答案，再核对，猜中就赚了 |

## 4. 信息架构与布局

固定全屏画布（`100vw × 100vh`），三个浮层 overlay 在 iframe 底图之上：

```
┌─────────────────────────────────────────────────────────┐
│  [Home] ● Report Overlay  source graph…   [All P0 P1 P2 Off] [Refresh] │  ← overlay-header
│                                                          │
│        ┌──────────────────────────┐   ┌──────────────┐  │
│        │                          │   │  Inspector   │  │
│        │   架构图 iframe 底图       │   │  (400px 右栏) │  │
│        │   节点带 P0/P1/P2 描边     │   │  Diagnosis   │  │
│        │   + 优先级 badge          │   │  Evidence    │  │
│        │                          │   │  Operators   │  │
│        │                          │   │  Actions     │  │
│   ┌────────────────────────┐      │   │  Mapped Nodes│  │
│   │  Timeline 面板（可拖高） │      │   │              │  │
│   │  Steps / Streams / Cov  │      │   │              │  │
│   └────────────────────────┘      └───┴──────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 4.1 顶部 Header
左：Home 返回、品牌标识、source graph 加载状态。右：优先级过滤段控件（All / P0 / P1 / P2 / Off）、Refresh。

### 4.2 右侧 Inspector（核心诊断面板）
当前选中节点的报告，五段式：
1. **Diagnosis** — 一句话归因 + 4 个指标瓦片（带颜色 tone 和进度条）。
2. **Evidence** — 支撑这条结论的事实点。
3. **Operators** — 相关算子 + 耗时/占比。
4. **Actions** — 建议的下一步动作。
5. **Mapped Nodes** — 节点导航列表（受过滤影响）。

### 4.3 底部 Timeline 面板（三个 Tab）
- **Steps** — Step 10–19 每步的 Computing/CommNO/Free 堆叠条，warmup 步单独标黄。
- **Streams** — Step 15 的模块级时序，带**可播放的播放头**（Play/拖拽/点击 seek），下方 13 条 stream 泳道，每条显示 work/wait 占比。
- **Coverage** — demo 对两份报告的覆盖度自评（covered / partial / missing）。

## 5. 核心交互：三面板联动

| 触发 | 反应 |
|---|---|
| 点击架构图节点 | Inspector 切换到该节点报告 |
| 点击 Inspector 的 Mapped Nodes | Inspector 切换 + 架构图聚焦/居中该节点 |
| 点击 Timeline 模块行 / 泳道段 / 算子 chip | seek 到该段 + Inspector 切换 + 架构图聚焦 |
| Stream 播放（9s 动画） | 播放头扫过各模块段，自动切 Inspector 并联动架构图 dim 高亮 |
| 切换优先级过滤 | 架构图 overlay、Mapped Nodes 同步增删；Off 时整个 overlay 关闭 |

### 关键机制
- **节点叠加**：注入 SVG（优先级 badge + 彩色描边），监听 `MutationObserver` 应对 Graphviz 重渲染（80ms debounce 重绘）。
- **折叠回退**：选中节点若在折叠容器内不可见，沿祖先链找最近可见节点来高亮。
- **聚焦 dim**：选中后架构图非相关节点 `opacity: 0.08`，主选中节点带辉光。
- **持久化**：Timeline 高度存 `localStorage`，支持拖拽 + 键盘。

复用 PTO 设计系统三层 token：`foundation.css` / `semantic.css` / `components.css`。暗色主题，紫色系 accent。优先级专用色 P0 `#ff2d7a` / P1 `#ff9d00` / P2 `#ffe600`。无外部 UI 库，overlay SVG inline 注入。支持 `?preview` 参数输出干净底图。

## 7. 数据来源与「数据层」现状

### 7.1 真实的数据链路
报告 HTML（`report (5).html` / `ds3.2_report (9).html`）本身**是从一份结构化 JSON 渲染出来的**——profiling 工具产出 JSON，再由模板生成报告页面。也就是说，**数据层本来就存在**，标准链路是：

```
profiling 原始数据 → 结构化 JSON → 报告 HTML
                          └────────→ （理想）本 demo 直接消费
```

### 7.2 当前 demo 的问题：拿不到那份 JSON
本次没有拿到上游 JSON，所以 demo 把所有报告数据**手写成了常量**（`REPORTS`、`STEP_TIMELINE`、`MODULE_TIMELINE`、`STREAM_SUMMARY`、`COVERAGE_AUDIT`）。所谓「联动」是人工对齐的一次性快照——换一份报告整个数据层要重写。

### 7.3 建议：补一个「反向解析」的 JSON 中间文件
在拿到上游 JSON 之前，可以**从报告 HTML 反向解析**出一份 JSON，作为 demo 与报告之间的稳定中间层：

```
report (5).html / ds3.2_report (9).html
        │  反向解析（一次性脚本）
        ▼
deepseek_v32_report.json   ← 新增中间文件
        │  运行时读取
        ▼
demo（不再硬编码，只渲染 JSON）
```

这份 JSON 的 schema 应覆盖：节点↔报告映射、step 拆分、module timeline、stream summary、算子表、coverage。一旦 demo 改为「只渲染 JSON」，未来无论是接上游真 JSON、还是换模型，都只换 JSON 不动代码。**这是从「演示」走向「工具」的第一步，也是下文产品形态分析的前提。**

## 8. 产品形态分析：当前形态是不是最优解？

### 8.1 先回到产品目标
让性能工程师在架构语境下，**可定位、可下钻、可读懂**地理解一份性能报告。

### 8.2 当前形态拆解
单文件 HTML ＝ ①别人的架构图（iframe 底图）＋ ②外挂在它上面的 overlay 标记 ＋ ③右侧/底部诊断面板 ＋ ④手写数据。

### 8.3 这个形态「对」了什么
**「以架构图为锚点」的直觉是对的。** 用户面对报告时的第一个问题就是「这是我模型的哪一段」。把图当主屏、报告挂上去，正面回答了这个问题——这点应当保留。

### 8.4 这个形态的「天花板」（是形态本身的局限，不是 bug）

1. **「装饰别人的 iframe」是脆弱耦合。** overlay 和底图是两个独立文档，靠 `MutationObserver` 互相追、靠 same-origin 才能跨文档操作。底图一旦换版本、或换浏览器/部署方式，叠加层就可能失效。这是「外挂」形态的固有代价。

2. **架构图是为单一模型预渲染的、节点是人工策展的。** 报告里有 88 个时间线节点、548 条 kernel 记录，demo 只挑了 15 个挂报告。形态决定了它只能展示「策展子集」，给不了完整数据。

3. **空间视图和时间视图被分了主次，但诊断恰恰在时间一侧。** 架构图是**空间**的（模型长什么样），step/stream 是**时间**的（执行过程中发生了什么）。当前 timeline 被压在底部当配角——可这次报告的 P0（HOST_BOUND）和 P2（wait-anchor 假热点）**本质都是时间问题**：芯片何时空闲、谁在等谁。把诊断的主战场放在配角面板里，是形态与目标的错配。

### 8.5 候选形态对比

| 形态 | 描述 | 适合 | 短板 |
|---|---|---|---|
| **A. 图上叠加（当前）** | 架构图为底，报告结论挂上去 | 「在我模型哪」一目了然 | iframe 脆弱耦合；时间视图二等；只能展示策展子集 |
| **B. 报告内嵌架构小地图** | 以报告页为主，加一个「跳到架构」minimap | 数据完整、不依赖外部图 | 丢掉「空间锚点」的核心价值，退回报告思维 |
| **C. 分析工作台（推荐）** | 架构图 / Inspector / 时间线是**三个平等面板**，由一个数据模型 + 一个选中态统一驱动 | 空间与时间并重；可扩展；联动天然一致 | 需把 iframe 重构为应用内一等组件 |
| **D. IDE 插件** | 嵌入 PTO toolkit，紧挨模型源码 | 离工程师工作流最近 | 工程量大，依赖插件生态成熟 |

### 8.6 结论与建议

**当前 A 形态适合做「概念验证」，但不是达成产品目标的最优形态。** 它的「图为锚点」直觉要保留，但有两处需要演进：

- **把 iframe 底图升级为应用内的一等组件**——消除跨文档脆弱耦合，架构图、Inspector、时间线都吃同一份 JSON、同一个「当前选中」状态。这就是形态 **C：分析工作台**。
- **让时间线从「底部配角」升级为可与架构图平级切换/并列的主视图**——因为这次乃至多数推理性能问题，答案在时间轴上。理想交互是「空间 ⇄ 时间」一键切换，或左右并列、选中态互相同步。

**推荐路径**：C 是当前 demo 的自然下一步，且第 7.3 节的 JSON 数据层正是它的地基。**D（IDE 插件）作为长期方向**——用户侧已有 PTO toolkit 扩展，若把这套分析能力做进 IDE、挨着模型源码，是离工程师工作流最近的终局形态。

## 9. 优化建议

> 按「用户体验缺口」排序，实现成本放在每条末尾。

### 体验缺口类（优先）

**U1. Stream 播放经过 5 个段时 Inspector「卡住不动」**
`MODULE_TIMELINE` 里 `step_start_init`、`spec_token_verify_main`、`MTPLayer`、`spec_token_verify_mtp`、`final_step_sample` 五段的 `nodeId` 全部指向同一个 `logits` 节点。播放头扫过这 5 段（占总时长一大半）时 Inspector 反复显示同一内容，用户会以为联动断了。
→ 建议：为这些 runtime 阶段段建立独立的「阶段视图」内容，让播放过程持续有信息增量。

**U2. 联动「单向多、反向缺」**
Timeline → Inspector → 架构图 这条链通；但从 Mapped Nodes 选中节点时，Timeline 对应段**不会高亮**。三面板没有统一的「当前对象」高亮态。
→ 建议：把「选中节点」做成全局状态，任意面板变更都同步另外两个。（与第 8.6 节工作台形态一致）

**U3. 过滤态下点 Timeline 会跳到「被过滤掉」的节点**
优先级过滤只作用于架构图 overlay 和 Mapped Nodes，不作用于 Timeline。过滤到 P0 时仍能点 P2 的段，触发后 Inspector fallback 到第一个可见节点——点 A 跳到 B。
→ 建议：过滤态同步约束 Timeline 段的可点击性，或给明确提示。

**U4. 「Off」过于激进**
当前 Off 把 Inspector 和整个 Timeline 一起隐藏。用户大概率只是想关掉架构图上的彩色标记。
→ 建议：Off 只关 overlay 标记，保留面板；或拆成独立的「overlay 开关」。

**U5. 首次加载缺引导**
默认选中 Decoder Block，Inspector 有内容，但架构图没有聚焦到它。
→ 建议：初始化时对默认节点执行一次聚焦，让三面板开局就对齐。

**U6. 「Mapped Nodes」名不副实**
这一段列的是全部可见节点的导航列表，不是当前节点的子节点；下钻后无面包屑可回退。
→ 建议：①改名为「节点导航」，②Inspector 标题区加层级面包屑，③真正的子节点单列一段。

**U7. Coverage Tab 是给 demo 作者看的，不是给用户看的**
covered/partial/missing 是 demo 的元信息，对性能工程师价值低。
→ 建议：移到「关于/数据来源」弱化入口，Tab 让位给更有用的视图（如 Top-10 算子全表）。

**U8. Refresh 按钮语义不清**
`MutationObserver` 已自动重绘，Refresh 只是手动兜底。
→ 建议：默认隐藏，或改成只在检测到不同步时浮现的「重新对齐」提示。

### 数据与工程类

**O1. 落地 JSON 数据层（产品化前提）**
详见第 7 节：先做反向解析 JSON 中间文件，让 demo 改为「只渲染 JSON」。这是走向工作台形态的地基。

**O2. 播放速度与真实时间脱钩**
Stream 播放固定 9 秒，与真实 wall span（约 11.3ms）无映射。
→ 建议：提供正比/可调倍速，或在轴上标真实时间刻度。

**O3. `metricTone` 靠正则猜色调易错配**
`/free|underfeed|wait/ → critical` 这种文案匹配，换个 label 描述就配错颜色。
→ 建议：tone 作为数据字段显式声明。

**O4. 可访问性补全**
优先级段控件、Timeline tabs、Mapped Nodes 按钮均缺 `role` / `aria-pressed` / `aria-current` / `role="tab"`。

**O5. iframe 跨文档依赖 same-origin**
`file://` 直接打开时浏览器可能限制跨文档脚本。
→ 建议：文档注明需经本地 server 访问；中长期按第 8.6 节把 iframe 改为内联一等组件。

## 10. 评审结论

这是一个**完成度高、设计语言统一**的联动 demo——SVG overlay 注入、折叠节点祖先回退、防抖重绘、可拖拽 Timeline 这些细节处理得扎实，作为「报告 × 架构图联动」的概念验证是成立的。

核心待改的不是视觉，而是两层：
- **短期**：联动的一致性（U1–U5）——三面板缺一个统一的「当前对象」，Stream 播放大段无信息增量。
- **中期**：形态本身（第 7、8 节）——补 JSON 数据层、把「外挂 overlay」演进为「分析工作台」，让时间视图与架构图平级。

修好短期问题，体验从「能演示」上升到「能用来真分析」；走通中期演进，才真正达成产品目标。
